# 4.4 派发更新

  这小节我们来讲解派发更新，那什么时候去派发更新呢？就是我们修改数据的时候，当我们修改响应式数据时会触发它的set拦截，所以我们来看set中的逻辑

set: function reactiveSetter(newVal) {
  const value = getter ? getter.call(obj) : val
  /* eslint-disable no-self-compare */
  if (newVal === value || (newVal !== newVal && value !== value)) {
    return
  }
  /* eslint-enable no-self-compare */
  if (process.env.NODE_ENV !== 'production' && customSetter) {
    customSetter()
  }
  // #7981: for accessor properties without setter
  if (getter && !setter) return
  if (setter) {
    setter.call(obj, newVal)
  } else {
    val = newVal
  }
  childOb = !shallow && observe(newVal)
  dep.notify()
}

set方法会接收一个参数也就是修改后的值，这里会先用新值和旧值做比较，如果相同的话就什么都不做，这里旧值获取与在get方法中获取value一样。下面在开发环境这个逻辑先不去分析，因为这里我们没有传入customSetter。接下来一个逻辑有点意思，就是只有getter而没有setter时候return，这也是为了与用户定义行为一致。因为用户没有定义set时就不能修改数据的，但是我们重新定义后增加了set，但是set中什么都不做也就修改不成功了，这样就与用户预期一致了。
  接下来如果判断setter存在，也就是用户定义了set方法，那就调用setter方法，如果用户没有定义get和set那么就正常去修改val值为newVal。因为替换之后深度响应式就失效了，这里重新做一个深度响应式然后还是用childOb保存，这样在访问数据的时候触发get时收集依赖就会用到新的childOb。
  最后我们来看dep.notify()方法，这个方法我们其实可以猜到它就是触发更新用的方法，因为我们之前没有任何逻辑有做触发更新操作。我们先来看源码：

notify() {
    // stabilize the subscriber list first 首先稳定订阅者列表
    const subs = this.subs.slice()
    if (process.env.NODE_ENV !== 'production' && !config.async) {
      // subs aren't sorted in scheduler if not running async
      // we need to sort them now to make sure they fire in correct
      // order
      // SUB没有在调度程序中排序。如果没有运行异步，我们需要现在对它们进行排序，以确保它们以正确的顺序启动
      subs.sort((a, b) => a.id - b.id)
    }
    for (let i = 0, l = subs.length; i < l; i++) {
      subs[i].update()
    }
  }

  首先这里将subs数组浅克隆一份保存到subs中，接下来在开发环境且没有运行异步时候对subs进行排序，subs是数据的订阅者列表，也就是说一旦这个数据发生变化就要通知它的所有订阅者进行更新。所以之后逻辑就是遍历subs，调用subs[i].update方法。我们来看update方法：

  update () {
    /* istanbul ignore else */
    if (this.lazy) {
      this.dirty = true
    } else if (this.sync) {
      this.run()
    } else {
      queueWatcher(this)
    }
  }

Update有三个分支判断， lazy是与计算属性相关，sync与同步watcher相关，我们这里先分析queueWatcher方法。
queueWatcher方法定义在scheduler.js中，scheduler是调度程序的意思。我们来看源码

/**
 * Push a watcher into the watcher queue.
 * Jobs with duplicate IDs will be skipped unless it's
 * pushed when the queue is being flushed.
 * 将观察者推入观察者队列。
 * 具有重复ID的作业将被跳过，除非在刷新队列时推送。
 */
export function queueWatcher (watcher: Watcher) {
  const id = watcher.id
  if (has[id] == null) {
    has[id] = true
    if (!flushing) {
      queue.push(watcher)
    } else {
      // if already flushing, splice the watcher based on its id
      // if already past its id, it will be run next immediately.
      // 如果已经刷新，则根据其id拼接观察程序。
      // 如果已经超过其id，则将立即运行下一个观察程序。

      let i = queue.length - 1
      while (i > index && queue[i].id > watcher.id) {
        i--
      }
      queue.splice(i + 1, 0, watcher)
    }
    // queue the flush
    if (!waiting) {
      waiting = true

      if (process.env.NODE_ENV !== 'production' && !config.async) {
        flushSchedulerQueue()
        return
      }
      nextTick(flushSchedulerQueue)
    }
  }
}

  首先获取到watcher的id，这里传入的this就是watcher，如果has[id]是否不存在执行if分支，将has[id]置为true，这里有必要先来看几个全局变量

const queue: Array<Watcher> = []
const activatedChildren: Array<Component> = []
let has: { [key: number]: ?true } = {}
let circular: { [key: number]: number } = {}
let waiting = false
let flushing = false
let index = 0

has是一个空对象，这里key的类型需要是一个number，然后flushing和waiting初始值都是false，接下来呢就用到了flushing，如果它是一个false就将watcher添加到queue数组中，如果为true的话就取到queue队列最后一个元素的索引保存到i中，然后当I > index并且 queue[i].id > watcher.id时进入while循环，执行的逻辑很简单就是将i--，退出循环的话就将watcher插入到queue[i + 1]位置。接下来判断!waiting为false的就将它赋值为true，然后执行nextTick(flushSchedulerQueue)方法，nextTick这里简单理解为异步就好了，后面我们会专门分析它。 我们来看flushSchedulerQueue方法，这个方法逻辑是比较多的，我们拆分开来分析它

function flushSchedulerQueue() {
  currentFlushTimestamp = getNow()
  flushing = true
  let watcher, id

  // Sort queue before flush.
  // This ensures that:
  // 1. Components are updated from parent to child. (because parent is always
  //    created before the child)
  // 2. A component's user watchers are run before its render watcher (because
  //    user watchers are created before the render watcher)
  // 3. If a component is destroyed during a parent component's watcher run,
  //    its watchers can be skipped.
  // 在刷新之前对队列进行排序。
  // 这确保：
  // 1.组件从父级更新到子级。（因为父项总是在子项之前创建。）
  // 2.组件的用户观察程序在其渲染观察程序之前运行（因为用户观察程序是在渲染观察程序前创建的）
  // 3.如果组件在父组件的watcher回调中被销毁，则可以跳过其观察程序

  queue.sort((a, b) => a.id - b.id)
}

  首先通过getNow方法获取到当前时间的时间戳，getNow定义如下：

let getNow: () => number = Date.now

  然后将flushing设置为true，再定义两个空变量watcher和id。再往下有大段的注释，就是给queue中的watcher根据id从小到大进行了排序，这里又出现了一个user watcher的概念，我们之前还提到过一个渲染watcher，渲染watcher是在vue挂载时创建的用于渲染的观察者，这里的user watcher是用户观察程序，我们写watch配置选项或者通过$watch方法去观察一些数据变化生成的其实就是user watcher，这里也提到了它是在渲染watcher之前创建的。 我们接着往下看

  // do not cache length because more watchers might be pushed
  // as we run existing watchers
  for (index = 0; index < queue.length; index++) {
    watcher = queue[index]
    if (watcher.before) {
      watcher.before()
    }
    id = watcher.id
    has[id] = null
    watcher.run()
    // in dev build, check and stop circular updates.
    if (process.env.NODE_ENV !== 'production' && has[id] != null) {
      circular[id] = (circular[id] || 0) + 1
      if (circular[id] > MAX_UPDATE_COUNT) {
        warn(
          'You may have an infinite update loop ' + (
            watcher.user
              ? `in watcher with expression "${watcher.expression}"`
              : `in a component render function.`
          ),
          watcher.vm
        )
        break
      }
    }
  }

  这里是一个for循环遍历queue数组，这里没有去cache它的长度，因为执行过程中queue可能会被push更多的观察者，所以这里选择每次去计算长度。注意这里的index不是新定义的变量，而是我们上面说过的全局变量。
  内部逻辑就先用watcher保存当前遍历到的观察者，然后调用它的before方法，这个before方法其实我们在讲生命周期小节时候就讲解过了，其实就是去调用更新前的生命周期钩子。接着往下用id保存watcher的id，然后将has[id]重置为null，接着执行watcher.run()方法，这里我们先看一下run方法的实现

run() {
  if (this.active) {
    const value = this.get()
    if (
      value !== this.value ||
      // Deep watchers and watchers on Object/Arrays should fire even
      // when the value is the same, because the value may
      // have mutated.
      // 即使值相同，深度观察者和对象/数组上的观察者也应该激发，因为值可能发生了变化
      isObject(value) ||
      this.deep
    ) {
      // set new value
      const oldValue = this.value
      this.value = value
      if (this.user) {
        const info = `callback for watcher "${this.expression}"`
        invokeWithErrorHandling(this.cb, this.vm, [value, oldValue], this.vm, info)
      } else {
        this.cb.call(this.vm, value, oldValue)
      }
    }
  }
}

run方法在this.active为true的时候才执行，这里我们认为它是true就好了，因为在new Watcher时候它被赋值为true。然后我们看if内部逻辑，先执行了get()方法去求值，让后判断两次value不相等，或者value是对象或者是深度监听时候，执行if分支逻辑，这里就是保存旧的value然后更新新的value，然后执行invokeWithErrorHanding方法，这个方法与else分支的方法一样，但是加了错误处理程序，因为用户定义程序是不稳定的，为了防止其报错导致程序不能正常运行，所以它要在try...catch...中去调用，invokeWithErrorHanding我们不会在这里分析。
  实际上对于我们渲染watcher而言，run方法简直不要太简单，它只执行第一句逻辑，也就是调用this.get，this.get方法实际就是执行updateComponent方法去更新组件。分析到这里派发更新的问题就已经迎刃而解了，看到这里你会不会有些小失望呢？原来派发更新也没有想象中的神奇，它也并不能准确的找到要更新的dom，只能找到数据更新的组件去更新。如果仅仅是这样的话，那Vue也谈不上什么效率了，每次修改数据都要去触发一次组件的更新，那么它会有极大的性能浪费。如果是这么low的话那Vue怎么可能这么受欢迎呢？那么我们就只能寄希望于组件更新时做优化了，在后面小节我们分析组件更新过程时再一起来探索它如何进行性能优化的吧！在这之前我们要先意识到一个问题：我们的flushSchedulerQueue方法还没有分析完，虽然已经明白派发更新是怎么一回事了，但是后续的小细节还是需要我们去分析明白的。
  言归正传，run方法后是一个开发环境的校验，它会尝试判断是否有一个循环更新，如果有的话就报一个警告并跳出循环。如何判断循环更新，什么时候命中这个警告我们之后分析，再来看for循环之后逻辑

  // keep copies of post queues before resetting state
  // 在重置状态之前保留post队列的副本
  const activatedQueue = activatedChildren.slice()
  const updatedQueue = queue.slice()

  resetSchedulerState()

  // call component updated and activated hooks
  callActivatedHooks(activatedQueue)
  callUpdatedHooks(updatedQueue)

  // devtool hook
  /* istanbul ignore if */
  if (devtools && config.devtools) {
    devtools.emit('flush')
  }

  这里activated是与keep-alive相关的我们暂时还不会分析它。callUpdatedHooks方法我们之前在生命周期章节已经介绍过了，实际就是依次执行updated钩子，最后一段逻辑是对开发工具使用的，我们也不用去管它。再来看resetSchedulerState方法

function resetSchedulerState() {
  index = queue.length = activatedChildren.length = 0
  has = {}
  if (process.env.NODE_ENV !== 'production') {
    circular = {}
  }
  waiting = flushing = false
}

  就最后初始化公共变量以便下次更新能够正常使用。到这里我们派发更新的内容基本就结束了。
  最后我们再填几个前面提到的坑，这里我们也可以提前说一些有关user watcher的逻辑，虽然我们还没有分析这部分内容，但是想必大家对watch已经很熟悉了，在这里稍微超前讲述一些内容也不是那么难理解，在queueWatcher执行过程中有一个对flushing判断逻辑，如果它为false，我们会走if逻辑，反之会进入else逻辑，这里问题就来了，那么什么时候flushing置为true呢？答案很简单，执行flushSchedulerQueue第一句逻辑就是将它设置为true，所以flushing就是判断是否正在更新中的标识位。真正的难解决的是flushSchedulerQueue是异步执行的，也就是所有同步操作包括queueWatcher也都执行完毕了，还怎么触发queueWatcher呢？如果flushSchedulerQueue执行完成后flushing又会重置为false，那么else逻辑永远不会执行到。这个烧脑的问题我们直接来公布答案吧！在执行watcher.run()的时候可能会执行一些修改逻辑，这样就又会执行到queueWatcher了，比如我们在watch监听一个数据改变后在回调函数里又里修改了一些数据。这里我们就又有一个疑问（好吧我承认是我自己的疑问？请不要骂我！！😝）。如果我们在watch回调中修改本组件的数据，那又会触发当前组件渲染watcher，这时has对象已经保存过这个watcher的id了queueWatcher。那这时候是不会把渲染watcher添加到执行队列，因为添加前的判断逻辑是has[id]==null，那这里就不能执行更新了吗？我们现在看watcher.run上一句：

  has[id] = null

  所以任何一个不起眼的语句都不能随便忽略的，我第一眼看到它就觉得它是重置数据为下一次更新做准备，但是忽略了它的调用时机导致我们多了一个疑问。然后这里我们就清晰起来了。这么讲可能不太直观，所以还是老办法，写一个dome：

<template>
  <div @click="change">
    hello world
    {{ isB ? b : c.d }}
    {{ msg }}
  </div>
</template>

<script>
export default {
  data() {
    return {
      isB: true,
      b: '我是app',
      c: {
        d: '我也是app'
      },
      msg: '监听我吧'
    }
  },
  watch: {
    msg() {
      this.msg = '我让你死循环！' + Math.random()
    }
  }
}
</script>

  在原有基础上做了个修改，增加了msg数据，然后增加了一个监听，监听器中又修改了响应式数据（不要在意我写的对不对），msg中的函数会在watcher.run时执行，然后它修改数据后又触发set拦截，然后执行notify方法，进而执行queueWatcher方法，这时便可以进入flushing为true的逻辑

let i = queue.length - 1
while (i > index && queue[i].id > watcher.id) {
  i--
}
queue.splice(i + 1, 0, watcher)

  再来看这段逻辑就很好理解了，index 是flushSchedulerQueue中循环更新的位置，如果i > index时说明还没更新到i位置，如果queue[i].id > watcher.id也成立的话也就是i位置的wathcer.id是比新触发的watcher.id大的，因为它是挨个排序的，所以这个新的watcher要插入更前面的地方去更新，所以让i--然后去找watcher正确插入位置。当第二个条件不满足时候才是正确的位置，但是如果不满足第一个条件的话，说明前面已经更新过了，如果再往前面插入是不会再遍历前面的项了，那么插入正确位置不更新也就没有意义了，所以这时的处理是把它放到当前更新项的后一位，下次立即更新它。
  我们写的这个小dome是有问题的，它在watch msg的回调中一直修改msg会一直触发更新，这就是上面说到的flushSchedulerQueue在开发环境中校验的东西。

if (process.env.NODE_ENV !== 'production' && has[id] != null) {
  circular[id] = (circular[id] || 0) + 1
  if (circular[id] > MAX_UPDATE_COUNT) {
    warn(
      'You may have an infinite update loop ' + (
        watcher.user
          ? `in watcher with expression "${watcher.expression}"`
          : `in a component render function.`
      ),
      watcher.vm
    )
    break
  }
}

  这里has[id]是true的，然后这里用circular对象统计当前id出现的次数has[id]是true的话说明这个数据是在run里修改的，否则它会是null，然后如果circular[id]大于100就会停止更新并报出警告说我们可能在watch xx数据时写了一个无限循环

export const MAX_UPDATE_COUNT = 100

MAX_UPDATE_COUNT这里被定义为100，所以这里是判断我们可能有无限循环，100次循环只是可能性不能100%确认。
