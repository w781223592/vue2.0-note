# 4.2 依赖收集（上）

  我们在使用vue过程中修改数据的变化会引起视图的变化，不知道你是否想过数据改变引起视图变化是如何实现的？还是想当然认为数据变化就会影响视图变化，显然不是这样的，是vue背后为我们做了很多工作。我们来考虑假如让我们去做这件事会怎么做呢？当然是修改数据时找到所有使用这个数据的地方，然后更新它们，这好像对我们来说没有什么难度。人是智能的，有识别能力的，做到这点并不难，但是代码要怎么实现呢？这就是难点所在了，代码没有人的智慧，它不会在修改去找哪里使用了这些响应式数据，它只能事先将所有使用到这个数据的地方收集起来，等到修改数据的时候，再将这些地方进行更新。那么怎么能知道哪里使用到了这个数据呢？我们对一个对象的属性是通过'.'去访问的，这个时候就触发了我们的get函数，每次使用数据都是对数据的一次访问，所以我们再get中去收集它们是不是就可以了呢？所以我们接着上一个小节来讲解get方法。

get: function reactiveGetter () {
  const value = getter ? getter.call(obj) : val
  if (Dep.target) {
    dep.depend()
    if (childOb) {
      childOb.dep.depend()
      if (Array.isArray(value)) {
        dependArray(value)
      }
    }
  }
  return value
}

  它首先定义了value，如果getter存在则将getter执行结果保存在value中，否则使value等于val，这个就保证了如果我们自定义了get函数在重新定义get后仍然能返回相同的结果，然后就是判断Dep.target是否存在，我们之前已经见过Dep，我们在上小节分析defineReactive方法时候第一句逻辑就是new Dep，所以Dep其实是就是个class类，接下来我们来看Dep的定义，它定义于observe/dep.js

export default class Dep {
  static target: ?Watcher;
  id: number;
  subs: Array<Watcher>;

  constructor () {
    this.id = uid++
    this.subs = []
  }

  addSub (sub: Watcher) {
    this.subs.push(sub)
  }

  removeSub (sub: Watcher) {
    remove(this.subs, sub)
  }

  depend () {
    if (Dep.target) {
      Dep.target.addDep(this)
    }
  }

  notify () {
    // stabilize the subscriber list first 首先稳定订阅者列表
    const subs = this.subs.slice()
    if (process.env.NODE_ENV !== 'production' && !config.async) {
      // subs aren't sorted in scheduler if not running async
      // we need to sort them now to make sure they fire in correct
      // order
      subs.sort((a, b) => a.id - b.id)
    }
    for (let i = 0, l = subs.length; i < l; i++) {
      subs[i].update()
    }
  }
}

  可以看到Dep是个比较小的类，我们可以找到它的target是一个静态属性，可以看到它的构造函数十分简单，先是将this.id设置为uid++，然后使this.subs = []

// The current target watcher being evaluated.
// This is globally unique because only one watcher
// can be evaluated at a time.
// 正在计算 的当前目标观察程序。
// 这是全局唯一的，因为一次只能计算一个观察者
Dep.target = null

  我们在下面找到了Dep.target的初始值被设为null，这里开始就有难度了，因为我们发现我们分析不下去了，这里代码好像出现了断层，因为Dep.target为null所以不会执行if判断中的逻辑，这里我们应该从get函数执行时机来分析了。首先Dep.target这里就是个全局的watcher，同一时间只有一个watcher可以被计算，我们要找到target是什么时候被赋值的，然后找到get函数的执行时机，这一切就解释通了。
  仔细分析Dep的定义，可以看到它其实与watcher是密不可分的，之前我们在Vue初始化时提到过一个渲染watcher的概念如下所示

new Watcher(vm, updateComponent, noop, {
  before() {
    if (vm._isMounted && !vm._isDestroyed) {
      callHook(vm, 'beforeUpdate')
    }
  }
}, true /* isRenderWatcher */)

  在挂载过程中创建了渲染watcher，而updateComponent的执行依赖于watcher的get函数执行，当时我们没有过多去分析有关响应式的内容，这里我们将这块内容补全。如果你已经忘记class Watcher的逻辑，你可以在watcher文档中回顾它，在Watcher的构造函数最后一段逻辑是执行它的get方法

  get () {
    pushTarget(this)
    let value
    const vm = this.vm
    value = this.getter.call(vm, vm)
    popTarget()
    this.cleanupDeps()
    return value
  }

  在这里我们弄一个watcher简化版的get逻辑出来分析，首先它是执行了pushTarget方法，我们接下来就看看它做了什么事，实际上这个函数是定义在dep.js中的

const targetStack = []

export function pushTarget(target: ?Watcher) {
  targetStack.push(target)
  Dep.target = target
}

  在之前定义了targetStack空数组，可以看到这里是通过栈的方式来保存target，然后将Dep.target设置为target，那这里的target参数是什么呢，这里传入的是this也就是这里正在计算的渲染watcher了，所以这里的Dep.target已经有值了，它是一个watcher。那我们继续往下分析看我们什么时候会调用get方法，我们继续往下看渲染watcher的get方法，它调用了this.getter，其实就是updateComponent方法，这又回到我们的挂载流程了，我们来写一个简单例子来辅助理解：

new Vue({
  el: '#app',
  data: {
    a: 'root'
  },
  render(h) {
    return h('div', this.a)
  }
})

  这个例子是不是非常简单呢？我们已经知道在执行updateComponent方法是会调用vm._render()方法，然后就会执行到render方法，然后这里我们访问了this.a，也就是会访问data中的a，这样就触发了我们get拦截

get: function reactiveGetter () {
  const value = getter ? getter.call(obj) : val
  if (Dep.target) {
    dep.depend()
    if (childOb) {
      childOb.dep.depend()
      if (Array.isArray(value)) {
        dependArray(value)
      }
    }
  }
  return value
}

  回到我们get()方法中，判断Dep.target存在然后执行dep.depend()，然后我们来看depend方法

depend () {
  if (Dep.target) {
    Dep.target.addDep(this)
  }
}

  就是执行了Dep.target.addDep方法，我们之前说过  是个  ，然后我们到  中去找  方法

/**
 * Add a dependency to this directive. 向该指令添加依赖项
 */
addDep (dep: Dep) {
  const id = dep.id
  if (!this.newDepIds.has(id)) {
    this.newDepIds.add(id)
    this.newDeps.push(dep)
    if (!this.depIds.has(id)) {
      dep.addSub(this)
    }
  }
}

  这里接收一个dep参数，我们传入的是this，也就是Dep构造的对象，先获取id，如果this.newDepIds中没有id，则执行if中逻辑，这里的newDepIds是new Watcher时添加的属性如下所示：

  this.deps = []
  this.newDeps = []
  this.depIds = new Set()
  this.newDepIds = new Set()

  如果没有id则往进添加id，并且将dep对象添加到this.newDeps中，再判断如果depIds中没有id就调用dep.addSub(this)，这里的this是什么，那就得看是谁调用的addDep，这里是Dep.target调用的，也就是watcher对象作为dep.addSub参数传入，然后我们接下来当然是看addSub方法了

addSub (sub: Watcher) {
  this.subs.push(sub)
}

addSub方法十分简单就是将传入的sub添加到this.subs中，到这里dep.depend()就执行完成了。然后我们再往下看如果childOb存在则执行childOb.dep.depend()，上一小节我们花了大篇幅去介绍childOb，childOb.dep是__ob__中取到的dep是在new Observer中new Dep出来的实例，而上面的dep是defineReactive中new Dep的实例。其实我们可以把dep看成是一个筐，专门用来装依赖的筐，这里的dep和childOb.dep是两种筐，这里我们会在后面说明。childOb.dep.depend()执行和dep.depend()过程相同这里就不再赘述了。

function dependArray(value: Array<any>) {
  for (let e, i = 0, l = value.length; i < l; i++) {
    e = value[i]
    e && e.__ob__ && e.__ob__.dep.depend()
    if (Array.isArray(e)) {
      dependArray(e)
    }
  }
}

  这里如果value是个数组的话就会执行dependArray方法，dependArray方法就是遍历value数组，如果数组元素有__ob__属性的话就执行e.__ob__.dep.denpend()方法，我们之前分析observe方法对数组处理是递归调用observe方法，而不是defineReactive方法去添加__ob__属性，所以这里我们数组是没有__ob__属性的，但是如果数组中的元素是一个对象的话它是会有__ob__属性的。
  最后判断数组中的元素是否还是个数组，如果还是数组的话递归调用dependArray方法保证所有依赖都收集到。
  这里逻辑是比较晦涩的，理解起来可能不是那么容易，因为依赖收集可能只是Vue执行过程中的小片段，我们这样只是把源码讲解一遍可能不是那么容易理解，下一小节我们通过一个小dome来结合实际场景并把依赖收集放到Vue整个执行过程中来分析将会容易理解的多。
