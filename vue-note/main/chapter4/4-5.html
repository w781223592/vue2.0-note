# 4.5 nextTick

  在介绍nextTick前我们需要先说明一下js的运行机制：
  我们JavaScript 语言的一大特点就是单线程，也就是说，同一个时间只能做一件事。为了协调事件、用户交互、脚本、UI 渲染和网络处理等行为，防止主线程的不阻塞，Event Loop 的方案应用而生。Event Loop 包含两类：一类是基于 Browsing Context，一种是基于 Worker。二者的运行是独立的，也就是说，每一个 JavaScript 运行的"线程环境"都有一个独立的 Event Loop，每一个 Web Worker 也有一个独立的 Event Loop。
  根据规范，事件循环是通过任务队列的机制来进行协调的。一个 Event Loop 中，可以有一个或者多个任务队列(task queue)，一个任务队列便是一系列有序任务(task)的集合；每个任务都有一个任务源(task source)，源自同一个任务源的 task 必须放到同一个任务队列，从不同源来的则被添加到不同队列。setTimeout/Promise 等API便是任务源，而进入任务队列的是他们指定的具体执行任务。 在事件循环中，每进行一次循环称为tick每一次 tick 的任务处理模型是比较复杂的，但关键步骤如下：

（1）执行一个宏任务（栈中没有就从事件队列中获取）  
（2）执行过程中如果遇到微任务，就将它添加到微任务的任务队列中  
（3）宏任务执行完毕后，立即执行当前微任务队列中的所有微任务（依次执行）  
（4）当前宏任务执行完毕，开始检查渲染，然后GUI线程接管渲染  
（5）渲染完毕后，JS线程继续接管，开始下一个宏任务（从事件队列中获取）

规范中规定task分为两大类，分别是宏任务（macro task）和微任务（micro task）。

宏任务：
  (macro)task，可以理解是每次执行栈执行的代码就是一个宏任务（包括每次从事件队列中获取一个事件回调并放到执行栈中执行）。 浏览器为了能够使得JS内部(macro)task与DOM任务能够有序的执行，会在一个(macro)task执行结束后，在下一个(macro)task 执行开始前，对页面进行重新渲染，流程如下： (macro)task->渲染->(macro)task->...

宏任务包括：

script(整体代码)  
setTimeout  
setInterval  
postMessage  
MessageChannel  
setImmediate  
I/O  
UI交互事件

微任务：
  microtask,可以理解是在当前 task 执行结束后立即执行的任务。也就是说，在当前task任务后，下一个task之前，在渲染之前。 所以它的响应速度相比setTimeout（setTimeout是task）会更快，因为无需等渲染。也就是说，在某一个macrotask执行完后，就会将在它执行期间产生的所有microtask都执行完毕（在渲染前）。

微任务包括：

Promise.then
MutationObserver
Object.observe
process.nextTick(Node.js 环境)

  我们为什么要来分析js异步机制呢？因为Vue的更新是异步执行的，为什么要去异步更新呢？我们知道在Vue中改变数据会触发更新，如果我们采用同步更新的方式，那么我改版几次数据就要更新几次dom，如果业务复杂，一个方法中可能修改很多的值，每次修改都要进行更新的等待，那我们更新效率会是及其低下的。如果采用异步的方式去更新就不同了，因为异步任务会在同步任务全部执行完毕时才会执行，如果这次同步任务更新了成百上千条数据，我们只要一次性全部更新就好了，这是极大的优化。
  下面我们就可以来分析nextTick方法了，首先Vue提供了两种调用nextTick，一种是全局API方式：Vue.nextTick，它定义于global-api/index.js中：

Vue.nextTick = nextTick

  另一种是通过vm.$nextTick的方式，它定义于render.js下，在renderMixin时被引入

Vue.prototype.$nextTick = function (fn: Function) {
  return nextTick(fn, this)
}

  不管是哪种方式实际上都是执行nextTick方法，下面就来了解nextTick的实现：

const callbacks = []
let pending = false

export function nextTick (cb?: Function, ctx?: Object) {
  let _resolve
  callbacks.push(() => {
    if (cb) {
      try {
        cb.call(ctx)
      } catch (e) {
        handleError(e, ctx, 'nextTick')
      }
    } else if (_resolve) {
      _resolve(ctx)
    }
  })
  if (!pending) {
    pending = true
    timerFunc()
  }
  // $flow-disable-line
  if (!cb && typeof Promise !== 'undefined') {
    return new Promise(resolve => {
      _resolve = resolve
    })
  }
}

  首先定义_resolve变量，然后往callbacks中push一个匿名函数，这个匿名函数有两个分支，if分支是判断cb存在然后就执行cb函数，如果没有传cb函数，那么再判断_resolve是否存在，如果存在的话就执行_resolve(ctx)。当然这里只是函数定义，并没有执行，所以在_resolve是undefined并没有什么影响。接着再往下如果pending不存在将pending赋值为true，然后执行timerFunc。
  最后如果我们没有传cb函数的话并且支持Promise方法的话，就return一个Promise对象，这里是将Promise的resolve函数赋值给_resolve。那么callbacks中函数或_resolve函数什么时候调用呢？那只能是在timerFunc中调用了，timerFunc显然是个异步程序，执行callbacks中函数时调用_resolve就会报错了，因为promise是在最后执行的，new Promise之后才给_resolve赋值。这里调用timerFunc前先将pending置为true，目的也很简单，就是为了防止重复调用timerFunc，就是当我们在一次操作中修改了多条数据，甚至去手动调用nextick方法时，是不会重复执行timerFunc方法的，只是在callbacks数组中多添加一次操作。
  下面我们就是这小节的重头戏timerFunc的定义了。我们开头花了很大篇幅去讲解js的运行机制就是为了此刻做准备的。timerFunc定义是比较复杂的，因为浏览器兼容性差距比较大，所以为了处理兼容性就导致timerFunc定义复杂。
  我们之前讲过在我们执行栈的同步任务执行完成后会先执行microtask，之后才会执行下一个macrotask，而每个macrotask之间又会穿插渲染，如果能够在microtask完成所有数据的更新，那么我们只需一次重新渲染就完成了本轮更新，这将是一个极大的性能提升，但是理想是美好的，现实是残酷的，并不是所有浏览器都支持microtask，所以我们只能优先使用microtask，当浏览器不支持时再降级为macrotask。具体我们来看代码实现：

export let isUsingMicroTask = false
// Here we have async deferring wrappers using microtasks.
// In 2.5 we used (macro) tasks (in combination with microtasks).
// However, it has subtle problems when state is changed right before repaint
// (e.g. #6813, out-in transitions).
// Also, using (macro) tasks in event handler would cause some weird behaviors
// that cannot be circumvented (e.g. #7109, #7153, #7546, #7834, #8109).
// So we now use microtasks everywhere, again.
// A major drawback of this tradeoff is that there are some scenarios
// where microtasks have too high a priority and fire in between supposedly
// sequential events (e.g. #4521, #6690, which have workarounds)
// or even between bubbling of the same event (#6566).
// 这里我们有一个使用微任务的异步延迟包装器。
// 在2.5中，我们使用了（宏）任务（与微任务相结合）。
// 然而，当状态在重新绘制之前发生变化时（例如#6813，输出-输入转换），它有一些微妙的问题。
// 此外，在事件处理程序中使用（宏）任务会导致一些无法回避的奇怪行为（例如#7109、#7153、#7546、#7834、#8109）。
// 因此，我们现在再次在任何地方使用微任务。
// 这种折衷的一个主要缺点是，在某些情况下，微任务的优先级太高，并且在假定的连续事件之间触发（例如#4521、#6690，具有变通方法），
// 甚至在相同事件的冒泡之间触发（#6566）。
let timerFunc
先定义 isUsingMicroTask、timerFunc
// The nextTick behavior leverages the microtask queue, which can be accessed
// via either native Promise.then or MutationObserver.
// MutationObserver has wider support, however it is seriously bugged in
// UIWebView in iOS >= 9.3.3 when triggered in touch event handlers. It
// completely stops working after triggering a few times... so, if native
// Promise is available, we will use it:
// nextTick行为利用了微任务队列，可以通过原生Promise.then或MutationObserver访问该队列。
// MutationObserver拥有更广泛的支持，但在iOS>=9.3.3中，当在触摸事件处理程序中触发时，它在UIWebView中
// 受到严重的错误。它在触发几次后完全停止工作…因此，如果原生Promise可用，我们将使用它

if (typeof Promise !== 'undefined' && isNative(Promise)) {
  const p = Promise.resolve()
  timerFunc = () => {
    p.then(flushCallbacks)
    // In problematic UIWebViews, Promise.then doesn't completely break, but
    // it can get stuck in a weird state where callbacks are pushed into the
    // microtask queue but the queue isn't being flushed, until the browser
    // needs to do some other work, e.g. handle a timer. Therefore we can
    // "force" the microtask queue to be flushed by adding an empty timer.
    // 在有问题的UIWebViews中，Promise.then不会完全崩溃，但它可能会陷入一种奇怪的状态，
    // 回调被推送到微任务队列中，但队列不会被刷新，直到浏览器需要做一些其他工作，例如处理计时器。
    // 因此，我们可以通过添加一个空计时器来“强制”刷新微任务队列。
    if (isIOS) setTimeout(noop)
  }
  isUsingMicroTask = true
}

  可以看到它的定义十分复杂，为了解决兼容性问题，它做了许多令人费解的操作，好在有足够多的注释让我们来了解它。这里先判断浏览器有如果有原生的Promise方法就通过Promise.resolve方法去生成一个微任务队列。然后将timerFunc定义为一个函数，函数执行逻辑就是通过Promise.then()来执行回调函数flushCallbacks，当然为了兼容ios的问题，如果是IOS环境则执行一个空计时器来强制刷新微任务队列，最后将isUsingMicroTask置为true，也就是使用了微任务。

else if (!isIE && typeof MutationObserver !== 'undefined' && (
  isNative(MutationObserver) ||
  // PhantomJS and iOS 7.x
  MutationObserver.toString() === '[object MutationObserverConstructor]'
)) {
  // Use MutationObserver where native Promise is not available,
  // e.g. PhantomJS, iOS7, Android 4.4
  // (#6466 MutationObserver is unreliable in IE11)
// 在原生Promise不可用的地方使用MutationObserver，例如PhantomJS、iOS7、Android 4.4（#6466 MutationOserver在IE11中不可靠）
  let counter = 1
  const observer = new MutationObserver(flushCallbacks)
  const textNode = document.createTextNode(String(counter))
  observer.observe(textNode, {
    characterData: true
  })
  timerFunc = () => {
    counter = (counter + 1) % 2
    textNode.data = String(counter)
  }
  isUsingMicroTask = true
}

  如果没有Promise则再判断不在IE浏览器内，并且有原生的MutationObserver方法，就通过MutationObserver方法来实现，具体MutationObserver是一个用来监听DOM改变的类。它构造函数接收的是一个回调函数，也就是监听到dom变化会就会执行回调。其原型上有三个方法：

（1）disconnect()方法会阻止实例接收通知，直到再次调用observe()，该观察者对象包含的回调函数都不会再被调用
（2）observe()方法，接收两个参数，第一个是target，也就是要观察的dom对象，第二个是配置选项options，也就是告诉观察者观察dom的哪些变化。
（3）takeRecords()方法，从 MutationObserver 的通知队列中删除所有待处理的通知，并将它们返回到 MutationRecord 对象的新 Array 中。

  这里我们只用到了observe方法，所以我们只要了解它的第二个方法即可。 我们来看Vue中是怎么通过MutationObserver巧妙的将flushCallbacks注册为microtask。首先定义counter为number 1，然后实例化MutationObserver并保存在observer变量中。然后创建一个文本节点，文本内容为String(counter)保存在textNode变量中。
  然后就调用observe方法监听textNode变化，监听配置为characterData: true，也就是监听字符数据的变化。因为MutationObserver是监听到变化才会执行回调，所以在我们只要修改textNode.data的值就可以触发flushCallbacks函数执行。所以接下来给timerFunc赋值为一个函数，函数执行逻辑就是修改textNode.data，这里通过counter = (counter + 1) % 2巧妙的实现0和1的切换。最后同样把isUsingMicroTask置为true。
  到这里我们就分析完了两种注册microtask的方法，但是如果浏览器完全不支持微任务的实现，我们只能选择降级处理方案，即注册macrotask。

else if (typeof setImmediate !== 'undefined' && isNative(setImmediate)) {
  // Fallback to setImmediate.
  // Technically it leverages the (macro) task queue,
  // but it is still a better choice than setTimeout.
  // 回退到setImmediate。从技术上讲，它利用了（宏）任务队列，但它仍然是比setTimeout更好的选择。
  timerFunc = () => {
    setImmediate(flushCallbacks)
  }
}

如上所示是一种macrotask注册实现，就是判断有没有原生setImmediate，有的话就将timerFunc定义为setImmediate(flushCallbacks)。从性能上来说setImmediate是优于setTimeout(fn, 0)的，因为setTimeout 在将回调注册为 (macro)task之前要不停的做超时检测，而setImmediate则不需要，而setTimeout(fn, 0)也不是立即执行的，它会有4ms默认延时。但是setTimeout不是一个标准特性，它只有在版本的ie浏览器以及10版本以上的node.js中被实现了。所以如果没有setImmediate的话，最后只能选择setTimeout了。其实之前版本中还有MessageChannel 实现，但是后来被废弃了，我们这里也不做介绍了。
  接下来我们看最后的选择

else {
  // Fallback to setTimeout.
  timerFunc = () => {
    setTimeout(flushCallbacks, 0)
  }
}

  现在你知道了js的水有多深了吧，我们最常用的方法setTimeout没想到最后成了最差的选择。
  最后我们来看flushCallbacks方法吧。

function flushCallbacks() {
  pending = false
  const copies = callbacks.slice(0)
  callbacks.length = 0
  for (let i = 0; i < copies.length; i++) {
    copies[i]()
  }
}

  flushCallbacks比较简单，其实就是将callbacks中每个函数取出来执行一次，但是这里是创建了callbacks的副本去执行，而callbacks在执行前就被清空了。pending在一开始也被重置为false。这意味着在nextTick中可以重新注册microtask或macrotask了，因为timerFunc函数执行条件是flushing为false，而timerFunc作用就是将flushCallbacks注册为microtask或者macrotask。
  nextTick其实是个非常有意思的东西，下面是它简化版实现，我们来测试一下它的有趣之处

const callbacks = []
let pending = false

function nextTick(cb, ctx) {
  let _resolve
  callbacks.push(() => {
    if (cb) {
       cb.call(ctx)
    } else if (_resolve) {
      _resolve(ctx)
    }
  })
  if (!pending) {
    pending = true
    timerFunc()
  }
  // $flow-disable-line
  if (!cb && typeof Promise !== 'undefined') {
    return new Promise(resolve => {
      _resolve = resolve
    })
  }
}
function flushCallbacks() {
  pending = false
  const copies = callbacks.slice(0)
  callbacks.length = 0
  for (let i = 0; i < copies.length; i++) {
    copies[i]()
  }
}
let counter = 1
const observer = new MutationObserver(flushCallbacks)
const textNode = document.createTextNode(counter)
observer.observe(textNode, {
  characterData: true
})
const timerFunc = () => {
  counter = (counter + 1) % 2

我们来写几个有意思的小例子
nextTick(() => {
  console.log(1)
})
const p = Promise.resolve()
p.then(() => {
  console.log(2)
})
nextTick(() => {
  console.log(3)
})

  猜猜它的执行结果是什么？它的结果是1 3 2。这都是因为我们的pending标识，导致我们第二个nextTick执行时不会调用timerFunc函数去注册异步队列。只是会将回调函数添加到callbacks队列中 。此时的microtask队列是这样的

[flushCallbacks, () => {
  console.log(2)
}]

  callbacks队列是这样的

[() => {
  console.log(1)
}, () => {
  console.log(3)
}]

  也就是执行flushCallbacks时候会执行callbacks中的所有方法，这里callbacks保存的是两个nextTick的回调函数，执行完flushCallbacks后才执行p.then的回调。也就是说同级的多个nextTick中只有第一个会产生一个异步队列，其他都nextTick回调都会变成一个同步的过程。这样做好处是显而易见的，就是在一次事件循环中Vue多次修改数据也只会产生一个microtask，但是也有它的弊端，比如上面这种写法会得到我们预期之外的结果。当然也不是没有解决办法，我们nextTick函数也是有两种写法的，一种是传入回调函数去调用，另一种是通过Promise方式调用，这是因为我们不传cb时会返回一个Promise对象，采用这种方式必然会创建micro task，我们来改写一下

nextTick(() => {
  console.log(1)
})
const p = Promise.resolve()
p.then(() => {
  console.log(2)
})
nextTick().then(() => {
  console.log(3)
})

  这样就符合预期了打印出了1 2 3，因为它们三个都是microtask，谁也不比谁优先执行。
  然后我们来看flushCallbacks函数一开始选择将pending设置为false，然后copy一份callbacks，这是为了能够实现nextTick嵌套，假如flushCallbacks一开始没有将pending设置为false，我们修改一下flushCallbacks的实现

function flushCallbacks() {
  for (let i = 0; i < callbacks.length; i++) {
    callbacks[i]()
  }
  pending = false
  callbacks.length = 0
}

  因为执行完flushCallbacks肯定要重置pending和callbacks，不然会以后就无法调用nextTick了，所以我们把这段逻辑放到后面，然后执行下面逻辑

nextTick(() => {
  console.log(1)
  const p = Promise.resolve()
  p.then(() => {
    console.log(2)
  })
  nextTick(() => {
    console.log(3)
  })
})

  这里期望的结果是输出1 2 3，但是这里执行内部nextTick时，pending = false，所以它只是在callbacks中push了() => { console.log(3) }，等到循环的最后一项时候就会执行它了，而p.then则是添加到microtask队列中，等flushCallbacks执行完成后才会执行，所以这里最终会输出1 3 2，为了解决这种问题，所以在flushCallbacks一开始就将pending重置，也就是允许这个时候注册microtask，这时如果有嵌套的nextTick时，它会在遍历执行callbacks某一个函数时调用nextTick，这样带来一个问题就是callbacks数组还没有清空，为了不影响下一次nextTick执行，还需要重置callbacks，但是直接重置callbacks的话就会影响本次for循环执行，所以这里通过slice创建了它的一个副本，这样遍历的是新创建的副本，这样直接清空callbacks就没有任何问题了。 这里设计的思路是同级的nextTick放在一个microtask中执行，非同一级nextTick放在不同的microtask中。
