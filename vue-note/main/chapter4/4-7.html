# 4.7 计算属性

  上一个小节我们讲解了Vue检测变化要注意的事项，同时也讲解数组响应式的实现，这个小节我们来讲解计算属性computed和侦听器watch。这两个属性是Vue的重要选项，如果对它们足够熟悉的话，你可以用它们写出十分优雅的代码，如果你对这两个选项不够了解的话，你可能会觉得这两个属性用处不大。下面我们先来看计算属性的实现。
  我们先来了解计算属性的实现，与我们前面分析响应式数据定义的data选项类似，计算属性的初始化也是在initState时进行的，这个其实之前分析时候是有提及的。在这之前我们需要先要阅读initState文档中对initComputed的分析。接下来我们分析一下在initState文档中未提到的内容。
  在执行initComputed函数中会执行defineComputed方法，这个方放也是通过Object.defineProperty劫持对象，我们正常定义的计算属性函数会作为其get拦截函数，这也是我们为什么定义计算属性函数的时候必须要有return。 但是我们知道计算属性也是响应式的，但是它不能直接修改自己的值，必须修改依赖项时它才会更新，显然我们定义的计算属性get没有这个能力，所以设置其get拦截时还需要做一些其他的事，这个我们之前是没有提到的。下面我就来看看defineComputed方法中出现的createComputedGetter方法，我们先来看源码：

function createComputedGetter (key) {
  return function computedGetter () {
    const watcher = this._computedWatchers && this._computedWatchers[key]
    if (watcher) {
      if (watcher.dirty) {
        watcher.evaluate()
      }
      if (Dep.target) {
        watcher.depend()
      }
      return watcher.value
    }
  }
}

  可以看到这里返回了一个函数computedGetter，这个应该是一个标准的闭包，既然是闭包那么就会有个受保护变量了，这里没有定义变量，但是有一个参数key，这里巧妙的通过闭包实现对key的持有，因为get函数是没有参数的，所以这里要通过闭包来实现，否则就获取不到key了。
  我们来看内层函数computedGetter实现，如果this._computedWatchers存在就使watcher = this._computedWatchers[key]，this._computedWatchers是一个通过Object.create(null)创建的对象，所以它是存在的，initState文档也讲过它里面存放着就是计算属性watcer，所以我们接着往下看。
  如果watcher存在才继续往下执行，我们这里是存在的。然后如果watcher.dirty为true则执行watcher.evaluate方法，这个条件也是满足的，所以我们来看evaluate方法实现：

  evaluate() {
    this.value = this.get()
    this.dirty = false
  }

  这里其实是执行了this.get()方法，然后要把dirty置为false。这里其实稍微有些复杂，我们先继续往下分析，这里执行get方法时执行pushTarget，实际上就是把Dep.target赋值为当前watcher的过程，也就是当前计算属性watcher，然后就会执行watcher.depend()，这个过程也就是收集依赖的过程了，你是不是觉得的depend方法有点熟悉，因为我们依赖收集小节已经讲过了。但这可能只是错觉，我们之前讲的都是dep的depend方法，在watcher中也定义了同名方法，我们来看它的实现：

depend() {
  let i = this.deps.length
  while (i--) {
    this.deps[i].depend()
  }
}

  如果你没能理解依赖收集的过程，那么这里也很难理解，因为依赖收集过程是比较绕的，它涉及dep和watcher方法的相互调用所以导致理解起来可能复杂，这里需要你有一定的理解能力才能读懂这个过程，这很难靠别人完成。但是在这里我尽可能将它简单的说明白，同时我也希望你能不看下面内容自己理清楚这块设计。其实依赖收集是一个双向的过程，我们的watcher会收集它订阅的依赖dep在它的dep数组中，dep也会收集它的订阅者在它的subs数组中。
  这块是最难理解也是设计最巧妙的地方。首先计算属性的watcher必须已经收集到dep，这个方法才会执行，这里有两个难点需要解决。

  1. 什么时候收集的dep？
  2. 为什么这里已经有依赖了还要再次收集？
      首先我们来解决第一个问题，我们访问计算属性时触发get拦截，也就是执行上面所述的computedGetter方法，computedGetter内部执行evaluate方法时会调用this.get方法，watcher的ge方法首先要执行pushTarget，这样Dep.target就指向了当前计算属性watcher。然后会调用this.getter，也就是我们书写的计算属性函数，这时候我们会依赖一些响应式数据，而访问响应式数据又触发了它的get拦截，从而进行了依赖收集，这时候调用了dep.depend方法就会将computed watcher收集到对应dep中，同时watcher也会收集到dep，那这样就完成了依赖收集了，这里我们就解决了第一个问题。
      下面我们来解决第二个问题，我们先不直接分析这个问题。我们先想一想上面依赖收集完成后是不是就结束了呢？有没有结束要看收集依赖后修改数据时能不能正常更新页面，我们假如修改了计算属性依赖项数据，这时候肯定会更新计算属性返回，然后我们发现好像没有然后了。为什么这么说呢？因为我们从来没有给计算属性做响应式拦截，计算属性也不会派发更新。也就是说数据更新了但是没有更新页面，这和没有收集依赖是一样的，或者说我们依赖没有收集完全。其实如果其他地方用了这部分数据也是没有问题的，原因其实很简单，如果这些依赖项只在计算属性中使用过的话，我们当前组件的渲染watcher根本不会订阅这些dep，原因出在Dep.target上，不得不说Dep.target设计非常巧妙，它是经常变化的，但这也导致一些隐蔽的问题，比如在计算属性中访问响应式数据触发get拦截时，Dep.target是被修改为computed watcher，而能触发页面更新的只有我们的渲染watcher。解决方法也是比较简单，再收集一次依赖就好了，watcher.depend就是做了这件事。再收集一次依赖有什么不同呢？当然也是Dep.target不同了，因为watcher的get方法调用结束后执行popTarget会将Dep.target重置为上一个值，也就是我们渲染watcher了，因为当前组件渲染更新其实都是在渲染watcher的get函数中进行的，所以对应Dep.target都是当前渲染watcer，因为计算属性调用watcher.get后只是会暂时修改Dep.target，而且执行完后立即恢复成了渲染watcher，所以再次收集依赖是有效的，当然这个过程会有重复收集的可能但是在watcher.addDep方法中的判断消灭了这种可能性。在computedGetter最后返回watcher.value也就是计算属性计算结果，这个就不展开分析了。
      关于这部分内容其实是比较新的逻辑，它在之前版本上做了很多修改，早期的depend也不长现在这样，所以这部分内容需要我们自己去分析理解，因为网上比较难找到最新的分析，这次修改使computed代码精简许多。
      到这里我们计算属性其实已经分析完了，但是我们不妨继续分析更新时会做哪些事看看有没有什么意外收获。响应式数据被修改时会调用dep.notify，然后会执行watcher.update方法
update() {
  /* istanbul ignore else */
  if (this.lazy) {
    this.dirty = true
  } else if (this.sync) {
    this.run()
  } else {
    queueWatcher(this)
  }
}

  这里this.lazy为true，将this.dirty置为true，这里置为true的话就可以重新调用get来计算新值了。这里其实也是非常细节的，我们之前收集依赖时，dep先收集了computed watcher，然后才收集渲染watcher，恰巧initComputed也是在当前激活的组件创建其渲染watcher之前的，这说明计算属性的id要比渲染watcher要小，顺序正确就比较好了，这里我们调用update更新时将计算属性watcher.get限制解开，这样就能重新进行计算了，什么时候能重新计算呢？也就是渲染watcher执行时候，模板中如果绑定了计算属性，那就会触发计算属性的get函数——computedGetter，从而重新计算新值。
  如果你对计算属性足够了解就会知道计算属性有一个缓存机制，只有修改了计算属性值才会触发计算属性的重新计算，也就是我们写的计算属性函数重新执行。这里的重新计算是指重新执行computed watcher的get函数，因为我们写的计算属性函数是作为watcher的getter传入的，并在watcher.get中执行的。那么这里缓存功能是怎么实现的呢？最新版的计算属性体现的并不明显，v2.5.17前会多次触发触发渲染，在2.5.17版本后会多次触发watcher.get然后对比新旧值从而触发更新，我们目前分析的是v2.6.11，最新的computed就和上述过程相同了。它既不会过多的进行计算也不会过多的进行更新，因为它本身是不能触发更新的。
  computed不具备更新能力，而是将更新能力放到了它的依赖项上，当依赖项更新时会触发dep.notify去派发更新，而notify方法其实遍历去调用订阅它的watcher的update方法，而如果遍历到计算属性的watcher时将其dirty设置为true。其实Vue中一般不会随便起名字，dirty在编程中一般表示脏数据也就是，也就是过期的数据。这里表面计算属性被修改了然后被标记为dirty，之后就能从新计算了。再往后遍历就会执行渲染watcher的update，这时会走else分支执行queueWatcher，也就是触发组件更新的方法然后就和上面所说的一样执行计算属性get函数然后修正计算属性值，这时它不再是dirty。之后再访问它都不会重新计算了，这就实现了其缓存能力，在evaluate方法中将this.value赋值为this.get方法返回值，也就是说计算属性的值是在watcher.value中缓存的。
