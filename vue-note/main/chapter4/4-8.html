# 4.8 侦听器

  在initSate中有这样一段初始化watch逻辑

if (opts.watch && opts.watch !== nativeWatch) {
  initWatch(vm, opts.watch)
}

  opts上存在watch配置并且watch不是js原生方法时才执行，因为火狐浏览器是自带watch属性的。
  我们来看initWatch方法

function initWatch (vm: Component, watch: Object) {
  for (const key in watch) {
    const handler = watch[key]
    if (Array.isArray(handler)) {
      for (let i = 0; i < handler.length; i++) {
        createWatcher(vm, key, handler[i])
      }
    } else {
      createWatcher(vm, key, handler)
    }
  }
}

  遍历watch对象，然后拿到其每一项作为handler，handler可以是一个数组，也可以不是一个数组，如果是一个 数组的话就遍历数组并调用createWatcher，否则直接调用createWatcher就好了。

function createWatcher (
  vm: Component,
  expOrFn: string | Function,
  handler: any,
  options?: Object
) {
  if (isPlainObject(handler)) {
    options = handler
    handler = handler.handler
  }
  if (typeof handler === 'string') {
    handler = vm[handler]
  }
  return vm.$watch(expOrFn, handler, options)
}

  我们来看一下createWatcher方法的实现，如果handler是一个纯对象，那就让options等于handler，然后使handler = handler.handler。如果hanlder是一个string类型，那就从vm实例上取这个东西。
  后return vm.$watch(expOrFn, handler, options)，接下来我们看$watch方法的实现。

Vue.prototype.$watch = function (
    expOrFn: string | Function,
    cb: any,
    options?: Object
  ): Function {
  const vm: Component = this
  if (isPlainObject(cb)) {
    return createWatcher(vm, expOrFn, cb, options)
  }
  options = options || {}
  options.user = true
  const watcher = new Watcher(vm, expOrFn, cb, options)
  if (options.immediate) {
    const info = `cllback for immediate watcher "${watcher.expression}"`
    pushTarget()
    invokeWithErrorHandling(cb, vm, [watcher.value], vm, info)
    popTarget()
  }
  return function unwatchFn () {
    watcher.teardown()
  }
}

  $watch方法是定义于state.js文件下stateMixin方法中，它被挂载到了Vue原型上，。该方法接三个参数，第一个参数expOrFn，可以是一个string或Function；第二个参数cb，可以是任意类型；第三个参数options是一个Object类型。
  如果cb是一个纯对象，那就return createWatcher()，如果不是一个对象那就正常执行下面逻辑如果options不存在就让它是空对象。然后把options.user置为true，也就是在new Watcher时把user置为true，也就是user watcher标识。
  然后new 一个Watcher保存在watcher中，这里传入的第二个参数不是function而是一个string，也就是我们要监听的数据key，第三个参数是我们的回调函数，第四个参数是我们的配置项options。user watcher的初始化过程相比计算属性watcher和渲染watcher有所不同，它第二个参数是一个字符串的话，那么watcher.getter就不是一个函数，第一个不同之处如下：

if (typeof expOrFn === 'function') {
  this.getter = expOrFn
} else {
  this.getter = parsePath(expOrFn)
  if (!this.getter) {
    this.getter = noop
    process.env.NODE_ENV !== 'production' && warn(
      `Failed watching path: "${expOrFn}" ` +
      'Watcher only accepts simple dot-delimited paths. ' +
      'For full control, use a function instead.',
      vm
    )
  }
}

  这里expOrFn是一个字符串所以会执行else逻辑，这个方法会对expOrFn做一个校验，也就是执行parsePath方法，如果它是合法字符串的话就会返回一个函数，这个函数功能就是取要监听的值。如果是返回undefined的话证明校验没通过，就在开发环境报一个警告，大致意思是：无法监视路径：xxx。Watcher只接受简单的点分隔路径。要完全控制，请改用函数。
  返回函数的话this.getter就变成一个取值函数，在构造函数最后会调用this.get()进行一次求值。调用this.get()方法时执行this.getter方法实际是执行取值函数，对响应式对象取值就会触发get()拦截，这个时候就会收集依赖了，这时候收集的watcher就是user watcher。以上就是初始化的过程，我接下来再分析执行过程。
  侦听器的回调函数执行时机是在监听的数据改变后执行，这个过程也非常简单，因为user watcher订阅了数据变化，只要数据改变就会触发user watcher更新，这里，更新最终执行的是Watcher.prototype.run()方法。

run() {
  if (this.active) {
    const value = this.get()
    if (
      value !== this.value ||
      // Deep watchers and watchers on Object/Arrays should fire even
      // when the value is the same, because the value may
      // have mutated.
      // 即使值相同，深度观察者和对象/数组上的观察者也应该激发，因为值可能发生了变化
      isObject(value) ||
      this.deep
    ) {
      // set new value
      const oldValue = this.value
      this.value = value
      if (this.user) {
        const info = `callback for watcher "${this.expression}"`
        invokeWithErrorHandling(this.cb, this.vm, [value, oldValue], this.vm, info)
      } else {
        this.cb.call(this.vm, value, oldValue)
      }
    }
  }
}

  这里执行this.get去重新求值，然后对比新旧值，如果发生了改变，则缓存旧值并将新值替换，如果this.user为true，则调用 invokeWithErrorHandling方法，下面我们来看它的定义

export function invokeWithErrorHandling (
  handler: Function,
  context: any,
  args: null | any[],
  vm: any,
  info: string
) {
  let res
  try {
    res = args ? handler.apply(context, args) : handler.call(context)
    if (res && !res._isVue && isPromise(res) && !res._handled) {
      res.catch(e => handleError(e, vm, info + ` (Promise/async)`))
      // issue #9511
      // avoid catch triggering multiple times when nested calls
      res._handled = true
    }
  } catch (e) {
    handleError(e, vm, info)
  }
  return res
}

  其实就是调用了回调函数cb，只不过是包裹在try…catc…中来进行错误处理。res可以是一个Promise对象，所以我们回调可以写async/await。到这里其实user watcher的实例化过程已经执行过程就已经讲解完了。下面我们来看watch的一些配置项。
  我们接着往下看$watch方法，其实就是对immediate配置的处理，也就是初始化时会执行一次回调，同样执行invokeWithErrorHandling方法，不同之处在于调用回调之前执行了pushTarget()，回调之后又调用popTarget()。这部分操作是做什么事呢？其实是防止收集依赖的，因为在初始化watch时还没有实例化渲染watcher，这时候Dep.target可能还是父组件的渲染watcher，如果这个时候回调函数中访问一些响应式数据就会触发依赖收集，因为initData是在initWatch之前的，所以这时候data中数据已经被劫持了。但是这个时候触发依赖收集的话它的依赖会被父组件渲染watcher订阅，这个显然是不对的，所以Vue做法是这个时候不进行依赖收集也就是执行pushTarget()，注意这里传入的参数为空，所以pushTarget接收的参数是undefined，这时候把Dep.target赋值为undefined就不会进行依赖收集了，执行完回调后还需要把Dep.target恢复为上一次的值，所以还要执行一次popTarget()，这样就没有问题了。
  我们接着把$watch看完，最后它返回一个unwatchFn函数，这是Vue提供的一个取消侦听的api，逻辑如下：

teardown() {
  if (this.active) {
    // remove self from vm's watcher list
    // this is a somewhat expensive operation so we skip it
    // if the vm is being destroyed.
    // 从vm的观察者列表中删除self这是一个有点昂贵的操作，因此如果vm正在被销毁，我们将跳过它。
    if (!this.vm._isBeingDestroyed) {
      remove(this.vm._watchers, this)
    }
    let i = this.deps.length
    while (i--) {
      this.deps[i].removeSub(this)
    }
    this.active = false
  }
}

  这个方法定义也是十分简单，如果组件没有开始销毁的话，就找到当前组件实例上的_watchers列表将当前user watcher删除，这个操作开销是有点大的，所以如果组件开始卸载的话就不做这个操作了。
接下来就找到当前user watcher收集的依赖列表this.deps，循环调用removeSub方法去将watcher从订阅者列表移除，最后  再把this.active置为false，这样可以防止一些异步的更新去执行run方法。   我们写watch监听时还有个常用配置deep，我们来看deep属性在哪些逻辑中使用过，首先是watcher.get()方法

// "touch" every property so they are all tracked as
// dependencies for deep watching
// “触摸”每个属性，因此它们都作为依赖项进行跟踪，以便进行深度监视
if (this.deep) {
  traverse(value)
}

  如果this.deep为true则执行traverse方法，并传入this.getter方法计算的值value，traverse方法定义于observer/ traverse.js文件下

const seenObjects = new Set()

/**
 * Recursively traverse an object to evoke all converted
 * getters, so that every nested property inside the object
 * is collected as a "deep" dependency.
 */
// 递归遍历对象以调用所有已转换的getter，从而将对象中的每个嵌套属性作为“深度”依赖项收集
export function traverse (val: any) {
  _traverse(val, seenObjects)
  seenObjects.clear()
}

  这里先是定义了一个全局变量seenObjects是一个set数组，traverse方法就是调用_traverse方法，然后清空seenObjects。
  下面我们开看_traverse方法定义：

function _traverse (val: any, seen: SimpleSet) {
  let i, keys
  const isA = Array.isArray(val)
  if ((!isA && !isObject(val)) || Object.isFrozen(val) || val instanceof VNode) {
    return
  }
  if (val.__ob__) {
    const depId = val.__ob__.dep.id
    if (seen.has(depId)) {
      return
    }
    seen.add(depId)
  }
  if (isA) {
    i = val.length
    while (i--) _traverse(val[i], seen)
  } else {
    keys = Object.keys(val)
    i = keys.length
    while (i--) _traverse(val[keys[i]], seen)
  }
}

  这里接收两个参数，第一个参数val也就是我们传入的value，第二个参数seen，也就是我们定义的seenObjects。
  这里判断val不是一个数组并且不是对象或者对象被冻结，或者val是VNode的实例的话就什么都不去做直接return。
  如果val.__ob__存在说明val是响应式的，然后就获取val.__ob__.dep.id缓存在depId中，如果seen中已经有depId了就什么都不做return，否则就将depId添加到set中。
  再往下判断如过val是数组，然后就遍历数组去递归调用_traverse，否则的话它是一个对象那就遍历对象去调用_traverse。
  这里有一个非常奇怪的地方，那就是seenObjects到底有什么用，_traverse方法作用是毋庸置疑的，即使你看不明白，但是注释已经说的很明白了，就是挨个访问一遍数据然后触发get拦截去收集依赖，这样就做到了深度监听，当然这个缺陷也很明显，当你对象比较庞大的时候，这将会是很大的性能开销。这里最令人迷惑的就是seenObjects，添加完所有的depId后也不做保存，最后直接又清空了。而理论上是不会出现重复的depId，Vue自己也是这么认为的，所以一旦出现重复depId就会跳出递归。然而你永远不知道用户会做出什么操作，下面我来证明这一点。

  obj: {
    data: obj
  }

  我们定义这样一个obj对象，其data属性又引用了obj，这是一个循环引用。这种情况下去做深度监听会造成无限递归的死循环，但是有了seenObjects就不怕这种情况发生了。
  到这里我们还没有讲完，这是在this.deep为true时执行watcher的get方法不同，在更新时调用的run()方法也有所不同：

if (
  value !== this.value ||
  // Deep watchers and watchers on Object/Arrays should fire even
  // when the value is the same, because the value may
  // have mutated.
  // 即使值相同，深度观察者和对象/数组上的观察者也应该激发，因为值可能发生了变化
  isObject(value) ||
  this.deep
) {
  // set new value 
}

  我们只看这部分判断逻辑，它在两次计算的value不同时才执行，其实在value是对象的时候或者深度监听的时候也会执行这是什么意思呢？首先如果value是一个基础类型数据的话其实在响应式数据set拦截时已经判断过了，如果两次的值相同的话是不会触发更新的，所以这里value !== this.value其实不是为我们user watcher准备的，它在其他类型的watcher比较有用。
  我们再看isObject(value)的判断，对于这种情况下我写一个小例子去验证它，这种情况应该是比较少见的，我也没想到其他例子能触发它，对于对象只有其引用地址改变才会触发更新，但是这样就满足value不等的条件了，如果改变对象的某个属性，因为不是深度监听也不会触发改变。但是还有一种方法可以修改对象，那就是为对象添加属性，也就是我们去调用this.$set方法去添加对象属性，我们之前说过这相当于修改了对象。如果是一个数组的话，只需要执行push方法就可以了。这两种情况下value值是不会改变的，但是数组和对象都发生了改变，应该去执行if内部逻辑。

new Vue({
  el: '#app',
  data: {
    obj: {
      a: 1
    },
    a: []
  },
  watch: {
    obj() {
      console.log('检测到了obj')
    },
    a() {
      console.log('检测到了a')
    }
  },
  mounted() {
    this.$set(this.obj, 'b', 2)
    this.a.push(2)
  }
})

  对于this.deep的判断的必要性有待商榷。到这里我们对于deep选项的配置就分析完毕了。接下来我们分析watch的sync配置。
  正常watch是异步执行，这是为了一些性能考虑，但是在一些场景下我们可能需要它同步执行，这时我们可以将它的sync属性配置为true

watch: {
  a: {
    handler() {},
    sync: true
  }
}

  这个逻辑也非常简单，在watcher的update方法中

update() {
  /* istanbul ignore else */
  if (this.lazy) {
    this.dirty = true
  } else if (this.sync) {
    this.run()
  } else {
    queueWatcher(this)
  }
}

  如果this.sync为true就直接执行this.run方法，而不是调用queueWatcher方法去异步触发run方法。到这里其实sync配置就讲完了，是不是非常简单。其实这里还可以做个扩展。
  Vue官方有个vue-test-utils的测试工具库，当你使用它测试时.vue文件时，数据会以同步方式触发组件变更，这对测试而言会提供很大的便利。那么它是怎么实现这个功能的呢？我们在开发过程极少去使用sync配置，那么vue—test-utils就需要把所有组件的watcher都添加sync：true。这在恐怕在无人干预的情况下很难做到，如果人为去修改那将是繁琐且容易引起bug的。为了解决这个问题vue提供了Vue.config.async全局配置，默认为true，代表是异步更新，若修改它为false则变为同步过程。在core/config.js可以看到下面配置：

  /**
   * Perform updates asynchronously. Intended to be used by Vue Test Utils
   * This will significantly reduce performance if set to false.
   */
  async: true,

  我们来看它如何实现对全局的控制，在observer/scheduler.js中有个queueWatcher方法，其中有这样一段逻辑

if (process.env.NODE_ENV !== 'production' && !config.async) {
  flushSchedulerQueue()
  return
}
nextTick(flushSchedulerQueue)

  这里有对!config.async判断，我们之前没有对它展开分析，也就是async被设置为false时并且在开发环境会执行其逻辑，也就是直接执行flushSchedulerQueue，而不去通过nextTick异步执行。   但是到这里还没有完，这样是有bug的，这bug是在dep.notify方法中被解决：

notify() {
  // stabilize the subscriber list first 首先稳定订阅者列表
  const subs = this.subs.slice()
  if (process.env.NODE_ENV !== 'production' && !config.async) {
    // subs aren't sorted in scheduler if not running async
    // we need to sort them now to make sure they fire in correct
    // order
    // SUB没有在调度程序中排序。如果没有运行异步，我们需要现在对它们进行排序，以确保它们以正确的顺序启动
    subs.sort((a, b) => a.id - b.id)
  }
  for (let i = 0, l = subs.length; i < l; i++) {
    subs[i].update()
  }
}

  我们再dep.subs数组中会收集它的订阅者，然后触发更新的时候会遍历subs数组调用update方法，这时候由于我们收集依赖时是根据我们访问数据的顺序去收集的，这个时候subs中的watcher不是按id排序的，如果没有按id顺序去更新的话就可能出现问题，所以更新前需要排序。但是我们正常会是一个异步过程，也就是watcher不会直接更新，而是现将触发更新的watcher都放到一个queue数组中，等所有同步逻辑执行完成后执行flushSchedulerQueue方法的时候先进行排序后再进行更新。如果变成同步逻辑的话，每次调用subs[i].update()时候都触发flushSchedulerQueue同步执行，所以排序就没有了意义，因为每次queue队列只会有一个watcher，这样更新顺序就依赖于subs数组中元素的顺序了，所以就只能在遍历之前增加一个排序操作。
  那为什么异步执行的排序不直接放在notify中呢？那是因为一次notify只能处理一个依赖更新，而一个同步任务中可能会有多个依赖发生改变，所以异步执行的时候只能放在flushSchedulerQueue中进行排序才能保证最后更新顺序无误。
