# 5.14 generate代码生成（4）

  上小节我们分析了v-if和v-once的代码生成过程，这个小节我们再看看其他分支内容，首先我们写如下一个demo

new Vue({
  el: '#app',
  data: {
    styl: {
      width: '100px',
      background: '#000',
    },
    b: 'c'
  },
  template: `
  <div class="a" :class="b" style="background: red" :style="styl">
    <p v-for="item in 4" v-if="true" v-once>1</p>
  </div>`
})

  这里使用了for循环，又使用了v-if指令和v-once指令，这和我们之前的分析过程没有什么不同，我们在genOnce中对for循环的处理是基于v-once包含在v-for指令标签内的，我们这里是同级别的，所以并不会走到那部分逻辑，所以这段模板差异主要在于p标签使用的v-for指令，至于外层div我们可以暂时去忽略它，它主要是用来研究我们genData的。
  在处理p元素时调用完genOnce最后会执行genStatic，genStatic中实际上还会调用genElement方法，这样判断到el.for就会执行genFor逻辑：

export function genFor (
  el: any,
  state: CodegenState,
  altGen?: Function,
  altHelper?: string
): string {
  const exp = el.for
  const alias = el.alias
  const iterator1 = el.iterator1 ? `,${el.iterator1}` : ''
  const iterator2 = el.iterator2 ? `,${el.iterator2}` : ''

  if (process.env.NODE_ENV !== 'production' &&
    state.maybeComponent(el) &&
    el.tag !== 'slot' &&
    el.tag !== 'template' &&
    !el.key
  ) {
    state.warn(
      `<${el.tag} v-for="${alias} in ${exp}">: component lists rendered with ` +
      `v-for should have explicit keys. ` +
      `See https://vuejs.org/guide/list.html#key for more info.`,
      el.rawAttrsMap['v-for'],
      true /* tip */
    )
  }

  el.forProcessed = true // avoid recursion
  return `${altHelper || '_l'}((${exp}),` +
    `function(${alias}${iterator1}${iterator2}){` +
      `return ${(altGen || genElement)(el, state)}` +
    '})'
}

  首先将el.for保存在exp中，我们知道el.for就是解析出来的要遍历的数组或对象的那个变量名。然后获取alias，alias就是我们遍历项的别名item。至于iterator这里是没有的。
  再往下它做了一个简单判断，如果是在组件中并且key不存在，那么会报一个警告，容我们必须去为v-for定义key，至于普通元素标签就不做强制限制了。至于我们平常写普通标签时不写key也会标红之类的，那其实不是vue给我们的提示，可能是一些编辑器插件或者eslist插件去给我们报的错误。
  最就是将el.forProcessed设置为true，避免无限循环，最后生成的代码就是一个_l函数，它的第一个参数是exp，第二个参数则是一个函数，这个函数return的是一段genElement生成的代码，最后结果是_l((4),function(item){return _c('p')})。这就是对于v-for的处理结果了，当然后续还有v-once处理静态的结果我们就不去分析了。接下来我们来看看这个_l函数。

  target._l = renderList

  这是专门用来处理我们模板循环的函数

export function renderList (
  val: any,
  render: (
    val: any,
    keyOrIndex: string | number,
    index?: number
  ) => VNode
): ?Array<VNode> {
  let ret: ?Array<VNode>, i, l, keys, key
  if (Array.isArray(val) || typeof val === 'string') {
    ret = new Array(val.length)
    for (i = 0, l = val.length; i < l; i++) {
      ret[i] = render(val[i], i)
    }
  } else if (typeof val === 'number') {
    ret = new Array(val)
    for (i = 0; i < val; i++) {
      ret[i] = render(i + 1, i)
    }
  } else if (isObject(val)) {
    if (hasSymbol && val[Symbol.iterator]) {
      ret = []
      const iterator: Iterator<any> = val[Symbol.iterator]()
      let result = iterator.next()
      while (!result.done) {
        ret.push(render(result.value, ret.length))
        result = iterator.next()
      }
    } else {
      keys = Object.keys(val)
      ret = new Array(keys.length)
      for (i = 0, l = keys.length; i < l; i++) {
        key = keys[i]
        ret[i] = render(val[key], key, i)
      }
    }
  }
  if (!isDef(ret)) {
    ret = []
  }
  (ret: any)._isVList = true
  return ret
}

  这里可以看到renderList接收了两个参数，一个val，另一个render函数。这里首先判断val是数组或者是字符串，然后根据val的长度new一个新的Array保存在ret中。接下来就要遍历val，实际上我们字符串也是可以通过str[0]这种形式去访问对应索引位置字符的，所以就要执行我们render函数，并将遍历值作为第一个参数传入，索引作为第二个参数传入。
  第二个判断则是判断是一个数字，这时会根据数字作为数组长度来new新数组，最后同样循环调用render，并将render生成的vnode都添加到ret数组。
  第三个判断则稍微有些复杂，是对对象的遍历的判断。首先要判断浏览器是否支持Symbol，并且val是否定义了Symbol.iterator方法。如果定义了就要对自定义迭代器做一个处理，关于迭代器我们这里就不做过多介绍了。然后我们看没有自定义迭代器时的处理，也就是我们一般情况下的处理，跟Object.keys数组的长度去新建一个数组进行遍历，这时的render函数会传入三个参数，分别是value、key、index。
  接下来这个判断有点莫名奇妙isDef是判断非undefined且非null，前面加非实际上就是判断是undefined或者null，vue也定义了专门的判断方法isUndef，这里为什么非要这么绕呢。这里就是保证ret是一个数组。最后给ret打上_isVList标识后返回。这就是我们_l函数的全部内容了。
  接下来我们去走一下div的genData逻辑，因为我们定义了class和style，所以这里会执行

  // module data generation functions
  for (let i = 0; i < state.dataGenFns.length; i++) {
    data += state.dataGenFns[i](el)
  }

  对于class会生成"staticClass:"a",class:b,"，对于style会生成"staticStyle: {"background":"red"},style:(styl),"，最终我们生成代码如下： "_c('div',{staticClass:"a",class:b,staticStyle:{"background":"red"},style:(styl)},(true)?_m(0):_e(),0)"
  如果我们在v-for模板上添加ref属性就会执行genData中这两段逻辑：

  // ref
  if (el.ref) {
    data += `ref:${el.ref},`
  }
  if (el.refInFor) {
    data += `refInFor:true,`
  }

  最后生成这样一个data："{ref:"a",refInFor:true}"。   接下来我们试试genComponent逻辑吧，再来写个小demo

const child = {
  name: 'child',
  template: `<p>child</p>`
}
new Vue({
  el: '#app',
  components: { child },
  template: `
  <div is="child">1</div>`
})

  给div添加一个is=child，这样就会渲染child组件了。这时我们el.component会等于child，那么在执行genElement方法时就会执行到genComponent

// componentName is el.component, take it as argument to shun flow's pessimistic refinement
function genComponent (
  componentName: string,
  el: ASTElement,
  state: CodegenState
): string {
  const children = el.inlineTemplate ? null : genChildren(el, state, true)
  return `_c(${componentName},${genData(el, state)}${
    children ? `,${children}` : ''
  })`
}

  然后这里会执行genChildren方法，这里会找div下有没有其他节点，这里我们是有一个文本1的，所以children这里会执行genChildren得到"[_v("1")]"，最后生成代码时会调用genData，genData方法会执行这样一段逻辑

  // record original tag name for components using "is" attribute
  if (el.component) {
    data += `tag:"${el.tag}",`
  }

  记录原本的tag。
  最后生成的代码块是"_c("child",{tag:"div"},[_v("1")])"，到这里就结束了，但是我们组件的模板是什么时候解析呢？那就要等上面render函数执行，解析到tag是组件child标签时才会初始化child组件并根据其模板<p>child</p>生成其render函数。那么最后我们来看页面发现只生成了<p>child</p>，原本的div被组件根标签p替换了。但是文本1被移除了，其实这是组件处理的结果，因为组件标签就是会忽略非插槽中的所有中间文本。假如将child中添加一个slot标签，那么文本1就会被渲染

const child = {
  name: 'child',
  template: `<p>child<slot/></p>`
}

  就会渲染child1到页面。
  我们再来看一个动态参数属性处理，这个东西我们上面分析genProps漏过了它，使用动态参数时这个属性会被添加到el.dynamicAttrs中

new Vue({
  el: '#app',
  data: {
    a: 't',
    b: 'b',
    c: {
      e: 1,
      f: 2
    }
  },
  template: `
  <div v-bind:[a]="b" v-bind="c">
    a
  </div>`
})

  这个时候执行genData时会执行

  // v-bind dynamic argument wrap
  // v-bind with dynamic arguments must be applied using the same v-bind object
  // merge helper so that class/style/mustUseProp attrs are handled correctly.
  if (el.dynamicAttrs) {
    data = `_b(${data},"${el.tag}",${genProps(el.dynamicAttrs)})`
  }

  这里会生成一个_b方法，再执行genProps，genProps我们上一小节已经分析过了，这里我们主要看_b方法

  target._b = bindObjectProps

  _b方法实际上是bindObjectProps方法，我们先不看这个方法，先继续看，这里我们有个v-bind="c"，这里的v-bind不同于我们v-bind:指令，我们解析指令是会单独处理v-bind:的，但是这里v-bind是会当成一个普通指令来处理的。我们在generate中首先是new CodegenState生成一个state，在这个过程中会在state中加入各种指令方法，也包括我们自定义的指令

  this.directives = extend(extend({}, baseDirectives), options.directives)

  我们之前也提到过这里面包括一个bind指令。在parse解析到v-bind="c"时，把其当做一个普通的指令添加到el.directives中，我们知道像v-if、v-for、v-bind等指令是不会被添加到el.directives中的。
  那么在执行genData就有所不同了，genData函数开头会执行

  // directives first.
  // directives may mutate the el's other properties before they are generated.
  const dirs = genDirectives(el, state)

  这个方法定义如下

function genDirectives (el: ASTElement, state: CodegenState): string | void {
  const dirs = el.directives
  if (!dirs) return
  let res = 'directives:['
  let hasRuntime = false
  let i, l, dir, needRuntime
  for (i = 0, l = dirs.length; i < l; i++) {
    dir = dirs[i]
    needRuntime = true
    const gen: DirectiveFunction = state.directives[dir.name]
    if (gen) {
      // compile-time directive that manipulates AST.
      // returns true if it also needs a runtime counterpart.
      needRuntime = !!gen(el, dir, state.warn)
    }
    if (needRuntime) {
      hasRuntime = true
      res += `{name:"${dir.name}",rawName:"${dir.rawName}"${
        dir.value ? `,value:(${dir.value}),expression:${JSON.stringify(dir.value)}` : ''
      }${
        dir.arg ? `,arg:${dir.isDynamicArg ? dir.arg : `"${dir.arg}"`}` : ''
      }${
        dir.modifiers ? `,modifiers:${JSON.stringify(dir.modifiers)}` : ''
      }},`
    }
  }
  if (hasRuntime) {
    return res.slice(0, -1) + ']'
  }
}

  如果el.directives不存在的话是什么都不会做，在这里dirs只有一个元素

{
  arg: null,
  isDynamicArg: false,
  modifiers: undefined,
  name: "bind",
  rawName: "v-bind",
  value: "c"
}

  接下来遍历dirs，将当前遍历项保存到dir，将needRuntime设置为true，然后就是去state.directives中取同名指令函数，这里也就是bind函数。
  如果我们获取到指令的定义那么就运行它，如果我们只是在编译时处理那么gen函数就不需要有返回值，那样执行到这里就停止。如果说运行时还需要对应的指令项，那gen就应该返回一个true并赋值给needRuntime。
  这样接下来判断needRuntime为true就会将hasRuntime置为true，并且将res处理为一个directives属性字符串最终返回给data。所以这里是可以控制最后是否返回data的directives项的。
  我们来看bind指令函数定义

export default function bind (el: ASTElement, dir: ASTDirective) {
  el.wrapData = (code: string) => {
    return `_b(${code},'${el.tag}',${dir.value},${
      dir.modifiers && dir.modifiers.prop ? 'true' : 'false'
    }${
      dir.modifiers && dir.modifiers.sync ? ',true' : ''
    })`
  }
}

  就是为el添加一个wrapData，显然是没有返回值的，所以这里不需要在运行时保留其指令项。这里同样也是生成_b函数的调用，这里参数比较多，第一个是wrapData调用时传入的code，第二个则是元素tag名，第三个是指令value，第四个是标记是否有.prop修饰符，第五个标记是否有.sync修饰符。
  这样在genData函数最后会执行这样一段逻辑

  // v-bind data wrap
  if (el.wrapData) {
    data = el.wrapData(data)
  }

  这里会传入data，也就是wrapData接收的code，然后我们来看_b方法，前面已经说过_b方法就是bindObjectProps方法。
  我们来看它的定义

/**
 * Runtime helper for merging v-bind="object" into a VNode's data.
 */
export function bindObjectProps (
  data: any,
  tag: string,
  value: any,
  asProp: boolean,
  isSync?: boolean
): VNodeData {
  if (value) {
    if (!isObject(value)) {
      process.env.NODE_ENV !== 'production' && warn(
        'v-bind without argument expects an Object or Array value',
        this
      )
    } else {
      if (Array.isArray(value)) {
        value = toObject(value)
      }
      let hash
      for (const key in value) {
        if (
          key === 'class' ||
          key === 'style' ||
          isReservedAttribute(key)
        ) {
          hash = data
        } else {
          const type = data.attrs && data.attrs.type
          hash = asProp || config.mustUseProp(tag, type, key)
            ? data.domProps || (data.domProps = {})
            : data.attrs || (data.attrs = {})
        }
        const camelizedKey = camelize(key)
        const hyphenatedKey = hyphenate(key)
        if (!(camelizedKey in hash) && !(hyphenatedKey in hash)) {
          hash[key] = value[key]

          if (isSync) {
            const on = data.on || (data.on = {})
            on[`update:${key}`] = function ($event) {
              value[key] = $event
            }
          }
        }
      }
    }
  }
  return data
}

  这里首先要保证我们传入的参数value为object，如果不为object的话则报一个警告。对于我们demo来说处理v-bind:[a]="b"属性后data为"_b({},"div",_d({},[a, b]))"，这里_d函数我们已经分析过了，最后会返回一个object类型的属性列表。而我们定义v-bind后面的内容也会绑定一个对象或数组。
  value正常会执行else中的逻辑，首先判断value是不是一个数组，如果是数组会调用toObject方法并赋值给value，toObject方法会将对象集合合并为一个对象。这里就是保证value是一个对象。
  接下来通过for in遍历value，然后会检查是否是class或style或者一些vue中保留的属性，如果是这样的属性就将data赋值给hash。如果不是这些原生的属性或保留属性的话执行else逻辑。
  else中会先获取type属性，如果asProp为true或者config.mustUseProp检查为true的话，也就是有.prop修饰符或者config.mustUseProp检查通过，那就要把domProps赋值给hash，否则把attrs赋值给hash。
  然后分别对key转驼峰和连字符，如果hash中不存在其驼峰形式和连字符形式的属性名的话才将hash[key]赋值为value[key]。
  最后如果是.sync修饰符的话，就会添加一个update:key的事件监听，然后监听方法就是将value[key]修改为监听到的值。
  其实这个方法整体来看也比较简单，就是将属性都添加到attrs或者domProps中，当然对于style、class这种属性是和data.attrs同一级的，所以为hash赋值的时候要赋值为data。
  我们可以看到对于动态参数v-bind:[key]而言，它既会调用_d函数又会调用_b函数，那么它们分别在执行过程中扮演了什么角色呢。实际上_d函数只是把动态参数属性和静态参数属性一股脑放在一个对象中，而不会去添加到data中，它只是负责去取这个动态的key值。而_b函数则是会将这些属性做细致的划分，并将其添加到data对应的属性中。
  我们最后再看处理完v-bind:[a]="b"后处理v-bind="c"会生成什么样的结果，因为这里的两个分支都是直接去替换data的，而不是在原本data上 += 其他字符串。可以看到这里是又嵌套了一层_d方法"_b(_b({},"div",_d({},[a,b])),'div',c,false)"。这也是符合预期的，就是把上一步处理完的data再次交给_b方法做进一步的处理，最后再返回data。
  生成代码后对应流程其实就是我们第1、2小节的内容了，baseCompile方法执行完毕返回

  return {
    ast,
    render: code.render,
    staticRenderFns: code.staticRenderFns
  }

  baseCompile函数是在compile中调用的，compile得到baseCompile返回这个对象后compile为它添加一些编译过程中的错误回调后返回给compiled。compile方法又是在compileToFunctions中调用，compileToFunctions拿到compiled就对它处理错误并将其生成真正可执行的代码

    const res = {}
    const fnGenErrors = []
    res.render = createFunction(compiled.render, fnGenErrors)
    res.staticRenderFns = compiled.staticRenderFns.map(code => {
      return createFunction(code, fnGenErrors)
    })

  最后将res缓存并返回。最后$mount中的到就是res对象，解构出的render就是真正可执行的render函数，已经静态缓存的render函数数组staticRenderFns。到此编译结束。之后的处理过程就和我们手写render函数的过程一致了，所以说vue其实是只认render函数的，我们即使是写模板也会被转换为render函数最终才能处理。   到这里我们关于代码生成大部分内容基本就都分析完毕了，其实我们会发现genData函数中还有很多分支没有去分析，还有genHandlers、genScopedSlots等函数没有去分析，还有一些v-html、v-text等指令没有涉及到。这部分内容我们将会放到下一章节专门去分析指令。
