# 5.12 generate代码生成（2）

  这个小节开始我们来分析genElement中的gen系列函数，我们知道genElement中有很多if、else if逻辑，但是那都是处理特殊情况下的逻辑，真正普通状态下的逻辑其实是在else中的

  } else {
    // component or element
    let code
    if (el.component) {
      code = genComponent(el.component, el, state)
    } else {
      let data
      if (!el.plain || (el.pre && state.maybeComponent(el))) {
        data = genData(el, state)
      }
      const children = el.inlineTemplate ? null : genChildren(el, state, true)
      code = `...`
    }
    return code
  }

  可以看到他是处理component和element的，但是我们不先来分析它，我们从代码编写最简单的情况开始分析，也就是我们只写一些静态节点

new Vue({
  el: '#app',
  template: `<div v-pre>
    <p>1</p>
    <p>2</p>
  </div>`
})

  这是一个极简的例子，一个根div标签下有两个p标签，并且p中各自有一个文本节点，我们来分析它的执行过程，这里我们直接给div绑定v-pre指令，直接标记它是静态节点就可以，这可以跳过很多判断，即使不去写v-pre那么它也会被标记为静态根。
  在代码生成过程中调用genElement方法时会执行

  if (el.parent) {
    el.pre = el.pre || el.parent.pre
  }
  if (el.staticRoot && !el.staticProcessed) {
    return genStatic(el, state)
  }

  这里el.pre本身就是true，因为div是跟节点，el.parent是不存在的。然后显然这个节点会被标记为staticRoot，注意这里只有div被标记为静态根，因为p标签中只有一个文本子节点，这在5.10小节我们已经了分析过了，这种情况不会被标记。然后它就会执行到第一个if分支的genStatic方法

// hoist static sub-trees out
function genStatic (el: ASTElement, state: CodegenState): string {
  el.staticProcessed = true
  // Some elements (templates) need to behave differently inside of a v-pre
  // node.  All pre nodes are static roots, so we can use this as a location to
  // wrap a state change and reset it upon exiting the pre node.
  const originalPreState = state.pre
  if (el.pre) {
    state.pre = el.pre
  }
  state.staticRenderFns.push(`with(this){return ${genElement(el, state)}}`)
  state.pre = originalPreState
  return `_m(${
    state.staticRenderFns.length - 1
  }${
    el.staticInFor ? ',true' : ''
  })`
}

  首先要把el.staticProcessed设置为true，避免重复地处理。然后有一段注释，大致意思是要保存一个pre状态在state.pre中，用完之后要将其恢复。所以这里我们先要把state.pre状态保存下来，然后根据el.pre状态去设置state.pre。在这里el.pre为true，所以会将state.pre修改为true。但是originalPreState会保存state.pre原本的状态false。
  然后会往state.staticRenderFns数组中添加这么一个字符串代码，这段代码可以看到是以with开头的，它会改变作用域链，将使代码执行时优先去with绑定的对象中去寻找变量。这里用with绑定了this，这也就是为什么我们之前说为什么明明_c是vm实例上的方法，但是生成的代码却直接调用它，就是因为这里用了with语法。
  我们可以看到，这里其实是递归调用了genElement方法，然后会将state重置为false，最后返回一个_m函数的调用。我们先来看第二次调用genElement方法会发生什么。这里我们还是去处理div元素，但是由于这时el.staticProcessed为true了，所以不会再次进入if判断了，然后接下来的各种特殊情况，比如有v-once、v-if、v-for等指令的情况，显然也都是不存在的，所以会直接进入上面说的else逻辑。
  接下来其实就是会执行genData逻辑了

export function genData (el: ASTElement, state: CodegenState): string {
  let data = '{'
  // pre
  if (el.pre) {
    data += `pre:true,`
  }
  // module data generation functions
  for (let i = 0; i < state.dataGenFns.length; i++) {
    data += state.dataGenFns[i](el)
  }
  
  data = data.replace(/,$/, '') + '}'
  return data
}

  其实genDate逻辑十分简单，就是拼接了一个对象字符串后返回，这个方法还是十分复杂的，有很多的if控制，但是我们没有办法一步步去分析，因为那是非常混乱的，因为很多分支都是互斥的，比如绑定v-pre就会导致很多编码环节被跳过，所以很多分支也不会执行，比如我们现在的demo就只会执行这些逻辑，实际上就是生成一个{pre: true}对象字符串返回。虽然这个方法很复杂，但是我们可以尽量写一些demo去覆盖它的各个分支，这样我们用到时候再来分析它，很显然这里执行genData时只走了这些逻辑。
  如果我们连v-pre指令都去掉的话，这时候div其实会被标记为plain，也就是纯的标签，不含有任何属性。这个时候甚至不执行genData函数，而是会执行

const children= el.inlineTemplate ? null : genChildren(el, state, true)

  这里如果是内联模板的话显然不能直接作为子节点的，所以会赋值为null，但是如果是正常节点的话会执行genChildren方法，接下来我们就来看这个方法。

export function genChildren (
  el: ASTElement,
  state: CodegenState,
  checkSkip?: boolean,
  altGenElement?: Function,
  altGenNode?: Function
): string | void {
  const children = el.children
  if (children.length) {
    const el: any = children[0]
    // optimize single v-for
    if (children.length === 1 &&
      el.for &&
      el.tag !== 'template' &&
      el.tag !== 'slot'
    ) {
      const normalizationType = checkSkip
        ? state.maybeComponent(el) ? `,1` : `,0`
        : ``
      return `${(altGenElement || genElement)(el, state)}${normalizationType}`
    }
    const normalizationType = checkSkip
      ? getNormalizationType(children, state.maybeComponent)
      : 0
    const gen = altGenNode || genNode
    return `[${children.map(c => gen(c, state)).join(',')}]${
      normalizationType ? `,${normalizationType}` : ''
    }`
  }
}

  先判断有没有children，如果有的话才继续往下执行。先取children数组的第一个元素保存在新定义的一个el变量中。
  接下来会判断一个单个v-for子节点的情况，当children数组长度为1，并且其子元素绑定了v-for指令，并且不是template和slot这种可能多个子节点的标签，那就会做进一步处理，首先是根据传入的第三个参数checkSkip做三元判断，这里我们传入的是true。如果它不为true的话normalizationType会是一个空字符串，这里为true，那么就会判断el是否是一个组件，是组件就返回,1，否则返回,0。
  最后返回altGenElement或genelement函数的调用，并拼接normalizationType，这里没有定义altGenElement，所以实际上又递归调用genElement。
  继续往下看是对一般情况下normalizationType的定义，还是通过checkSkip做三元表达式判断，返回getNormalizationType执行结果或者0。
getNormalizationType其实是定义标准化级别

// determine the normalization needed for the children array.
// 0: no normalization needed
// 1: simple normalization needed (possible 1-level deep nested array)
// 2: full normalization needed
function getNormalizationType (
  children: Array<ASTNode>,
  maybeComponent: (el: ASTElement) => boolean
): number {
  let res = 0
  for (let i = 0; i < children.length; i++) {
    const el: ASTNode = children[i]
    if (el.type !== 1) {
      continue
    }
    if (needsNormalization(el) ||
        (el.ifConditions && el.ifConditions.some(c => needsNormalization(c.block)))) {
      res = 2
      break
    }
    if (maybeComponent(el) ||
        (el.ifConditions && el.ifConditions.some(c => maybeComponent(c.block)))) {
      res = 1
    }
  }
  return res
}

  就是遍历children数组，如果是非元素节点，那么就不做标准化，这时res为0。如果是满足needsNormalization函数判断或者el.ifConditions中元素有满足这个条件判断的，那么认为是需要完全标准化的，此时返回res为2。
  如果判断是一个组件，或者el.ifConditions中有元素是组件的话，则认为需要简单的初始化，也就是只有一级数组嵌套的情况，此时使res = 1。
  上面几个条件只有如果满足完全标准化的才会直接返回结果，否则只有全部遍历才能判断是否是较低一级的标准化类型。
  需要完全标准化的条件

function needsNormalization (el: ASTElement): boolean {
  return el.for !== undefined || el.tag === 'template' || el.tag === 'slot'
}

  满足是v-for绑定的元素，或是一个template标签，或是一个slot标签，注意这里不能是单独只有一个绑定v-for的元素，否则不会执行到这一步，也就是绑定v-for的元素必须存在兄弟元素。
  然后定义gen为altGenNode或者genNode，最后返回一个children.map的数组，并且对children的元素经过gen函数处理后通过join方法拼接为字符串并用','分隔。
  说到这里我们是不是比较懵呢？其实它还是有迹可循的，我们genChildren返回的好像就是[el, el], 0 | 1 | 2这样一个模板字符串，这像极了我们函数调用时传入的参数。然后我们回到genElement方法中执行完genChildren后是这样一段代码

  code = `_c('${el.tag}'${
    data ? `,${data}` : '' // data
  }${
    children ? `,${children}` : '' // children
  })`

  我们可以翻译一下code = _c(tag, data, [el, el], 0 | 1 | 2)，children就是我们上面的这个字符串，我们还可以再稍作变化code = _c(tag, data, children, normalizationType)，再结合我_c函数是不是第四个参数就是normalizationType呢？我们来回顾一下_c函数和$createElement

  vm._c = (a, b, c, d) => createElement(vm, a, b, c, d, false) // 模板方法
  vm.$createElement = (a, b, c, d) => createElement(vm, a, b, c, d, true) // 手写render

  可以看到都是调用了createElement函数，createElement第一个参数都是用来保证对vm实例的持有，而二、三、四、五参数则都是调用_c和$createElement方法传入的，然后他们区别在于模板调用时最后传入一个false，手写render时传入一个true。来看createElement

export function createElement(
  context: Component,
  tag: any,
  data: any,
  children: any,
  normalizationType: any,
  alwaysNormalize: boolean
): VNode | Array<VNode> {
}

  最后一个参数alwaysNormalize译为总是标准化，在我们模板方法中，我们在编译时就生成是否标准化的判断了，所以这里不需要再进行判断标准化了。而手写render就不行了。
  我们还有个genNode函数没有分析，它在生成children过程中也是非常重要的

function genNode (node: ASTNode, state: CodegenState): string {
  if (node.type === 1) {
    return genElement(node, state)
  } else if (node.type === 3 && node.isComment) {
    return genComment(node)
  } else {
    return genText(node)
  }
}

  这个方法看着就非常条理清晰，如果type为1就还是要执行genElement方法，所以这个方法是用来处理元素节点的，但是generate方法中调用的却是这个方法，我一度以为它就是生成代码的入口，实际上它只是处理元素节点的，真正可以当做入口的实际上应该是genNode函数，但是我们之前分析parse时已经了解到了，文本节点是不允许作为根节点的，根节点必须是一个元素标签，所以生成代码开始时将genElement方法作为入口也是没问题的，并且还省去不必要的判断。
  然后else if判断的就是注释节点，执行genComment方法

export function genComment (comment: ASTText): string {
  return `_e(${JSON.stringify(comment.text)})`
}

  这个方法比较简单，就返回_e方法传入comment.text文本。

  target._e = createEmptyVNode

  _e方法就是createEmptyVNode方法，其实就是创建一个注释节点的虚拟dom，我们之前2.4小节已经分析过了，这里就不赘述了。
  如果是其他情况就执行genText方法，当做一个文本节点来处理，因为type为2的情况实际上就是使用了插值表达式的文本，而type为3就是普通文本，所以这里也是合情合理的。

export function genText (text: ASTText | ASTExpression): string {
  return `_v(${text.type === 2
    ? text.expression // no need for () because already wrapped in _s()
    : transformSpecialNewlines(JSON.stringify(text.text))
  })`
}

  这里使用了_v方法，如果是一个表达式的话则传入text.expression，这里说不需要加()，因为会被包裹进_s中，如果不是表达式则用transformSpecialNewlines处理文本

// #3895, #4268
function transformSpecialNewlines (text: string): string {
  return text
    .replace(/\u2028/g, '\\u2028')
    .replace(/\u2029/g, '\\u2029')
}

  这是为了解决#3895、#4268的bug，\u2028是行分隔符，\u2029是段落分割符。这个我们就不细说了。

  target._v = createTextVNode

  就是创建一个文本节点，这个在2.4小节也可以找到。
  至于text.expression它本身就是一个_s{${exp}}这样的字符串，所以不需要做额外处理了，这个我们在parse时就有了解。

  target._s = toString

  _s方法其实是Object.prototype.toString方法，这个方法能保证exp是一个字符串，与JSON.stringify方法不同，如果已经是个字符串就不做转换了。
  说了这么多我们可以言归正传一下了，对于我们的demo，首先执行genElement方法时会调用genStatic，之后又递归调用了genElement，这时会进入genElement的else逻辑，从而执行genChildren。 genChildren方法实际上会遍历我们div的子节点并循环调用genNode方法，这样我们的两个p标签会执行genElement方法，其实这个时候我们两个p标签之间因为有换行，所以还会被插入一个空格文本，所以div.children是[p, text, p]这样具有三个子元素。p标签同样会走我们genElement的else逻辑，所以又会执行到genChildren，第一个p的子节点是一个文本节点，文本内容为1。这种情况下调用genNode时就会执行genText生成一个创建文本vnode的函数。当然这么描述可能不太准确，因为它实际上是生成了一个生成文本vnode的函数字符串，经过new Function才能生成最后可执行代码。
  然后对第一个p标签genChildren处理结束，这时p的children为"[_v("1")]"，然后会回到genElement去生成code，这时code为"_c('p',[_v("1")])"，然后同样处理第二个text节点，这个节点为空我们就不去分析了。到第三个p节点和第一个p处理方式相同，会生成一个code为"_c('p',[_v("2")])"，那么现在div的三个子节点都处理完了，那就回到div第二次调用genElement的时候了。
  div的children处理后就变成了"[_c('p',[_v("1")]),_v(" "),_c('p',[_v("2")])]"，然后生成代码code为"_c('div',[_c('p',[_v("1")]),_v(" "),_c('p',[_v("2")])])"。这个结果其实就类似我们render函数的格式了。
  但是现在还没有结束，因为我们这个demo整个都是一个静态根，对于静态根的处理有些不同，回到genStatic函数中

  state.staticRenderFns.push(`with(this){return ${genElement(el, state)}}`)
  state.pre = originalPreState
  return `_m(${
    state.staticRenderFns.length - 1
  }${
    el.staticInFor ? ',true' : ''
  })`

  它会将静态的内容添加到staticRenderFns中，并且会在前面拼接with(this)。最后会返回一个_m函数。其定义如下

export function renderStatic (
  index: number,
  isInFor: boolean
): VNode | Array<VNode> {
  const cached = this._staticTrees || (this._staticTrees = [])
  let tree = cached[index]
  // if has already-rendered static tree and not inside v-for,
  // we can reuse the same tree.
  if (tree && !isInFor) {
    return tree
  }
  // otherwise, render a fresh tree.
  tree = cached[index] = this.$options.staticRenderFns[index].call(
    this._renderProxy,
    null,
    this // for render fns generated for functional component templates
  )
  markStatic(tree, `__static__${index}`, false)
  return tree
}

  接收两个参数，第一个是静态根在state.staticRenderFns中的索引，第二个参数是是否在v-for循环中。
  在vue实例上添加一个_staticTrees来做缓存，如果能取到缓存结果的话并且不是在v-for循环中的，就取缓存中的结果。如果没有的话就从staticRenderFns中取出来执行并缓存。
  其实就是对于静态根生成的vnode做缓存，这样在更新时就不需要重新render这部分内容了，可以说这部分设计也是非常巧妙的。
  这里还又执行了一个markStatic方法

function markStatic (
  tree: VNode | Array<VNode>,
  key: string,
  isOnce: boolean
) {
  if (Array.isArray(tree)) {
    for (let i = 0; i < tree.length; i++) {
      if (tree[i] && typeof tree[i] !== 'string') {
        markStaticNode(tree[i], `${key}_${i}`, isOnce)
      }
    }
  } else {
    markStaticNode(tree, key, isOnce)
  }
}

  如果tree是一个数组，并且其元素不是一个字符串就循环调用markStaticNode，否则直接调用markStaticNode。

function markStaticNode (node, key, isOnce) {
  node.isStatic = true
  node.key = key
  node.isOnce = isOnce
}

  将node.isStatic标记为true，node.key赋值为'__static__0'的形式，最后根据isOnce的值给node.isOnce赋值。
  分析了这么多其实genStatic方法的结果非常简单，它最后返回的就是_m(0)。最后回到我们generate函数中，得到的code就是我们的_m(0)了。
  最后返回的结果还要包装成下面的对象：

  return {
    render: `with(this){return ${code}}`,
    staticRenderFns: state.staticRenderFns
  }

  那这个小节我们就分析到这里了，下个小节我们通过其他示例来分析genElement的其它分支内容。
