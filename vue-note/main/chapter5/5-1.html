  学习本章前我们先来了解一下编译是做什么事。不知道你是否好奇我们前面讲的内容不管是挂载还是更新，还是响应式都绕不开render函数，那为什么我们开发时却很少写render函数，而是写template呢？显然是有一个我们不了解的东西去把template转化成了render函数，这就是编译。下面我们就一起来看看编译的过程吧！

# 5.1 初探编译器

  我们知道Vue的入口文件有两个，一种是带编译版本的entry-runtime-with-compiler.js（Runtime + Compiler），另一种是entry-runtime.js（Runtime-Only）不带编译器的版本。那这两个版本有什么区别呢？
  早在我们找Vue入口文件时，我们就已经分析过它们的区别了，runtime + compiler版本$mount方法多出一些逻辑，这些逻辑是在options.render不存在时执行的，我们之前说过，Vue是只认render函数，无论我们怎么写代码最后都是要转化为render函数的。所以编译就是把我们template模板转化为render函数的过程。
  如果你接触过早期vue-cli的话一定对这个不会陌生，我们在搭建项目时会让你选择带编译版本或者不带编译版本，如果选Runtime-Only包体积会小4kb左右，我们也推荐使用Runtime-Only版本。
  对于我们开发人员来说，这两个版本最直观的区别就是：编译版本可以在options中配置template选项，而不带编译版本不行。但是我们如果开发都去写render函数的话，那恐怕是难以令人接受的，它不仅不直观而且繁琐。好在单文件组件sfc解决了这个问题，我们任然可以写template，任然可以使用我们惯用的html标签写法。但是使用template是不是就又会有编译的问题了呢？确实是这样的，因为我们代码只认render函数。如果脱离render我们就无法生成vnode，挂载和更新都无从实现。
  实际上我们sfc中的template也是需要在编译的，但是不同的是它不是在运行时进行编译的，而是借助webpack和vue-loader预先编译成render函数的。这样既能省去编译部分代码还能让我们开发更符合html习惯。所以这就是为什么我们推荐runtime only版本vue但还要学习编译的原因。因为无论预编译还是运行时编译都绕不开它。
  基于2.2小节的基础上我们接下来就来分析Runtime + Compiler版本$mount方法中的编译部分逻辑：

  if (template) { // 编译相关
    const { render, staticRenderFns } = compileToFunctions(template, {
      outputSourceRange: process.env.NODE_ENV !== 'production',
      shouldDecodeNewlines,
      shouldDecodeNewlinesForHref,
      delimiters: options.delimiters,
      comments: options.comments
    }, this)
    options.render = render
    options.staticRenderFns = staticRenderFns
  }

  可以看到编译的其实就是执行compileToFunctions方法了，它返回render函数和staticRenderFns，我们把它解构出来添加到opitons中，实际上就是添加到this.$options上。所以接下来的重点就是搞清楚compileToFunctions到底是什么，也就是找到编译的入口在哪里，这就是本小节接下来要做的事。编译入口是十分复杂的，比Vue入口还要难找。接下来先看compileToFunctions方法：

import { baseOptions } from './options'
import { createCompiler } from 'compiler/index'

const { compile, compileToFunctions } = createCompiler(baseOptions)
export { compile, compileToFunctions }

  该方法定义于platforms/web/comiler/index.js下，是createCompiler()执行结果所解构出来的一项，所以就再看createCompiler方法，顾名思义它是创建编译器的方法，它的定义是公共部分代码，也就是与平台不相关，所以是在src/compiler/index文件定义

// `createCompilerCreator` allows creating compilers that use alternative
// parser/optimizer/codegen, e.g the SSR optimizing compiler.
// Here we just export a default compiler using the default parts.
// `createCompilerCreator `允许创建使用替代
// 解析器/优化器/代码生成器的编译器，例如SSR优化编译器。
// 这里我们只使用默认部分导出默认编译器
export const createCompiler = createCompilerCreator(function baseCompile (
  template: string,
  options: CompilerOptions
): CompiledResult {
  const ast = parse(template.trim(), options)
  if (options.optimize !== false) {
    optimize(ast, options)
  }
  const code = generate(ast, options)
  return {
    ast,
    render: code.render,
    staticRenderFns: code.staticRenderFns
  }
})

  可以看到createCompiler是createCompilerCreator执行结果，所以这里肯定是闭包返回内层的函数，createCompilerCreator意思是编译器创建者的创建者，它的参数baseCompile是一个函数，但是我们这里就不关注它了，因为它并不在这里调用。
  接下来就去找createCompilerCreator方法定义，它定义在create-compiler.js下：

export function createCompilerCreator (baseCompile: Function): Function {
  return function createCompiler (baseOptions: CompilerOptions) {
    function compile (
      template: string,
      options?: CompilerOptions
    ): CompiledResult {
      ...
      const compiled = baseCompile(template.trim(), finalOptions)
      ...
      return compiled
    }
    return {
      compile,
      compileToFunctions: createCompileToFunctionFn(compile)
    }
  }
}

  可以看到它确实是一个闭包函数，内层函数就叫createCompiler，然后我们看内层函数它接收一个baseOptions，这是平台相关的配置platforms/web/comiler/index文件所做的事就是把平台相关配置baseOptions传入。
  函数内部又定义了一个compile函数，它返回值就是传入的baseCompile函数执行结果。当然这个东西我们不必太去关注，因为我们现在还没用到它。我们目的只是找到$mount中调用的compileToFunctions方法是什么，显然这里就返回了这个方法，但是它的定义又是createCompileToFunctionFn(compile)的返回结果。那里面肯定又是一层闭包了。
  createComileToFunctionFn定义在to-function.js，果然它是一个闭包函数，内层函数返回了compileToFunctions函数

export function createCompileToFunctionFn (compile: Function): Function {
  const cache = Object.create(null)

  return function compileToFunctions (
    template: string,
    options?: CompilerOptions,
    vm?: Component
  ): CompiledFunctionResult {
    options = extend({}, options)
    // check cache
    const key = options.delimiters
      ? String(options.delimiters) + template
      : template
    if (cache[key]) {
      return cache[key]
    }

    // compile
    const compiled = compile(template, options)

    // turn code into functions
    const res = {}
    const fnGenErrors = []
    res.render = createFunction(compiled.render, fnGenErrors)
    res.staticRenderFns = compiled.staticRenderFns.map(code => {
      return createFunction(code, fnGenErrors)
    })
    return (cache[key] = res)
  }
}

  这个是删减后的createCompileToFunctionFn方法，内层函数就是我们在$mount中调用的compileToFunctions方法，compileToFunctions内部执行我们传入的方法compile，这个方法会返回ast、render、staticRenderFns方法，然后保存到res中，然后还要做一个缓存优化，最后返回res，所以compileToFunctions返回了render和staticRenderFns。
  所以我们$mount中获取到的render和staticRenderFn就是这里返回的。那么为什么要把这入口方法定义的这么复杂呢？
  我们来整理一下

import { parse, optimize, generate } from 'otherFile'

const createCompiler = function createCompiler(baseOptions) {
  function baseCompile(
    template: string,
    options: CompilerOptions
  ): CompiledResult {
    const ast = parse(template.trim(), options)
    if (options.optimize !== false) {
      optimize(ast, options)
    }
    const code = generate(ast, options)
    return {
      ast,
      render: code.render,
      staticRenderFns: code.staticRenderFns
    }
  }
  const obj = {
    compile: function compile(template, options) {
      return baseCompile(template, options),
    compileToFunctions: function compileToFunctions() {
      const compiled = obj.compile()
      return {
        render: function createFunction(code, errors) {
          return new Function(code)
        },
        staticRenderFns: compiled.staticRenderFns.map(code => {
          return createFunction(code)
        })
      }
    }
  }
  return obj
}

  这样是不是就好看多了，createCompiler是这么一个东西，我们$mount中用到的就是createCompiler执行后返回obj中的comileToFunctions方法。
  但是大家有没有发现一个问题，这样写太过于死板了，完全没有变通能力，一个创建编译器的方法只能创建一种编译器，如果想要编译到其他平台，那只能依靠baseOptions的不同，假如baseCompile需要执行不同定义的话，那就需要重新定义整个创建编译器的方法了，这就出现了代码的冗余。
  实际上也存在这样的情况，因为它可能被用来创建SSR优化编译器，显然更好的做法就是把baseCompile函数作为参数传入，外面再包裹一层函数，这样就有了我们'创建编译器的创建者'函数。

const createCompiler = createCompilerCreator(baseCompile)

  然后这里createCompiler就是执行createCompilerCreator后返回的真正的创建编译器函数createCompiler，因为是一样的东西所以起同样的名字。然后它之后返回结果就是一个compile和compileToFunctions，这两个东西区别就是compile执行会生成代码字符串，经过compileToFunctions才能转换成可执行函数代码。 然后compileToFunction方法又被createCompileToFunctionFn方法包裹一层，这是为了通过闭包方式去缓存编译结果，如果传入的参数没有变化的话就不需要每次重复去编译浪费性能了。

function createCompileToFunctionFn(compile: Function): Function {
  const cache = Object.create(null)
  return function compileToFunctions() {}
}

  了解它的设计思路后是不是看着这些代码也没那么复杂了呢？找到编译入口并对其有个大致了解后我们就可以开始正式的分析了。
