# 5.9 parse之生成AST（5）

  上小节我们分析对结束标签的处理，那么接下来我们来分析对文本节点的处理，也就是我们的chars钩子函数

  if (!currentParent) {
    if (process.env.NODE_ENV !== 'production') {
      if (text === template) {
        warnOnce(
          'Component template requires a root element, rather than just text.',
          { start }
        )
      } else if ((text = text.trim())) {
        warnOnce(
          `text "${text}" outside root element will be ignored.`,
          { start }
        )
      }
    }
    return
  }

  如果currentParent不存在的话，说明没有根元素，也就是说template下就是一段文字或者说文本没有任何标签元素包裹，直接暴露在template下。第一种情况下显然text和template是相同的，这时要报一个警告说组件模板需要是一个根元素，而不是仅仅是文本。如果是第二种情况，说明除了文本外可能还有其他元素，并且这个文本要经过trim处理，以保证它不是空格，这种情况下回忽略文本内容，只去保留元素内容。

  // IE textarea placeholder bug
  /* istanbul ignore if */
  if (isIE &&
    currentParent.tag === 'textarea' &&
    currentParent.attrsMap.placeholder === text
  ) {
    return
  }

  再往下是对IE下textarea placeholder的bug处理

  <div id="textarea">
    <textarea placeholder="place input..."></textarea>
  </div>

  如果我们输出textarea.innerHTML时会打印

'<textarea placeholder="place input...">place input...</textarea>'

  我们发现在placeholder中的内容被设置到标签中，这个时候会满足placeholder等于text，这个时候解析到的text就是textarea双标签中的文本，显然满足这个条件时就是遇到了这个bug，那么这里直接忽略这段文本就可以了。

  const children = currentParent.children
  if (inPre || text.trim()) {
    text = isTextTag(currentParent) ? text : decodeHTMLCached(text)
  } else if (!children.length) {
    // remove the whitespace-only node right after an opening tag
    text = ''
  } else if (whitespaceOption) {
    if (whitespaceOption === 'condense') {
      // in condense mode, remove the whitespace node if it contains
      // line break, otherwise condense to a single space
      text = lineBreakRE.test(text) ? '' : ' '
    } else {
      text = ' '
    }
  } else {
    text = preserveWhitespace ? ' ' : ''
  }

  接下来获取currentParent.children，然后判断inPre为true或者text文本不为空时，就将text进行解码html实体，但是如果父元素满足isTextTag判断就不去解码。

// for script (e.g. type="x/template") or style, do not decode content
function isTextTag (el): boolean {
  return el.tag === 'script' || el.tag === 'style'
}

  el.tag如果是script或者style就不能进行解码了。
  接下来是else if判断children数组为空，并且不满足if的话说明text中是空白字符，children为空则说明是开始标签后面的空格，这时把它设置为空就行了。对于标签开始后的空格应该被忽略掉。
  接下来是判断whitespaceOption的值，这个配置我们前面已经分析过了，它有两个值'condense'(压缩)、'preserve'(保留)。这里显然是对children存在元素时的处理，也就是对标签之间的空白字符处理，默认为压缩。在压缩模式下，如果空白节点包含换行符则删除它，否则压缩为单个空格。如果是保留模式会直接替换为单个空格，这可能跟我们之前对whitespaceOption的分析不同，毕竟我是参考了vue3的文档，对于vue2我没有找到相关描述。
  最后whitespaceOptions没有定义的话就按照preserveWhitespace规则处理，如果其为true就保留一个空格，否则不保留，默认是保留的。
  接下来就是最后一段逻辑了

  if (text) {
    if (!inPre && whitespaceOption === 'condense') {
      // condense consecutive whitespaces into single space
      text = text.replace(whitespaceRE, ' ')
    }
    let res
    let child: ?ASTNode
    if (!inVPre && text !== ' ' && (res = parseText(text, delimiters))) {
      child = {
        type: 2,
        expression: res.expression,
        tokens: res.tokens,
        text
      }
    } else if (text !== ' ' || !children.length || children[children.length - 1].text !== ' ') {
      child = {
        type: 3,
        text
      }
    }
    if (child) {
      if (process.env.NODE_ENV !== 'production' && options.outputSourceRange) {
        child.start = start
        child.end = end
      }
      children.push(child)
    }
  }

  首先在text不为空时才会执行这段代码，这里主要是对文本内容的进一步处理，如果不在pre标签中并且whitespaceOption === 'condense'时，就需要把文本中的连续空格处理为一个空格。
  接下来会定义res和child，如果不是在v-pre绑定标签中，并且text不完全等于一个空格字符串，并且parseText解析文本不为空时会给child赋值。child的type属性为2，并且会从res中获取expression和tokens，还会保存其text值。
  parseText我们前面就已经见过了，它在调用时传入了delimiters，所以我们可以猜到它就是处理我们插值语法的，Vue的插值语法是我们非常常见的，并且是很重要的一个功能，我们这里先把后面内容分析完，最后我们再去分析parseText方法。
  然后这里如果text不是一个' '，那么它应该是一个有值的文本，那么就要给child赋值，赋值后的type为3，也就是一个文本节点。但是如果text为一个空格字符，但是children数组为空，说明是元素内的前导空格，那么还是要保留为一个空格。如过text是一个空格字符，并且children数组也是有值的，那么是标签间的空格，那也是要保留一个的，所以要判断前一个节点是不是空格字符（也就是children的最后一个元素的text是否是' '）。如果是的话就不处理了，如果不是的话就要为child赋值。
  最后将child添加到children数组中。
  最后我们来分析parseText方法

export function parseText (
  text: string,
  delimiters?: [string, string]
): TextParseResult | void {
  const tagRE = delimiters ? buildRegex(delimiters) : defaultTagRE
  if (!tagRE.test(text)) {
    return
  }
 // 省略...
  return {
    expression: tokens.join('+'),
    tokens: rawTokens
  }
}

  parseText方法接收两个参数，一个是我们要处理的文本，delimiters则是配置的插值符号，默认是[', ']，然后我们来看这个正则生成器

const defaultTagRE = /\{\{((?:.|\r?\n)+?)\}\}/g
const regexEscapeRE = /[-.*+?^${}()|[\]\/\\]/g

const buildRegex = cached(delimiters => {
  const open = delimiters[0].replace(regexEscapeRE, '\\$&')
  const close = delimiters[1].replace(regexEscapeRE, '\\$&')
  return new RegExp(open + '((?:.|\\n)+?)' + close, 'g')
})

  首先默认的正则已经定义好了defaultTagRE，buildRegex 主要是将我们自定义的delimiters生成其正则。
  其实就是判断我们写的插值符号有没有regexEscapeRE中的特殊符号，因为这些特殊符号需要转义，所以定义open和close就是为了给我们自定义内容需要转义的部分加入转义符，然后拼接生成正则。
  回到parseText方法中，如果检查没有插值语句则直接return。否则会继续我往下处理，最后返回一个带有expression和tokens的属性。
  我们来看中间过程

  const tokens = []
  const rawTokens = []
  let lastIndex = tagRE.lastIndex = 0
  let match, index, tokenValue
  while ((match = tagRE.exec(text))) {
    index = match.index
    // push text token
    if (index > lastIndex) {
      rawTokens.push(tokenValue = text.slice(lastIndex, index))
      tokens.push(JSON.stringify(tokenValue))
    }
    // tag token
    const exp = parseFilters(match[1].trim())
    tokens.push(`_s(${exp})`)
    rawTokens.push({ '@binding': exp })
    lastIndex = index + match[0].length
  }

  首先定义一些变量，然后执行while循环，循环条件是tagRE.exec匹配到值不为null，然后保存match.index到index。如果index > lastIndex，然后就将第一个插值前的文本内容添加到rawTokens数组中，然后tokens中同样要添加这部分内容，但是需要做JSON.stringify转换。
  然后对于获取到的内容会解析其过滤器，比如有如下模板

  <div>我是{{name}}</div>

  match结果是['{ { name } }', ' name ']，exp其实就是解析第二个元素中有没有使用过滤器。那么rawTokens为['我是']，tokens为['"我是"']。
  然后会再往tokens中push一个_s(name)，rawTokens中会再push一个{ '@binding': "name" }对象，然后修改lastIndex。然后跳到下一个循环，继续去匹配插值模板，这里正则定义时设置了g标志，对于RegExp.exec方法：

当正则表达式设置 g 标志位时，可以多次执行 exec 方法来查找同一个字符串中的成功匹配。当你这样做时，查找将从正则表达式的 lastIndex 属性指定的位置开始。（test() 也会更新 lastIndex 属性）。

注意

即使再次查找的字符串不是原查找字符串时，lastIndex 也不会被重置，它依旧会从记录的 lastIndex 开始。

  最后循环结束后执行

  if (lastIndex < text.length) {
    rawTokens.push(tokenValue = text.slice(lastIndex))
    tokens.push(JSON.stringify(tokenValue))
  }

  最后如果lastIndex小于text.length，说明插值解析完成后还要一部分静态内容，那么这部分静态内容都要添加到rawTokens和tokens数组中。
  最后返回的expression要将tokens转化为字符串'"我是"+_s(name)'，而tokens则是rawTokens对象。
  最后根据返回的对象将child赋值

  child = {
    type: 2,
    expression: res.expression,
    tokens: res.tokens,
    text
  }

  到这里我们就分析完对文本节点的处理了，最后我们就剩一个对注释节点处理的方法comment钩子函数了。

  comment (text: string, start, end) {
    // adding anything as a sibling to the root node is forbidden
    // comments should still be allowed, but ignored
    // 将任何内容作为同级添加到根节点是禁止的。注释仍应被允许，但应被忽略
    if (currentParent) {
      const child: ASTText = {
        type: 3,
        text,
        isComment: true
      }
      if (process.env.NODE_ENV !== 'production' && options.outputSourceRange) {
        child.start = start
        child.end = end
      }
      currentParent.children.push(child)
    }
  }

  这个就非常简单了，生成一个注释节点的child添加到currentParent.children，这里的type也是3，但是多两个isComment属性。
  到这里我们的parse方法就都分析完毕了，最后会返回root，也就是AST，AST是抽象语法树，所以它应该是一个树的结构，所以这里返回了root根元素其实就是一个类似我们html dom的树结构。
