# 5.11 generate代码生成（1）

  上小节我们分析AST的优化过程，那么这个小节我们来分析根据AST生成代码的过程。还是回到我们compiler/index.js文件中，我们baseCompile方法最后执行generate函数生成code。
  下面我们来看看generate方法，在codegen/index.js中：

export function generate (
  ast: ASTElement | void,
  options: CompilerOptions
): CodegenResult {
  const state = new CodegenState(options)
  // fix #11483, Root level <script> tags should not be rendered.
  const code = ast ? (ast.tag === 'script' ? 'null' : genElement(ast, state)) : '_c("div")'
  return {
    render: `with(this){return ${code}}`,
    staticRenderFns: state.staticRenderFns
  }
}

  可以看到它定义比较简单，接收的第一个参数就是我们生成的ast，第二个参数同样是我们的options。 然后我们可以看到首先是new CodegenState构造器去生成state，我们来看一下这个构造器的定义：

export class CodegenState {
  options: CompilerOptions;
  warn: Function;
  transforms: Array<TransformFunction>;
  dataGenFns: Array<DataGenFunction>;
  directives: { [key: string]: DirectiveFunction };
  maybeComponent: (el: ASTElement) => boolean;
  onceId: number;
  staticRenderFns: Array<string>;
  pre: boolean;

  constructor (options: CompilerOptions) {
    this.options = options
    this.warn = options.warn || baseWarn
    this.transforms = pluckModuleFunction(options.modules, 'transformCode')
    this.dataGenFns = pluckModuleFunction(options.modules, 'genData')
    this.directives = extend(extend({}, baseDirectives), options.directives)
    const isReservedTag = options.isReservedTag || no
    this.maybeComponent = (el: ASTElement) => !!el.component || !isReservedTag(el.tag)
    this.onceId = 0
    this.staticRenderFns = []
    this.pre = false
  }
}

  首先把options、warn等属性挂载到this上，然后这里提取transformCode，这个东西其实是标记为further的。目前是没有这个东西的，所以简单认为transforms是一个空数组就可以了。然后是提取modules中的genData存放到this.dataGenFns中。这个东西其实之前已经分析过了，就是分别处理class和style属性的两个函数。
  然后就是合并指令，一些base指令比如v-on、v-bind、v-cloak，还有一些平台相关指令v-html、v-text、v-model等。
this.maybeComponent则是判断el.component存在或者不是平台保留标签的方法。然后初始化onceId为0，staticRenderFns为[]、pre为false。
  最后初始化的state就是这么一个对象。然后就要生成我们的code，这里注释说为了解决一个bug，根级的script标签不应该被渲染，如果不是script标签，那么应该执行genElement方法并且会把ast和state传入。如果ast不存在会渲染一个空的div标签，但是它是通过'_c("div")'这么一个字符串来实现的。
  如果生成代码的话就成了_c方法的调用，这个方法我们应该是比较熟悉的，我们的h函数全称是什么呢？回顾我们第3.1小节就可以知道它全称是vm.$createElement方法。那里我也讲过它是我们手写render函数时调用的方法，而我们在写template模板时会调用另一个方法vm._c，他两本质上没有太大区别，只是在调用createElement方法时传入第六个参数一个为true一个为false。所以这里_c就是我们这里定义的vm._c方法，但是这里没有this，对于this指向我们可以之后再来分析。
  我们再来看genElement方法

export function genElement (el: ASTElement, state: CodegenState): string {
  if (el.parent) {
    el.pre = el.pre || el.parent.pre
  }

  if (el.staticRoot && !el.staticProcessed) {
    return genStatic(el, state)
  } else if (el.once && !el.onceProcessed) {
    return genOnce(el, state)
  } else if (el.for && !el.forProcessed) {
    return genFor(el, state)
  } else if (el.if && !el.ifProcessed) {
    return genIf(el, state)
  } else if (el.tag === 'template' && !el.slotTarget && !state.pre) {
    return genChildren(el, state) || 'void 0'
  } else if (el.tag === 'slot') {
    return genSlot(el, state)
  } else {
	... 省略部分代码
  }
}

  我们可以看到其实这里就是有很多if判断，我们一个个来看，首先如果存在父节点，那么el.pre等于它本身的el.pre或者父节点的el.pre。
  接下来如果是静态根就执行genStatic并返回，至于后面的判断条件我们都没见过，其实可以猜想到他是genStatic中添加的了，为防止我们重复去执行。然后再往下基本是相同的方法处理el.once、el.for、el.if。
  再往下如果是template标签则稍有不同，还要满足不是具名插槽，并且!state.pre，如果满足这几个条件，就执行genChildren，如果执行结果为空会返回一个'void 0'字符串。再如果是一个slot标签就执行genSlot方法。
  最后一个else逻辑如下

  // component or element
  let code
  if (el.component) {
    code = genComponent(el.component, el, state)
  } else {
    let data
    if (!el.plain || (el.pre && state.maybeComponent(el))) {
      data = genData(el, state)
    }

    const children = el.inlineTemplate ? null : genChildren(el, state, true)
    code = `_c('${el.tag}'${
      data ? `,${data}` : '' // data
    }${
      children ? `,${children}` : '' // children
    })`
  }
  // module transforms
  for (let i = 0; i < state.transforms.length; i++) {
    code = state.transforms[i](el, code)
  }
  return code

  我们知道el.component存在说明是个is绑定的组件，这时会通过genComponent方法处理。否则判断!el.pain或者是el.pre为true并且是一个组件时执行genData方法。我们知道el.pre是存在v-pre指令才会被赋值为true。
  接下来是判断如果不是内联模板时执行genChildren方法并保存在children中。然后就给code赋值为这样一个模板字符串，实际上就与我们render函数写法很像了。其实我们已经清楚对于template最后都是会转话为render函数的。还是那句话Vue渲染时其实是只认render函数的。
  最后这个for循环其实不会执行，因为transforms是个空数组。最后return了code。
  到这里我们其实就将代码生成的大致流程分析完了，但是显然我们还有很多gen系列方法没有分析，这些方法其实才是我们代码生成的核心所在。在gen函数中我们会看到很多类似_c的以_开头的系列函数，它会被用来生成特定的结构，我们先来看看这些方法的定义。
  在core/instance文件夹下有一个render-helpers文件夹，里面有很多js文件，我们打开index.js会发现

import { toNumber, toString, looseEqual, looseIndexOf } from 'shared/util'
import { createTextVNode, createEmptyVNode } from 'core/vdom/vnode'
import { renderList } from './render-list'
import { renderSlot } from './render-slot'
import { resolveFilter } from './resolve-filter'
import { checkKeyCodes } from './check-keycodes'
import { bindObjectProps } from './bind-object-props'
import { renderStatic, markOnce } from './render-static'
import { bindObjectListeners } from './bind-object-listeners'
import { resolveScopedSlots } from './resolve-scoped-slots'
import { bindDynamicKeys, prependModifier } from './bind-dynamic-keys'

export function installRenderHelpers (target: any) {
  target._o = markOnce
  target._n = toNumber
  target._s = toString
  target._l = renderList
  target._t = renderSlot
  target._q = looseEqual
  target._i = looseIndexOf
  target._m = renderStatic
  target._f = resolveFilter
  target._k = checkKeyCodes
  target._b = bindObjectProps
  target._v = createTextVNode
  target._e = createEmptyVNode
  target._u = resolveScopedSlots
  target._g = bindObjectListeners
  target._d = bindDynamicKeys
  target._p = prependModifier
}

  这里定义了很多_系列函数，我们先不着急分析这些方法，我们可以在用到这些方法时再具体分析。我们先看这些方法是挂载到哪里的。也就是看installRenderHelpers在哪里被调用。
  在render.js中的renderMixin函数中

export function renderMixin (Vue: Class<Component>) {
  // install runtime convenience helpers
  installRenderHelpers(Vue.prototype)

  Vue.prototype.$nextTick = function (fn: Function) {
    return nextTick(fn, this)
  }

  Vue.prototype._render = function (): VNode {    
  }
}

  首先就是执行了installRenderHelpers方法，并将Vue.prototype传入，所以这些_系列方法都被挂载到Vue的原型对象上。了解这个之后下一个小节我们就来具体分析gen系列函数做了什么事。
