# 5.2 从入口开始

  上个小节我们找到了编译的入口，并分析了其大致设计思路，那么这个小节我们就从入口开始来分析编译的过程：

  const { render, staticRenderFns } = compileToFunctions(template, {
    outputSourceRange: process.env.NODE_ENV !== 'production',
    shouldDecodeNewlines,
    shouldDecodeNewlinesForHref,
    delimiters: options.delimiters,
    comments: options.comments
  }, this)

outputSourceRange属性顾名思义输出源码范围，其实就是根据环境判断得来的一个Boolean值，它的具体作用我们等遇到它的时候再分析。
shouldDecodeNewlines和shouldDecodeNewlinesForHref是两个常量，定义于platforms/web/util/compat.js中，这些工具方法我们把它放到helper文档中。
delimiters和comments是从选项中去获取，所以它们是Vue的options选项配置，官网中都有说明，delimiters是改变插值符的，是一个有两个元素的数组，我们默认是两个花括号[", "]。comments则是配置html是否要保留注释，默认是不保留。   然后我们直接来看compileToFunctions方法，因为上一小节我们所有大致流程都分析过了，所以这个小节我们就不强调代码来源了

export function createCompileToFunctionFn(compile: Function): Function {
  const cache = Object.create(null)

  return function compileToFunctions(
    template: string,
    options?: CompilerOptions,
    vm?: Component
  ): CompiledFunctionResult {
  }
}

  它是由createCompileToFunctionsFn方法返回的，这里先定义了cache作为缓存使用的空对象。然后看compileToFunctions函数

  // 使用 extend 函数将 options 的属性混合到新的对象中并重新赋值 options

  // 检查选项参数中是否包含 warn，如果没有则使用 baseWarn

  options就是我们上面传入的一堆对象了。这里就是就是做了对象的拷贝，然后检查有没有定义warn属性，未定义则使用baseWran。得到warn之后就把options.warn直接删除。

  if (process.env.NODE_ENV !== 'production') {

  接下来这段代码是一个警告内容，它表达了一个很重要的内容，这段代码在开发环境尝试执行new Function('retrun 1')，如果执行失败并且错误信息有'unsafe-eval'或'CSP'等字样信息时就会给出一个警告。CSP是内容安全策略，如果你的策略比较严格，那么new Function将会受到影响。 内容安全策略 (opens new window)
  解决这个问题也非常简单，放松你的CS策略或者使用预编译。

  // check cache
  const key = options.delimiters
    ? String(options.delimiters) + template
    : template
  if (cache[key]) {
    return cache[key]
  }

  这里根据template以及delimiters配置去定义key，如果发现key已经被缓存了就直接返回缓存结果，而不需要重复编译。
  上面这些都是一些无关紧要的逻辑，那么接下来就是我们最重要的逻辑

  const compiled = compile(template, options)

  执行编译逻辑，虽然只有一行代码，但是它做了很多事，这块执行接收的compile函数参数，然后compile也会执行baseCompile函数并返回结果。当然这都是后话了，我们先把这部分内容看完。

  // check compilation errors/tips
  if (process.env.NODE_ENV !== 'production') {
    if (compiled.errors && compiled.errors.length) {
      if (options.outputSourceRange) {
        compiled.errors.forEach(e => {
          warn(
            `Error compiling template:\n\n${e.msg}\n\n` +
            generateCodeFrame(template, e.start, e.end),
            vm
          )
        })
      } else {
        warn(
          `Error compiling template:\n\n${template}\n\n` +
          compiled.errors.map(e => `- ${e}`).join('\n') + '\n',
          vm
        )
      }
    }
    if (compiled.tips && compiled.tips.length) {
      if (options.outputSourceRange) {
        compiled.tips.forEach(e => tip(e.msg, vm))
      } else {
        compiled.tips.forEach(msg => tip(msg, vm))
      }
    }
  }

  这段逻辑很简单就可以看到是处理一些编译过程中的错误和提示，这些错误和提示是保存在编译结果compiled上的。这里可以看到根据outputSourceRange的配置处理方式也是不同的。 接下来的代码也是比较重要的一环

  // turn code into functions
  const res = {}
  const fnGenErrors = []
  res.render = createFunction(compiled.render, fnGenErrors)
  res.staticRenderFns = compiled.staticRenderFns.map(code => {
    return createFunction(code, fnGenErrors)
  })

  这里定义res对象和错误处理数组，然后让res.render等于createFunction执行返回的结果。createFunction方法其实就是new Function方法的封装，这里传入了compiled.render，也就是得到的baseCompile执行后返回render函数字符串，然后通过new Function生成真正的render，如果发生错误就放在fnGenErrors中，我们来看createFunction方法：

function createFunction(code, errors) {
  try {
    return new Function(code)
  } catch (err) {
    errors.push({ err, code })
    return noop
  }
}

接下来是最后一段代码

// check function generation errors.
// this should only happen if there is a bug in the compiler itself.

if (process.env.NODE_ENV !== 'production') {

}     
return (cache[key] = res)

  这段代码同样是在非生产环境下执行的，这段代码主要的作用是用来打印在生成渲染函数过程中的错误，也就是上面定义的常量 fnGenErrors 中所收集的错误。注释中写的很清楚，这段代码的作用主要是用于开发 codegen 功能时使用，一般是编译器本身的错误，所以对于我们来讲基本用不到。
  最后就是res缓存起来并返回。所以我们在$mount中取到的render就是真正的render函数了。
  接下来我们回过头看看compile方法

  function compile(
    template: string,
    options?: CompilerOptions
  ): CompiledResult {
    return compiled
  }

首先接收两个参数一个template，另一个options，这两个参数都是$mount函数中传入的，最后返回的就是我们上面说的compiled。

  const finalOptions = Object.create(baseOptions)
  const errors = []
  const tips = []

  let warn = (msg, range, tip) => {
    (tip ? tips : errors).push(msg)
  }

    if (process.env.NODE_ENV !== 'production' && options.outputSourceRange) {

    }

  }

  直接来看开头长长的一段逻辑，首先就是通过Object.create创建一个继承自平台相关配置的baseOptions，然后定义用来存放error和tip的数组errors和tips。然后定义warn函数，如果options存在并且outputSourceRange存在同样是要执行不同warn函数。
  接下来就是合并自定义模块，合并自定义指令，然后把options除了模块和指令外的东西全部copy到finalOptions中，然后给finalOptions设置warn函数。

    const compiled = baseCompile(template.trim(), finalOptions)
    if (process.env.NODE_ENV !== 'production') {
      detectErrors(compiled.ast, warn)
    }
    compiled.errors = errors
    compiled.tips = tips
    return compiled

  接下来就是执行我们的baseCompile方法获得compiled，然后添加errors和tips后返回，这就和我们之前在compileToFunction方法中的逻辑对应上了。
  然后$mount中的compileToFunctions实际上是这么得来的

const { compile, compileToFunctions } = createCompiler(baseOptions)

  是从createCompile方法执行结果中取到的，这里传入了baseOptions是与平台相关的一些配置项，也就是我们compile方法开头使用baseOptions的来源。
  然后我们要看baseCompile方法做了什么要先来看createCompiler方法的定义

export const createCompiler = createCompilerCreator(function baseCompile(
  template: string,
  options: CompilerOptions
): CompiledResult {
  const ast = parse(template.trim(), options)
  if (options.optimize !== false) {
    optimize(ast, options)
  }
  const code = generate(ast, options)
  return {
    ast,
    render: code.render,
    staticRenderFns: code.staticRenderFns
  }
})

  这里就可以看到baseCompile方法其实就是我们整个编译的核心逻辑了，parse就是解析我们模板生成ast，然后对生成的ast做一个optimize优化，最后通过generate方法生成code。这里options就是之前传入的baseOpitons。
  所以我们编译过程就是分为三步：生成AST、AST优化、代码生成。后续我们分析也会围绕这三个方面来重点分析。
