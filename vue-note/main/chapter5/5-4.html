# 5.4 parse之解析模板（下）

  上个小节我们主要分析了在非纯文本元素script、style、textarea情况并且找到第一个'<'的索引为0的情况下的四种情况判断，最后分析的解析到开始标签，那么这个小节我们就来接着往下分析下个循环会做什么事。
  解析完开始标签后，那么后面的内容正常情况下应该是子内容了，这时候假设还是上面的模板，那么现在只剩下这部分内容

我是模板</div>

  然后执行循环时候我们解析到'<'的索引textEnd显然不是0所以就会执行下面逻辑

  let text, rest, next
  if (textEnd >= 0) {}
  if (textEnd < 0) {}
  if (text) {}
  if (options.chars && text) {}

  这里先定义text、rest、next，然后又是四个判断，我们先看textEnd>=0的情况：

  if (textEnd >= 0) {
    rest = html.slice(textEnd)
    while (
      !endTag.test(rest) &&
      !startTagOpen.test(rest) &&
      !comment.test(rest) &&
      !conditionalComment.test(rest)
    ) {
      // < in plain text, be forgiving and treat it as text
      // <在纯文本中，要宽容并将其视为文本
      next = rest.indexOf('<', 1)
      if (next < 0) break
      textEnd += next
      rest = html.slice(textEnd)
    }
    text = html.substring(0, textEnd)
  }

  为什么这里还要判断等于0呢？等于0不是在上面判断的吗？仔细想想，上面真的能处理所有情况吗？即使第一位是'<'也可能不是标签的开头，那么上面的判断实际上是上面都没有做，也不能跳过本次循环。所以'<'完全可以作为小于号出现在文本内容中。了解这个后我们来看接下来的逻辑。
  这里截取'<'开始到最后的字符串保存在rest中，然后rest开头满足这个几个情况时执行循环：
  1. 不是结束标签时
  2. 不是开始标签时
  3. 不是注释节点时
  4. 不是条件注释时

  那么就认为它是纯文本的'<'，这个时候再去从rest的第二个位置开始找下一个'<'的位置，如果找不到则跳出循环。如果找到了就textEnd += next，然后把rest赋值为截取第二个'<'开始到最后的字符串，直到最后找不到'<'或者遇到下个标签的开头为止。这时textEnd就是标签的开头的索引，也可能是匹配不到第二个'<'，那说明剩下字符串都是纯文本。
  先说第一种情况'我是<模板</div>'，这时textEnd是第二个'<'的索引，最后text处理是'我是<模板'，然后接下来就跑到第三个if判断text是否存在了

  if (text) {
    advance(text.length)
  }

  存在则前进text的长度。接下来执行第四个if判断

  if (options.chars && text) {
    options.chars(text, index - text.length, index)
  }

  通过options.chars处理文本节点，同样的chars方法我们放到后面分析parse时再分析。
  如果是第二种情况'我是<模板'，就只有文本内容，这时text处理就是'我是'两个字符，然后剩余'<模板'，同样的会处理text，之后的逻辑都不会执行所以进入下一次循环。
  当然这还没有完，在大while循环开头让last = html，如果本次循环都没有执行advance时显然它两还是相等，最后剩余字符串'<模板'进入新的循环后显然进入前面任何一个判断都不会执行，即使进入textEnd等于0和textEnd>=0的判断也没有任何有效代码执行。最后也是满足last等于html。 这个时候会执行while循环最后逻辑，也就是html === last判断

  if (html === last) {
    options.chars && options.chars(html)
    if (process.env.NODE_ENV !== 'production' && !stack.length && options.warn) {
      options.warn(`Mal-formatted tag at end of template: "${html}"`, { start: index + html.length })
    }
    break
  }

  这个时候直接通过options.chars处理剩余文本，然后跳出整个循环。如果说在非生产环境，并且stack长度为0并且有options.warn方法的话会报一个警告说是模板末尾的标签格式错误。这里stack我们还是暂时不解释是做什么的。
  到这里我们上面四个if判断只有第二个没有分析了，我们来看看它逻辑

  if (textEnd < 0) {
    text = html
  }

textEnd < 0其实就是没有'<'，这种情况显然是没有标签的，那么它就是纯文本了，也就是说目前未处理的字符串都是文本节点，所以直接让text = html，然后经过后面第三个、第四个if去处理text。这样下次循环判断时html就是空了，就不会再执行循环了。
  当然上面几个都是比较极端的情况。正常情况下，我们的文本都会被一个双标签包裹，也就是上面所说的第一种情况：'我是<模板</div>'，经过上述处理后前面的文本被处理了，然后剩下'</div>'进入下次循环。 这时显然就是textEnd === 0判断下的第五种情况：匹配到结束标签。这个我们一直没有分析，我们接下来就来分析它：

  const endTagMatch = html.match(endTag)
  if (endTagMatch) {
    const curIndex = index
    advance(endTagMatch[0].length)
    parseEndTag(endTagMatch[1], curIndex, index)
    continue
  }

  结束标签的情况比较简单，先保存当前index到curIndex中，然后前进匹配到字符的长度，再执行parseEndTag方法后跳到下一次循环。
  我们来看parseEndTag方法，这里传入标签名称、开始索引和结束索引。

  function parseEndTag(tagName, start, end) {
    let pos, lowerCasedTagName
    if (start == null) start = index
    if (end == null) end = index

    // Find the closest opened tag of the same type
    // 查找相同类型的最近打开的标记
    if (tagName) {
      lowerCasedTagName = tagName.toLowerCase()
      for (pos = stack.length - 1; pos >= 0; pos--) {
        if (stack[pos].lowerCasedTag === lowerCasedTagName) {
          break
        }
      }
    } else {
      // If no tag name is provided, clean shop
      pos = 0
    }
    if (pos >= 0) {
      // Close all the open elements, up the stack
      for (let i = stack.length - 1; i >= pos; i--) {
        if (process.env.NODE_ENV !== 'production' &&
          (i > pos || !tagName) &&
          options.warn
        ) {
          options.warn(
            `tag <${stack[i].tag}> has no matching end tag.`,
            { start: stack[i].start, end: stack[i].end }
          )
        }
        if (options.end) {
          options.end(stack[i].tag, start, end)
        }
      }
      // Remove the open elements from the stack
      stack.length = pos
      lastTag = pos && stack[pos - 1].tag
    } else if (lowerCasedTagName === 'br') {
      if (options.start) {
        options.start(tagName, [], true, start, end)
      }
    } else if (lowerCasedTagName === 'p') {
      if (options.start) {
        options.start(tagName, [], false, start, end)
      }
      if (options.end) {
        options.end(tagName, start, end)
      }
    }
  }

  前三句话没什么营养，我们来看if(tagName)的判断，将tagName转化为小写赋值给lowerCasedTagName。pos赋值为stack.length - 1，也就是stack数组的最大索引，然后进行循环。
  这里循环就是从stack数组中从后往前去找与当前匹配到结束标签相同的标签，找到第一个就跳出循环。回想一下stack中存放的是什么，每当解析到一个开始标签就会将标签的信息对象push到stack中。作为一个双标签，它应该是成对存在的，而我们这里匹配到结束标签必然与stack中从后往前找到的第一个相同标签是成对存在的。当然你永远不知道用户会怎样做，如果没有成对存在也可以基于查找情况去做一些判断，从而给用户一些错误提示。
  若没有tagName就直接让pos = 0。
  接下来如果pos是大于等于0的，然后就再一次从后往前遍历stack，这里定义的索引是i，之前遍历的索引是pos，如果i > pos或者tagName不存在就要去报一个警告，第一次循环时i为stack的栈顶，如果i > pos说明pos不是栈顶索引，这就说明这个标签没有匹配到结束标签，也就是下面这种情况：

    <div id="app"><a>我是a</div>

  这种情况下stack中有两个元素，一个是div，一个是a，解析到div的结束标签时pos是0，所以这里遍历到a时i=1是大于pos的，说明a标签没有闭合，这就要报一个警告了。然后还要通过options.end方法处理没有闭合的情况。
  然后stack的长度设置为pos，实际上就是匹配开始标签就push，匹配到结束标签就可以把相应的开始标签pop，但是对于我们上面标签不能闭合情况显然pop是不够的，需要把对应开始标签以及后面的内容都释放，所以这里直接设置length。然后取到栈顶元素tag赋值给lastTag，也就是接下来处理的标签。
  接下来是pos < 0，这部分是分为两种情况去判断，一种是br标签、一种是p标签。注意这里是找不到相应开始标签情况。这就要解释一下浏览器另一个默认行为：如果有这样一个模板：

    <div>
      </br></p>
    </div>

  这时</br>能被解析为<br>。</p>能被解析为正常的<p></p>。所以在这里我们也需要和浏览器默认行为一致。如果是br的话要用options.start方法处理，如果是p的话则分别需要options.start和options.end处理。
  到这里我们就清楚stack到底是做什么用的了。再回过头来分析一下我们之前埋下的坑，在解析开始标签时执行了一个handleStartTag方法，里面有这样一段逻辑

  if (expectHTML) {
    if (lastTag === 'p' && isNonPhrasingTag(tagName)) {
      parseEndTag(lastTag)
    }
    if (canBeLeftOpenTag(tagName) && lastTag === tagName) {
      parseEndTag(tagName)
    }
  }

  如果expectHTML为true 的话，会执行里面内容。先判断lastTag是一个p标签的话并且再判断是否是短语内容标签。isNonPhrasingTag这些方法可以在util文档中找到。那这段逻辑是什么意思呢？
  我们来看这样一个模板

new Vue({
  el: '#app',
  template: '<p><h2>我是模板</h2></p>'
})

  在p标签中嵌套了h2标签，在parseHTML时解析到p开始标签后会将lastTag赋值为'p'，这是我们之前分析过的。然后在解析h2开始标签时，这时lastTag为就是'p'，然后通过isNonPhrasingTag判断h2是一个非短语内容标签，这样的标签是不能放到p标签中的，所以调用parseEndTag(lastTag)方法去处理p标签，这里主要是通过里面的options.end方法去处理p标签的。当然处理之后实际上是把p标签给闭合了:
'<p></p><h2>我是模板</h2></p>'
  这样stack就会删除p，之后匹配到后面p的结束标签就在stack中找不到p了，但是也不用慌，parseEndTag后面专门处理了p标签的情况，通过options.start方法处理，不用说这里是给它补全开始标签了
'<p></p><h2>我是模板</h2></p></p>'
  为什么会处理成这样我们之后分析了options中的方法就明白了，这里我们先知道就好了。
  我们之前一直讨论非纯文本标签的情况，接下来我们来分析纯文本标签。

if (!lastTag || !isPlainTextElement(lastTag)) {} 
else {
  let endTagLength = 0
  const stackedTag = lastTag.toLowerCase()
  const reStackedTag = reCache[stackedTag] || (reCache[stackedTag] = new RegExp('([\\s\\S]*?)(</' + stackedTag + '[^>]*>)', 'i'))
  const rest = html.replace(reStackedTag, function (all, text, endTag) {
    endTagLength = endTag.length
    if (!isPlainTextElement(stackedTag) && stackedTag !== 'noscript') {
      text = text
        .replace(/<!\--([\s\S]*?)-->/g, '$1') // #7298
        .replace(/<!\[CDATA\[([\s\S]*?)]]>/g, '$1')
    }
    if (shouldIgnoreFirstNewline(stackedTag, text)) {
      text = text.slice(1)
    }
    if (options.chars) {
      options.chars(text)
    }
    return ''
  })
  index += html.length - rest.length
  html = rest
  parseEndTag(stackedTag, index - endTagLength, index)
}

  如果是一个纯文本标签，则会执行else逻辑，先定义endTagLength初始值为0，然后stackedTag为lastTag的小写字符。reCache是全局定义的一个对象，然后这里是通过标签名去取它的正则，如果没有的话就去生成一个正则保存进去。那这个正则是用来干什么的呢？其实是用来匹配文本的，它有两个捕获组，一个是匹配任意内容，直到匹配到第二个捕获组内容停止，那第二个捕获组是</' + stackedTag + '[^>]*>，其实就是捕获结束标签</stackedTag>，当然这里stackedTag是一个动态内容，比如说前面匹配是stackedTag就是textarea，所以这个正则第一个捕获组匹配到的就是文本内容。
  然后这个正则就通过replace方法去截取html得到rest。replace回调函数则是获取结束标签长度赋值给endTagLength。然后这个判断有点奇怪，因为lastTag是纯文本标签才会执行这段代码，而这里又判断是非纯文本标签执行。我们先跳过他去看下面逻辑，这里还是判断需要不需要跳过开头空白字符，最后执行options.chars处理文本内容。当然这里要返回replace替换的内容，这里是替换为''。
  然后就是重新计算index值，并重新给html赋值，这两句相当于是advance方法。最后调用parseEndTag方法处理标签。
  到这里我们while循环就都分析完毕了。这是parseHTML的主要逻辑，执行完循环后就只剩下最后一段逻辑

  // Clean up any remaining tags
  // 清理所有剩余标签
  parseEndTag()

  没有传任何参数的话，执行parseEndTag时pos恒为0，最终会将stack数组清空。到目前为止我们有三种不同的parseEndTag调用，第一种传了三个参数，第二种只传了tagName，第三种情况就是这种什么都不传的情况，那就会用来清空stack。这几种情况调用的区别我们在后面会来分析。
