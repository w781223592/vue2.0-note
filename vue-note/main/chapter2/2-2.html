# 2.2 Vue实例挂载的实现

Vue实例挂载是通过vm.$mount方法实现的，$mount方法在很多文件下都有定义，因为它的实现和平台与构建都有关系，我们先看entry-runtime-with-compiler.js文件下的定义

const mount = Vue.prototype.$mount
Vue.prototype.$mount = function (
  el?: string | Element,
  hydrating?: boolean
): Component {
  el = el && query(el)
  // 省略若干判断
  return mount.call(this, el, hydrating)
}

它先用mount变量缓存了Vue.prototype.$mount方法，之后又重新定义了$mount方法，抛开中间if判断不谈，重写的$mount方法最后返回mount方法（也就是原始的$mount）的调用。 首先我们分析query方法

export function query (el: string | Element): Element {
  if (typeof el === 'string') {
    const selected = document.querySelector(el)
    if (!selected) {
      process.env.NODE_ENV !== 'production' && warn(
        'Cannot find element: ' + el
      )
      return document.createElement('div')
    }
    return selected
  } else {
    return el
  }
}

query方法接收一个参数el，el是字符串或者dom对象，首先判断el是否是一个字符串类型，尝试通过document.querySelector(el)方法去获取dom，如果获取不到dom元素，则在开发时报一个警告并通过document.createElement('div')方法创建一个空的div，如果能获取到则返回dom元素；若el已经是一个dom元素则直接返回el。

if (el === document.body || el === document.documentElement) {
    process.env.NODE_ENV !== 'production' && warn(
      `Do not mount Vue to <html> or <body> - mount to normal elements instead.`
    )
    return this
  }

得到el后，再看上面这段代码，判断el === document.body || el === document.documentElement，如果el是body元素或者html元素，则报一个警告，因为Vue的挂载会替换整个元素，显然body和html元素是不能被替换的。 然后我们继续分析下面的代码

if (!options.render) {
    let template = options.template
    if (template) {
      if (typeof template === 'string') {
        if (template.charAt(0) === '#') {
          template = idToTemplate(template)
          /* istanbul ignore if */
          if (process.env.NODE_ENV !== 'production' && !template) {
            warn(
              `Template element not found or is empty: ${options.template}`,
              this
            )
          }
        }
      } else if (template.nodeType) { // 如果是一个dom对象
        template = template.innerHTML
      } else {
        if (process.env.NODE_ENV !== 'production') {
          warn('invalid template option:' + template, this)
        }
        return this
      }
    } else if (el) { // 如果没有template
      template = getOuterHTML(el)
    }
// 编译相关代码到编译章节再分析
}

如果没有写render函数才执行上面逻辑，先判断options上有没有template，如果有template则判断它是否是字符串并且第一位是#，则使template = idToTemplate(template)，下面是idToTemplate方法

const idToTemplate = cached(id => {
  const el = query(id)
  return el && el.innerHTML
})
export function cached<F: Function> (fn: F): F {
  const cache = Object.create(null)
  return (function cachedFn (str: string) {
    const hit = cache[str]
    return hit || (cache[str] = fn(str))
  }: any)}

idToTemplate函数最后执行的其实就是cached函数传入的函数参数，那为什么不直接执行内部逻辑而是大费周章去定义cached函数并通过一些列转化甚至使用闭包最后才执行呢？这里的cached函数实现了纯函数的缓存，先说一下什么是纯函数吧，简单来说纯函数就是参数一样返回值也一样的函数，这里通过cached函数把返回结果根据传入字符串的不同缓存下来，之后再重复访问无需重复调用方法。在这里idToTemplate方法传入template字符串，实际上就是id，之后通过query方法返回其dom，最后返回innerHtml，如果返回值不存在则会报一个警告。 继续分析template不是字符串的情况，如果template.nodeType存在则是一个dom对象，这时直接使template = template.innerHtml，如果是其他情况，则报一个警告并返回this，如果没有template，则在el存在时，执行template = getOuterHTML(el)， getOuterHTML方法定义如下：

/**
 * Get outerHTML of elements, taking care
 * of SVG elements in IE as well.
 */
function getOuterHTML (el: Element): string {
  if (el.outerHTML) {
    return el.outerHTML
  } else {
    const container = document.createElement('div')
    container.appendChild(el.cloneNode(true))
    return container.innerHTML
  }
}

getOuterHTML将不同的el转化成相同的格式，getOuterHTML将传入的不同el进行处理，如果el.outerHTML存在则直接返回el.outerHTML，如果不存在则el可能是个文本节点之类的，就在外面包裹一层div然后返回它的innerHTML，getOuterHTML方法最后返回的是html字符串，之后就执行编译相关逻辑了，编译相关逻辑将在后面着重分析。这儿简单了解一下就好。如下所示，options本身是没有render方法的，执行下面逻辑后便被赋值了一个render，之后分析mount方法我们会看到render的用途。

  if (template) { // 编译相关
    const { render, staticRenderFns } = compileToFunctions(template, {
        outputSourceRange: process.env.NODE_ENV !== 'production',
        shouldDecodeNewlines,
        shouldDecodeNewlinesForHref,
        delimiters: options.delimiters,
        comments: options.comments
      }, this)
     options.render = render
     options.staticRenderFns = staticRenderFns
  }

到此为止，编译版本的$mount方法的实现我们基本上都分析完了，不管它有多复杂，都是为最后的结果做准备，实际上就是为了最后mount.call做准备。回过头来再看$mount方法，其实我们没有看到有关挂载的逻辑，所以挂载有关逻辑只能是在mount上，$mount重写实际上就是加上了编译版本，整体逻辑也是为了将template转化为render函数，事实上Vue渲染过程中是只认render函数的，无论你是怎么去写模板最后都要转化为render函数。 接下来我们分析mount方法，实际上mount方法就是only版本使用的$mount，它的定义在 runtime/index.js文件下

// public mount method
Vue.prototype.$mount = function (
  el?: string | Element,
  hydrating?: boolean
): Component {  
  el = el && inBrowser ? query(el) : undefined
  return mountComponent(this, el, hydrating)
}

该方法定义十分简单，接收两个参数el和hydrating，hydrating表示是否是服务端渲染，今后的分析中认为它就是false。mount方法与$mount方法一样先去处理el，处理前先判断el存在且是浏览器环境，因为Vue是有可能跑在服务端的，处理完el后返回mountComponent(this, el, hydrating)方法。 在core/util/env文件下定义了许多环境相关的判断，其中判断是否是浏览器环境判断如下：

const inBrowser = typeof window !== 'undefined'

这小节主要分析了compiler版本下$mount方法的执行过程，下一小节，我们将分析mountComponent方法的实现。
