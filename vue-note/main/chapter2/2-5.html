# 2.5 最终的挂载方法_update

前两个小节我们了解到vm._render方法返回值是vnode

这里我们打印了一下我们写的dome生成的vnode，然后我们再回过头看lifecycle.js下的vm._update(vm.render, hydrating)方法实际上是vm._update(vnode, false)，其定义就在当前文件下，在lifecycleMixin方法下定义了vm.prototype._update，lifecycleMixin方法的调用是在instance下index.js中执行的一系列混入方法，lifecycleMixin方法在Vue原型上挂载三个方法，我们现在只关心_update方法。

Vue.prototype._update = function (vnode: VNode, hydrating?: boolean) {
    const vm: Component = this
    const prevEl = vm.$el
    const prevVnode = vm._vnode
    const restoreActiveInstance = setActiveInstance(vm)
    vm._vnode = vnode
    // Vue.prototype.__patch__ is injected in entry points
    // based on the rendering backend used.
    if (!prevVnode) {
      // initial render
      vm.$el = vm.__patch__(vm.$el, vnode, hydrating, false /* removeOnly */)
    } else {
      // updates
      vm.$el = vm.__patch__(prevVnode, vnode)
    } 
    restoreActiveInstance()
    // update __vue__ reference
    if (prevEl) {
      prevEl.__vue__ = null
    }
    if (vm.$el) {
      vm.$el.__vue__ = vm
    }
    // if parent is an HOC, update its $el as well
    if (vm.$vnode && vm.$parent && vm.$vnode === vm.$parent._vnode) {
      vm.$parent.$el = vm.$el
    }
    // updated hook is called by the scheduler to ensure that children are
    // updated in a parent's updated hook.
  }

该方法调用时机有两个一个是首次渲染，另一个是数据更新时候，开始定义了一些变量，这部分变量是用于数据更新时使用的，我们先分析首次渲染的情况，接下来if...else也是基于这两种情况的判断，我们来看prevVnode为null的时候，也就是首次渲染时候，vm.$el = vm.__patch__(vm.$el, vnode, hydrating, false /* removeOnly */)，就是执行了__patch__方法，我们来看看__patch__方法的定义，其定义于web/runtime/index文件下，它的定义还要在通用$mount之前

// install platform patch function
Vue.prototype.__patch__ = inBrowser ? patch : noop

在浏览器环境定义为patch方法，因为Vue可以跑在服务端，如果不在浏览器环境就定义为空函数noop，然后再看patch方法的定义，patch方法定义于runtime/patch方法下：

import * as nodeOps from 'web/runtime/node-ops'
import { createPatchFunction } from 'core/vdom/patch'
import baseModules from 'core/vdom/modules/index'
import platformModules from 'web/runtime/modules/index'

// the directive module should be applied last, after all
// built-in modules have been applied.
const modules = platformModules.concat(baseModules)
export const patch: Function = createPatchFunction({ nodeOps, modules })

可以看到patch方法是createPatchFunction({ nodeOps, modules })，其中nodeOps是一些对dom方法的封装，baseModules是一些公共模块，platformModules是平台相关的模块（attrs、class、style的一些钩子定义），最后modules是baseModules和platformNodules的合集。然后再看createPatchFunction方法，其方法定义于core/ vdom/patch.js文件下

可以看到该方法是一个很复杂函数，内部定义的方法超过20个，但我们只需要分析其中一部分方法，我们看到最后createPatchFunction方法返回了patch方法，在这里我们先来思考一个问题，为什么Vue不直接定义patch方法，而是绕这么一个圈通过闭包的方式去返回patch呢？这里其实是使用了一个叫函数柯里化的技巧，如果直接定义patch方法，我们少不了每次去if...else判断是哪个平台，比如说如果是web平台就...，如果是weex平台就...，这样每次都需要判断平台和传入平台差异化参数，而使用函数柯里化技巧将这些差异化参数一次抹平了，通过闭包的方式实现nodeOps和modules的持有，使得我们在使用它的过程中不再需要关心是什么平台，也不需要去传这些差异化参数了，这种编程技巧Vue在很多地方都用到了，是一个十分值得我们学习的编程思想。 下面我们言归正传来分析一下createPatchFunction方法，首先它从backend中解构出modules和nodeOps，遍历hooks，在cbs = {}中添加hooks[i]属性为空数组，然后遍历modules数组，如果modulse[j]中有hooks[i]属性，就将其属性值push到obs对象对应的属性下。下面我们看看hooks是个什么东西

const hooks = ['create', 'activate', 'update', 'remove', 'destroy']

可以看到其实就是patch过程中的生命周期钩子字段定义。然后我们随便找个modules看看里面的内容，以runtime/modules/attrs.js为例，它里面定义了很多方法，我们只看它的导出内容

export default {
  create: updateAttrs,
  update: updateAttrs
}

导出的对象的key是hooks集合的一部分，到这里就很清晰了，cbs的各个属性就是patch过程中的生命周期，属性值就是这个周期要执行的函数集。 接下来定义了很多patch过程中的方法，所以我们直接来看patch方法。

 // oldeVnode为真实dom
  return function patch (oldVnode, vnode, hydrating, removeOnly) {}

分析patch方法之前我们先看看patch方法是怎么调用的在vm._update方法中调用的vm.__patch__(vm.$el, vnode, hydrating, false)实际上就是patch方法，这儿传入四个参数，第一个是vm.$el，第二个是vnode也就是之前们调用vm._render()生成的vnode，后两个参数都是false，再来看patch接收的参数同样也是四个，第一个参数被命名为oldVnode，所以它肯定是一个旧的vnode，实际上它是一个dom对象，也是要被替换的dom对象，那这里的vm.$el是什么呢，其实之前我们已经见过了$el，在mount过程中我们通过query(el)得到的el就是个dom对象了，在执行mountComponent方法的时候el被赋值给vm.$el。 接下来我们看看什么patch做了什么事。patch方法详解。 patch方法最终实现vnode到dom的转换，我们目前只分析了第一次渲染下的使用render函数的渲染，使用模板时候的编译，使用组件时候的处理，甚至还有很多init方法没有分析到，所以革命尚未成功，还需继续努力。接下来我们就分析有关组件的渲染。 下面把我们调试使用的dome列出来
