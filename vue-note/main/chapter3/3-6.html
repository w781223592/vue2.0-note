# 3.6 组件的注册

  上小节我们分析了组件的生命周期，这个小节我们来分析一下组件的注册过程，组件的注册是比较简单的，我们知道组件的注册方式有两种，一种是全局注册，一种是局部注册。 我们先来看下面dome

Vue.component('app', app)

new Vue({
  el: '#app',
  template: '<app></app>'
})

  我们全局注册组件时会使用Vue.component方法，这显然是一个全局api，我们之前已经见过Vue.mixins、Vue.use、Vue.extend等全局api，它们都在global-api下定义，那么Vue.component是不是也在这里被定义呢？我们发现上面三个api都有单独的文件去定义，但是component没有，它们一个共同点都是在index.js中去调用的，所以我们去index.js中去找相关逻辑，在initGlobalAPI最后有这样一段逻辑，显然它们就是扩展我们上述所说的api的

initUse(Vue)
initMixin(Vue)
initExtend(Vue)
initAssetRegisters(Vue)

  上面四个方法前三个我们其实已经分析过了，我们来看最后一个方法做了什么？究竟它与我们Vue.component有没有关系。initAssetRegisters方法定义于assets.js文件下，这是我们唯一一个没有分析过的全局api逻辑，其实现如下

const ASSET_TYPES = [
  'component',
  'directive',
  'filter'
]
export function initAssetRegisters (Vue: GlobalAPI) {
  /**
   * Create asset registration methods.
   */
  ASSET_TYPES.forEach(type => {
    Vue[type] = function (
      id: string,
      definition: Function | Object
    ): Function | Object | void {
      if (!definition) {
        return this.options[type + 's'][id]
      } else {
        /* istanbul ignore if */
        if (process.env.NODE_ENV !== 'production' && type === 'component') {
          validateComponentName(id)
        }
        if (type === 'component' && isPlainObject(definition)) {
          definition.name = definition.name || id
          definition = this.options._base.extend(definition)
        }
        if (type === 'directive' && typeof definition === 'function') {
          definition = { bind: definition, update: definition }
        }
        this.options[type + 's'][id] = definition
        return definition
      }
    }
  })
}

  可以发现这个逻辑确实与Vue.component有关，但是这里是处理了三个api，所以我们没有找到component.js文件，因为这里不单单处理了component，还有directive和filter。我们来分析具体逻辑，就是再Vue上添加这几个方法，接收两个参数，第一个参数是id，为string类型，第二个参数definition，是一个函数或者对象。我们来看内部逻辑，如果没有传入第二个参数就直接从options中找到对应type对象中对应id的值返回；否则我们执行else中逻辑，在开发环境中判断注册组件时就校验id是不是合法的组件名称，具体检验方法我们之前已经分析过了。然后再判断如果是注册组件的话就给它设置组件名，然后将definition赋值为一个基于definition本身为options的Vue的子构造器；如果是注册指令且第二个参数应该是个函数，满足这两个条件的话我们就将definition设置为一个标准化的指令对象，有关标准化的内容我们在合并配置文档中其实已经分析过了；最后就是将options中与type属性对应的对象添加id属性，属性值就是definition。最后我们返回definition。
  分析完Vue.component方法定义后我们来结合我们例子来看看它是如何工作的吧。我们这个例子实际上是与我们之前分析完全不同的，在这之前我们一直依赖render函数去渲染，而不是template，这是有比较大区别的，我们都知道vue有带编译版本也就是runtime + compiler版本，也有不带编译器的runtime only版本，前者是支持我们写template的，我们使用template会被编译器处理，在生成vnode时会将<app></app>当做一个标签去处理，这时tag是字符串'app'，而我们之前采用render函数时是这样写的

new Vue({
  el: '#app',
  render: (h) => h(app)
})

  这里的app是一个object类型，那么这有什么不同呢？我们来看之前提到生成vnode的_createElement方法，如下所示：

export function _createElement(
  context: Component,
  tag?: string | Class<Component> | Function | Object,
  data?: VNodeData,
  children?: any,
  normalizationType?: number
): VNode | Array<VNode> {
  let vnode, ns
  if (typeof tag === 'string') {
    let Ctor
    ns = (context.$vnode && context.$vnode.ns) || config.getTagNamespace(tag)
    if (config.isReservedTag(tag)) {
      // platform built-in elements
      if (process.env.NODE_ENV !== 'production' && isDef(data) && isDef(data.nativeOn) && data.tag !== 'component') {
        warn(
          `The .native modifier for v-on is only valid on components but it was used on <${tag}>.`,
          context
        )
      }
      vnode = new VNode(
        config.parsePlatformTagName(tag), data, children,
        undefined, undefined, context
      )
    } else if ((!data || !data.pre) && isDef(Ctor = resolveAsset(context.$options, 'components', tag))) {
      // component
      vnode = createComponent(Ctor, data, context, children, tag)
    } else {
      // unknown or unlisted namespaced elements
      // check at runtime because it may get assigned a namespace when its
      // parent normalizes children
      vnode = new VNode(
        tag, data, children,
        undefined, undefined, context
      )
    }
  } else {
    // direct component options / constructor
    vnode = createComponent(tag, data, context, children)
  }
}

  这里我们其实是节选了一部分逻辑的，我们很容易可以看出这两种写法会有什么区别，如果使用template写法的话tag会是一个string，它会走if逻辑，如果是用的render函数写法的话tag是一个object，它会走else逻辑去创建一个组件vnode。其实render函数也可以写成下面这样：

new Vue({
  el: '#app',
  // template: '<app></app>',
  render: (h) => h('app')
})

  将app改为字符串'app'，这也会走我们的if逻辑了，那接下来我们就看看if逻辑的内容，可以看到if中又有三个分支：先是判断tag是不是平台保留标签，如果是的话则会创建一普通vnode；然后是else if判断data不存在，然后执行resolveAsset方法，如果其返回不为空的话就去创建组件vnode，所以我们组件vnode创建是也是有两种情况的；最后一个分支就是tag标签是未知的，然后仍然创建一个vnode。
  下面我们来分析resolveAsset方法

export function resolveAsset (
  options: Object,
  type: string,
  id: string,
  warnMissing?: boolean
): any {
  /* istanbul ignore if */
  if (typeof id !== 'string') {
    return
  }
  const assets = options[type]
  // check local registration variations first
  if (hasOwn(assets, id)) return assets[id]
  const camelizedId = camelize(id)
  if (hasOwn(assets, camelizedId)) return assets[camelizedId]
  const PascalCaseId = capitalize(camelizedId)
  if (hasOwn(assets, PascalCaseId)) return assets[PascalCaseId]
  // fallback to prototype chain
  const res = assets[id] || assets[camelizedId] || assets[PascalCaseId]
  if (process.env.NODE_ENV !== 'production' && warnMissing && !res) {
    warn(
      'Failed to resolve ' + type.slice(0, -1) + ': ' + id,
      options
    )
  }
  return res
}

  该方法定义于util目录下options.js中，它第一个参数options，我们传入的是context.$options；第二个参数是type，我们传入的是components字符串；第三个参数id，我们这里传入了tag，也就是我们写的'app'；第四个参数我们没有传所以先忽略掉。 接下来我们看内部逻辑，首先判断id不是string时候return，显然我们这里必然是一个string，否则也不会执行这个逻辑，然后我们取options上的type属性，也就是去找components属性，并将其值保存到assets中。然后判断assets上有没有tag所对应的自有属性，如果存在的话就返回其值，否则通过camelize方法将连字符转小驼峰再去判断有没有属性，如果还没有则再通过capitalize方法转大驼峰去判断，如果都没有找到那么再尝试从它的原型上去找，然后取原型上获取的结果返回。 这里显然是能获取到我们的全局组件app的，我们也可以通过代码调试的方式来看一下assets是个什么东西。

  可以看到它就是个空的对象，但是它原型对象上有我们Vue的内置组件和我们全局注册的app组件，所以我们通过resolveAsset方法是从原型上获取了全局注册的组件。我们可以看到app是不同于其他组件的，它是一个构造函数，这也就对应上我们上面分析的initAssetRegisters方法了，它在处理component时就是基于Vue.extend创建了一个子类构造器，最后将它添加到Vue.options上，所以我们这里取到的就是个构造器。取到构造器后我们在createComponent时候将它作为第一个参数传入来创建组件vnode，我们之前3.1小节分析的情况是传入一个组件对象，但是不用担心，这个过程是一致的，我们传入对象时候只是多执行了一步Vue.extend去获取Ctor。再之后就是我们熟悉的过程了，我们也没必要重复赘述了。 但是我们的问题还没有解决，因为我们还不知道它们是怎么跑到context.$options原型上的。这肯定是与我们之前分析的合并配置有关了，我们来看Vue初始化中执行的mergeOptions方法，这里的options其实就是我们根vm实例的$options

  而vm.$options就是Vue.options和我们传入的options通过mergeOptions方法合并的。我们之前花了大篇幅去讲合并配置，但是有关components的合并没有去分析，这里我们就来看看关于components策略

function mergeAssets (
  parentVal: ?Object,
  childVal: ?Object,
  vm?: Component,
  key: string
): Object {
  const res = Object.create(parentVal || null)
  if (childVal) {
    process.env.NODE_ENV !== 'production' && assertObjectType(key, childVal, vm)
    return extend(res, childVal)
  } else {
    return res
  }
}

ASSET_TYPES.forEach(function (type) {
  strats[type + 's'] = mergeAssets
})

  我们来看options.js中的合并策略，ASSET_TYPES已经见过很多次了，这里就不解释了。这里遍历ASSET_TYPES这个数组然后给strats添加type + 's'属性，也就是这里将component描述为components，他们的合并策略都是mergeAssets函数。具体合并逻辑是先通过Object.create创建一个继承parentVal的对象，如果parentVal就创建一个空的对象，如果childVal存在就通过extend合并创建的对象和childVal，否则就直接返回创建的对象。这里还有一个开发环境的校验，我们来看assertObjectType方法

function assertObjectType (name: string, value: any, vm: ?Component) {
  if (!isPlainObject(value)) {
    warn(
      `Invalid value for option "${name}": expected an Object, ` +
      `but got ${toRawType(value)}.`,
      vm
    )
  }
}

  这里第一个参数name，也就是传入的key，这里就是我们的components，第二个参数value，这里传入的是childVal。执行逻辑就是简单判断value如果不是一个纯的对象的话就报一个警告，大致意思是我们写了一个无效的属性值，期望是一个对象，但是获得一个其他类型，这里toRawType会输出数据的原始类型，我们把它放到util文档中。
  这个合并是比较简单的，但是有两个地方需要注意：这里parentVal是通过继承来合并的，parentVal其实就是Vue.options[type + 's']；这里的childVal是我们书写的options配置中的这几个type属性，这里的几个配置项都要求是对象格式。
  这就解决了我们的问题，因为返回的对象是由Object.create方法继承parentVal而来的，所以Vue.options.components中的内容会继承到res的原型对象中。之后我们就可以在components的原型中访问到我们全局注册的app组件构造器，然后我们将获取的构造器传递给createComponent函数生成vnode，再通过patch方法渲染到页面上。
  上面我们分析了全局注册组件的过程，下面我们来看看局部注册组件是怎么工作的，我们在上面例子的基础上，在app组件中注册一个局部组件About，下面来看看我们是怎样书写的

<template>
  <div>
    hello world
    <About></About>
  </div>
</template>

<script>
import About from '../About.vue'
export default {
  data() {
    return {
      b: '我是app'
    }
  },
  components: { About }
}
</script>
