# 3.4生命周期（上）

  组件生命周期是我们组件化开发过程的一个重要内容，准确的理解生命周期的执行时机和过程有助于我们避免开发过程中的一些问题。
  我们先来看lifecycle.js中的callHook函数

export function callHook(vm: Component, hook: string) {
  // #7573 disable dep collection when invoking lifecycle hooks
  // 调用生命周期时禁用依赖收集
  pushTarget()
  const handlers = vm.$options[hook]
  const info = `${hook} hook`
  if (handlers) {
    for (let i = 0, j = handlers.length; i < j; i++) {
      invokeWithErrorHandling(handlers[i], vm, null, vm, info)
    }
  }
  if (vm._hasHookEvent) {
    vm.$emit('hook:' + hook)
  }
  popTarget()
}

  其中有些依赖收集相关内容我们先不去分析。我们之前合并options后的hook是一个数组。所以这里在存在hook数组时候变量handlers，循环调用invokeWithErrorHanding方法，最后判断如果vm._hasHookEvent为true，则执行vm.$emit('hook:' + hook)，这与组件的hook监听事件有关，我们也暂时不做分析。
  下面我们来看看invokeWithErrorHanding方法，我们先来看看传入的这几个参数是什么handlers[i]就是我们的生命周期函数，vm则是vm实例，第三个是null，第四个又是vm，第五个是info，既有重复参数又有空的参数，那么这个方法应该是公共方法，应该不止一个地方会用到，我们来看看它的内部实现。

export function invokeWithErrorHandling (
  handler: Function,
  context: any,
  args: null | any[],
  vm: any,
  info: string
) {
  let res
  try {
    res = args ? handler.apply(context, args) : handler.call(context)
    if (res && !res._isVue && isPromise(res) && !res._handled) {
      res.catch(e => handleError(e, vm, info + ` (Promise/async)`))
      // issue #9511
      // avoid catch triggering multiple times when nested calls
      res._handled = true
    }
  } catch (e) {
    handleError(e, vm, info)
  }
  return res
}

  该方法定义与util/error.js下，该方法就是去尝试去调用hook函数并尝试处理一些错误。 接下来我们看_init执行过程中执行了哪些生命周期

    vm._self = vm
    initLifecycle(vm)
    initEvents(vm)
    initRender(vm)
    callHook(vm, 'beforeCreate')
    initInjections(vm) // resolve injections before data/props
    initState(vm)
    initProvide(vm) // resolve provide after data/props
    callHook(vm, 'created')

  可以看到beforeCreate调用时机在initEvents和initRender之后，这时候data数据还没有被初始化，所以这个时候我们是取不到任何数据的。
  我们再来看为什么时候会执行mounted钩子，在mountComponent函数执行时，开头先执行了

callHook(vm, 'beforeMount')

  最后又执行了

  if (vm.$vnode == null) {
    vm._isMounted = true
    callHook(vm, 'mounted')
  }

  我们在_render中将$vnode赋值为_parentVnode，我们说它是父级占位Vnode，那么什么时候它为null呢，当然是没有父级的时候，也就是vm是根实例，这时候会执行mouted钩子，这是mountComponent最后的逻辑，到这个时候整个应用patch都已经执行完成了，所以很容易得出根组件的mounted是最后执行，这也符合我们逻辑，因为它表示是整个应用挂载完成，这意味着它所有的子组件都已经mounted了。但是我们可以看到的是beforeCreate、create、beforeMount是先执行的，因为这时候子组件还没有实例化，只有一个根组件，这意味着我们想在根组件生命周期中添加一些公共的属性并想在子组件生命周期中使用，那么一定得注意不要在mounted中去操作。
  那么子组件的mounted钩子会在什么时候调用呢？我们来看patch函数最后一段逻辑

invokeInsertHook(vnode, insertedVnodeQueue, isInitialPatch)

  invokeInsertHook逻辑如下

function invokeInsertHook(vnode, queue, initial) {
  // delay insert hooks for component root nodes, invoke them after the
  // element is really inserted 延迟为组件根节点插入钩子，在真正插入元素后调用它们
  if (isTrue(initial) && isDef(vnode.parent)) {
    vnode.parent.data.pendingInsert = queue
  } else {
    for (let i = 0; i < queue.length; ++i) {
      queue[i].data.hook.insert(queue[i])
    }
  }
}

  可以看到方法接收三个参数vnode、queue、initial，我们先看后两个参数的初始定义

const insertedVnodeQueue = []
let isInitialPatch = false

  queue开始是一个空数组，正如它的命名，它其实是vnode的插入队列，第三个参数标识是否是初始的patch

if (isUndef(oldVnode)) {
  // empty mount (likely as component), create new root element
  isInitialPatch = true
  createElm(vnode, insertedVnodeQueue)
}

  我们看patch方法中的这段逻辑，在没oldVnode情况下，实际上就是初始渲染时候，也可以说是挂载阶段，这时将isInitialPatch置为true。这直接影响到我们invokeInsertHook的执行逻辑。isInitialPatch为true并且vnode.parent存在的时候，也就是在初始化过程中并且是非根组件的patch过程中，因为这个时候才存在vnode.parent。这时候设置vnode.parent.data.pendingInsert为queue，否则就去遍历queue，并执行queue[i].data.hook.insert()方法。看到这里你可能会很懵，因为两个问题没有得到答案：⑴ queue数组什么时候被添加元素。⑵ data.hook是什么东西，又是什么时候被赋值的。
  不要急，下面我们来慢慢寻找答案。先来找什么时候改变了queue，因为data.hook也只是queue中元素的属性，我们可以看到insertVnodeQueue使用的地方不多，而且显然在invokeInsertHook中是使用它而不是修改它，那只有patchVnode中和createElm方法中了，patchVnode定义比较复杂，而且没有明显的修改逻辑，所以我们把目光放到createElm中，其实之前我们在patch文档中已经分析过createElm方法了，所以我们这里只分析部分内容

createChildren(vnode, children, insertedVnodeQueue)
if (isDef(data)) {
   invokeCreateHooks(vnode, insertedVnodeQueue)
}
insert(parentElm, vnode.elm, refElm)

  在createElm方法中有这样一段逻辑，在if中判断data是否为空，data就是vnode.data也就是我们在写render函数时传入的第二个配置参数对象，我们已经知道h函数第一个参数通常是tag标签名，第二个对象一般是data配置对象，如果没有配置也可以传字符串或者是数组作为children。还有一种情况就传入tag是一个object，也就是组件的配置，这种情况我们一般直接返回h(component)，而不去定义它的data。

  下面来看invokeCreateHooks

  function invokeCreateHooks(vnode, insertedVnodeQueue) {
    for (let i = 0; i < cbs.create.length; ++i) {
      cbs.create[i](emptyNode, vnode)
    }
    i = vnode.data.hook // Reuse variable
    if (isDef(i)) {
      if (isDef(i.create)) i.create(emptyNode, vnode)
      if (isDef(i.insert)) insertedVnodeQueue.push(vnode)
    }
  }

  开头for循环是对vnode的属性做一些处理，比如class，style，event等，这个我们就不重复分析了，主要看后面逻辑也就是看vnode.data.hook是否存在，如果存在再判断有没有create方法，有的话则调用，之后再判断有没有insert方法，如果有的话在这里我们把vnode push到insertedVnodeQueue中，也就是说这里insertedVnodeQueue中的vnode都是有vnode.data.hook.insert的。
  虽然我们的分析看似没有问题，但它有个严重的漏洞，一般的vnode是没有data.hook的，实际上正常它只执行到for循环便结束了，因为只有组件vnode才会有data.hook，而且组件vnode根本不会执行我们上面的逻辑，那么为什么这里还要做这个if判断，显然这个判断不是为这里准备的，所以我们不妨去看看其他地方是否调用了这个方法，我们来看下面逻辑：

if (createComponent(vnode, insertedVnodeQueue, parentElm, refElm)) {
  return
}

WARNING

只有组件vnode有hook是错误的一种说法。只能说受限于当初的认知，才有的这一段描述，但我不打算改正，在后面的章节中我们会发现特殊情况下一些普通节点vnode也可能被添加hook。

  事实上组件vnode走的是createElm方法开头这段逻辑，后面都不会被执行，这我们在3.2小节已经讲述过了，但是createComponent还有这么一段逻辑我们没有分析

if (isDef(vnode.componentInstance)) {
  initComponent(vnode, insertedVnodeQueue)
  insert(parentElm, vnode.elm, refElm)
  if (isTrue(isReactivated)) {
    reactivateComponent(vnode, insertedVnodeQueue, parentElm, refElm)
  }
  return true
}

vnode.componentInstance也是在3.2节分析data.hook.init方法时赋值的，它其实就是组件的实例，这执行了一个initComponent方法，下面来看看这个方法做了什么事情：

  function initComponent(vnode, insertedVnodeQueue) {
    if (isDef(vnode.data.pendingInsert)) {
      insertedVnodeQueue.push.apply(insertedVnodeQueue, vnode.data.pendingInsert)
      vnode.data.pendingInsert = null
    }
    vnode.elm = vnode.componentInstance.$el
    if (isPatchable(vnode)) {
      invokeCreateHooks(vnode, insertedVnodeQueue)
      setScope(vnode)
    } else {
      // empty component root.
      // skip all element-related modules except for ref (#3455)
      registerRef(vnode)
      // make sure to invoke the insert hook
      insertedVnodeQueue.push(vnode)
    }
  }

  这里我们又见到了vnode.data.pendingInsert，如果这个值存在则将他合并到数组insertedVnodeQueue中，然后将vnode.data.pendingInsert置为null，然后将vnode的elm属性赋值为它的挂载点dom。接下来又判断isPatchable(vnode)

  function isPatchable(vnode) {
    while (vnode.componentInstance) {
      vnode = vnode.componentInstance._vnode
    }
    return isDef(vnode.tag)
  }

  这个方法其实是将vnode赋值为渲染vnode，因为这里的vnode是个占位vnode，最后返回的vnode.tag是否为空。如果判断非空的话执行invokeCreateHooks，所以上面提到的invokeCreateHooks方法中对insertedVnodeQueue的修改是在这里执行的。这里的isPatchable是判断vnode能不能patch的方法，如果它为false的话，说明这个组件是个空的组件，也就是没有内容可以渲染，这个时候执行else逻辑，else逻辑其实也是给insertedVnodeQueue添加元素，确保能够调用vnode的insert钩子。执行完initComponent方法后createComponent方法又执行insert等逻辑，最后是返回了true，这意味着if(createComponent()) return执行了，这时候就跳出createElm函数继续执行patch方法了，这样就回到了我们上面提到的invokeInsertHook方法执行了。现在我们解决了一个问题，那就是insertedVnodeQueue修改的问题，那么另一个问题的答案是什么呢？其实答案早就有了，我们回溯一下3.1小节讲解生成组件vnode的方法时执行的方法也叫createComponent，里面执行了一个方法installComponentHooks，它给我们的占位节点注册了一系列钩子函数，比如我们之前分析的init钩子，其实insert也是在这里注册的，我们现在来看insert钩子做了什么事

insert (vnode: MountedComponentVNode) {
  const { context, componentInstance } = vnode
  if (!componentInstance._isMounted) {
    componentInstance._isMounted = true
    callHook(componentInstance, 'mounted')
  }
  if (vnode.data.keepAlive) {
    if (context._isMounted) {
      // vue-router#1212
      // During updates, a kept-alive component's child components may
      // change, so directly walking the tree here may call activated hooks
      // on incorrect children. Instead we push them into a queue which will
      // be processed after the whole patch process ended.
      queueActivatedComponent(componentInstance)
    } else {
      activateChildComponent(componentInstance, true /* direct */)
    }
  }
},

  这里我们简单看一下它其实就是再componentInstance._isMounted为false时候将其设置为true，并执行了callHook(componentInstance, 'mounted')。
  我们这里分析的内容比较多，你可能会觉得比较乱，静下心来不要忘记我们分析这些内容的目的，有目的的去分析，再结合源码阅读与代码调试，你会得到你想要的。那么还记得我们的目的是什么吗？就是分析子组件mounted钩子是什么时候调用的。我们上面是用分析的思路去讲述子组件mounted钩子调用，下面我执行顺序去梳理一遍，首先我们在3.3节调试demo基础上加一层子组件grandsonComp，如下所示：

let grandsonComp = {
  template: '<div>{{ msg }}</div>',
  data() {
    return {
      msg: 'Hello grandson'
    }
  },
  mounted() {
    console.log('grandson mount')
  }
}

let childComp = {
  // template: '<div>{{msg}}</div>',
  render(h) {
    return h('div', [this.msg, h(grandsonComp)])
  },
  created() {
    console.log('child create')
  },
  mounted() {
    console.log('child mount')
  },
  data() {
    return {
      msg: 'Hello Vue'
    }
  }
}
new Vue({
  el: '#app',
  mounted() {
    console.log('app mount')
  },
  render: h => h(childComp)
})

我们先从new Vue开始吧，new Vue时候我们执行根组件的初始化，最后执行挂载方法$mounted，之后就是我们熟悉的过程先去生成vnode，这里就是我们的子组件占位vnode，之后我们执行patch过程，首先根组件的挂载是有oldVnode的，因为它需要一个挂载点，所以执行patch时执行了如下else的逻辑

if (isUndef(oldVnode)) {
  // empty mount (likely as component), create new root element
  isInitialPatch = true
  createElm(vnode, insertedVnodeQueue)
} else {...}

  这时候isInitialPatch还是false，insertedVnodeQueue还是空数组，而else中也有一个createElm方法，执行这个方法时候我们目前要插入的vnode是子组件childComp的占位vnode，所以这个时候我们会执行createElm中的createComponent方法，createComponent方法开始又去执行vnode中的data.hook.init方法，这时候子组件开始创建实例，创建过程中执行this._init，也就是Vue初始化方法，只不过这里是创建Vue的子构造器Sub的实例，然后又执行子组件实例的$mount方法，去挂载子组件。
  这样又开始一个新的挂载过程，这个过程总是先挂载子，后挂载父，子组件挂载过程中又要生成子组件的vnode，也就是渲染vnode，这时发现了一个孙子组件grandsonComp于是又生成了grandsonComp占位vnode，然后子组件执行patch， 这里需要特别注意，子组件patch时又创建 新的isInitialPatch和insertedVnodeQueue变量，这时候的patch是没有oldVnode的，这意味着isInitialPatch被修改为true。 然后调用createElm先将文本'Hello Vue'插入，然后又发现grandsonComp组件vnode，所以只能继续执行createComponent(grandsonComp)去创建grandsonComp组件实例并去挂载它，grandsonComp挂载时会再次执行patch方法，实例创建完成后并挂载完成grandsonComp的内容'Hello grandson'，这个过程就是挂载普通节点的过程，所以并不涉及insertedVnodeQueue修改，所以我们一笔带过。这个时候就回到了子组件的patch过程中的createComponent方法，这时刚执行完了i(vnode, false)，我们继续往下执行：

function createComponent(vnode, insertedVnodeQueue, parentElm, refElm) {
  let i = vnode.data
  if (isDef(i)) {
    const isReactivated = isDef(vnode.componentInstance) && i.keepAlive  
    if (isDef(i = i.hook) && isDef(i = i.init)) {
      i(vnode, false /* hydrating */) // 调用init方法
    }
    if (isDef(vnode.componentInstance)) {
      initComponent(vnode, insertedVnodeQueue)
      insert(parentElm, vnode.elm, refElm)
      if (isTrue(isReactivated)) {
        reactivateComponent(vnode, insertedVnodeQueue, parentElm, refElm)
      }
      return true
    }
  }
}

  然后代码继续往下执行，这时vnode.componentInstance是存在的，因为孙组件挂载过程是在i.init执行时，所以接下来执行到initComponent()

function initComponent(vnode, insertedVnodeQueue) {
  if (isDef(vnode.data.pendingInsert)) {
    insertedVnodeQueue.push.apply(insertedVnodeQueue, vnode.data.pendingInsert)
    vnode.data.pendingInsert = null
  }
  vnode.elm = vnode.componentInstance.$el
  if (isPatchable(vnode)) {
    invokeCreateHooks(vnode, insertedVnodeQueue)
    setScope(vnode)
  } else { ... }
}

  然后这里vnode.data.pendingInsert是孙组件的insertedVnodeQueue，因为在patch孙组件最后vnode.data.pendingInsert会赋值为insertedVnodeQueue，但是由于它不会执行initComponent，也就不会执行invokeCreateHooks。这样vnode.data.pendingInsert就是个空数组了。所以下面的push逻辑其实是连接了一个空的数组。
  接下来这里isPatchable返回true，所以要执行invokeCreateHooks方法，这个方法通过名字可以看出来是调用create钩子的，但是它兼职处理了insertedVnodeQueue，因为在createElm中处理普通vnode时也调用了这个函数，但那时并没有data.hook，下面我只列出对insertedVnodeQueue处理的代码

i = vnode.data.hook
if (isDef(i)) {
   if (isDef(i.create)) i.create(emptyNode, vnode)
   if (isDef(i.insert)) insertedVnodeQueue.push(vnode)
}

  所以这个时候会往insertedVnodeQueue中push组件vnode，这里是grandsonComp的占位vnode，那么这里push后在哪使用呢？我们继续往下看，等子组件执行完createComponent方法后，直接return跳出了createElm方法，这样又回到了patch 过程中，注意这里的patch是子组件childComp所调用的，上面我们特别指出走的是oldVnode不存在的逻辑，因为它还没有被挂载过所以不存在旧的节点，子组件的挂载并不要求必须去替换一个挂载点，这和根组件是不同的，因为挂载必须要一个参考点，子组件可以使用父组件做参考点，而根组件就只能我们手动指定了。那么言归正传，oldVnode不存在的时候有个重要影响就是我们之前定义的isInitialPatch会被置为true，它的命名含义也是标识是否是初始的渲染，这个范围只是组件范围的首次渲染，然后我们忽略中间的一堆与我们当前分析无关的逻辑，直接到函数最后去找invokeInsertHook，这个逻辑对我们来说十分关键，所以我们这里要再次详细分析它：

function invokeInsertHook(vnode, queue, initial) {
  // delay insert hooks for component root nodes, invoke them after the
  // element is really inserted 延迟为组件根节点插入钩子，在真正插入元素后调用它们
  if (isTrue(initial) && isDef(vnode.parent)) {
    vnode.parent.data.pendingInsert = queue
  } else {
    for (let i = 0; i < queue.length; ++i) {
      queue[i].data.hook.insert(queue[i])
    }
  }
}

  分析前我们先把它当前的三个参数列出来分别是childVnode子组件渲染vnode、[grandsonComp占位vnode]、true，我们已经知道grandsonComp占位vnode是有insert钩子的，queue不为空是不是就能执行else中的循环去调用insert方法呢？我们知道这个时候grandsonComp的内容已经全部insert完成了，这时候是不是也该调用insert来执行其mounted生命周期钩子呢？答案是否定的，因为此时满足的是if条件，根本不会执行else逻辑。我们来看此时initial是true，那么vnode.parent是什么呢？vnode是childVnode，那么它的parent就是它的父级vnode，它父级是根组件，所以这里就是根组件的vnode，根组件的vnode又是个什么东西呢？在这里根组件只有childComp一个内容渲染，所以其实就是childComp的占位vnode。所以这里执行了if逻辑，这样将queue赋值给了childComp的占位vnode.data.pendingInsert。到这里子组件也插入完成了，是不是就该结束了，但是我们的insert方法还没有执行，mounted钩子还没有调用，所以我们代码还远远没有结束。
  执行完上述过程我们子组件的patch结束了，这只意味着createComponent方法执行完了i(vnode, false)，又是上述过程的重复，上面是创建孙子组件，这里是创建子组件，然后又执行到了initComponent方法，这个时候就不一样了，因为之前我们给vnode.parent.data.pendingInsert赋值为[grandsonComp占位vnode]，因为createComponent方法总是父级调用的，然后这里就去合并queue，当然这里初始的queue也是空数组，所以现在queue中只有孙子组件的占位vnode，之后再调用initComponent才又把childComp占位节点push进去。这时queue有两个元素，它遵循的还是先子后父原则，然后执行完createComponent所有逻辑后跳出createElm方法，再次进入根组件的patch中执行剩余逻辑，我们一开始也说了根组件是有oldVnode的，所以在最后执行invokeInsertHook时initial是false，所以这时候直接执行else逻辑，这时候就挨个调用组件mounted钩子，顺序也是我们所说的先子后父，因为push的顺序就是如此。这里做了注释这里延迟调用insert钩子是为了在真正插入元素后才调用。因为在子组件插入完成时并没有真正渲染出来，只有所有内容都插入并且将根组件插入时才真正将内容展示到页面，所以就是在插入根组件后去调用insert方法从而触发各个子组件的mounted钩子的，为什么这里说是子组件——别忘了我们根组件的mounted是在mountComponent函数最后专门去调用的。
  最后，我们来总结一些insertedVnodeQueue起了什么作用，实际上它是用来保存patch过程中当前组件下所有后代组件占位vnode，注意这里是所有后代，目的是为了保证根节点在插入页面再去调用各个组件的hook钩子。
