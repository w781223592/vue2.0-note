# 3.5生命周期（下）

  我们在mountComponent方法中执行new Watch是传入的第四个参数是一个对象，对象中有个before方法

new Watcher(vm, updateComponent, noop, {
  before() {
    if (vm._isMounted && !vm._isDestroyed) {
      callHook(vm, 'beforeUpdate')
    }
  }
}, true /* isRenderWatcher */)

  可以看到这调用了beforeUpdate生命周期钩子，那下面我们就来看before方法是什么时候被调用的，它是在observer/scheduler.js中的flushSchedulerQueue方法中北调用的。

function flushSchedulerQueue () {
  currentFlushTimestamp = getNow()
  flushing = true
  let watcher, id

  // Sort queue before flush.
  // This ensures that:
  // 1. Components are updated from parent to child. (because parent is always
  //    created before the child)
  // 2. A component's user watchers are run before its render watcher (because
  //    user watchers are created before the render watcher)
  // 3. If a component is destroyed during a parent component's watcher run,
  //    its watchers can be skipped.
  queue.sort((a, b) => a.id - b.id)
  // do not cache length because more watchers might be pushed
  // as we run existing watchers
  for (index = 0; index < queue.length; index++) {
    watcher = queue[index]
    if (watcher.before) {
      watcher.before()
    }
    id = watcher.id
    has[id] = null
    watcher.run()
    // in dev build, check and stop circular updates.
    if (process.env.NODE_ENV !== 'production' && has[id] != null) {
      circular[id] = (circular[id] || 0) + 1
      if (circular[id] > MAX_UPDATE_COUNT) {
        warn(
          'You may have an infinite update loop ' + (
            watcher.user
              ? `in watcher with expression "${watcher.expression}"`
              : `in a component render function.`
          ),
          watcher.vm
        )
        break
      }
    }
  }
  // keep copies of post queues before resetting state
  const activatedQueue = activatedChildren.slice()
  const updatedQueue = queue.slice()
  resetSchedulerState()
  // call component updated and activated hooks
  callActivatedHooks(activatedQueue)
  callUpdatedHooks(updatedQueue)
  // devtool hook
  /* istanbul ignore if */
  if (devtools && config.devtools) {
    devtools.emit('flush')
  }
}

  这个内容与响应式有关，我们之后会有专门的章节去分析，现在我只需要简单知道在每个nextick时会执行这个方法，它先对queue做了一个排序，然后遍历queue，queue中放的是一些watch集合，遍历时首先判断它们有没before方法，有的话执行before方法，这时候就调用了我们的beforeUpdate钩子，updated钩子函数则是在callUpdatedHooks中调用的，它是将queue创建了一个副本后作为参数传入的，我们来看看这个函数的定义

function callUpdatedHooks (queue) {
  let i = queue.length
  while (i--) {
    const watcher = queue[i]
    const vm = watcher.vm
    if (vm._watcher === watcher && vm._isMounted && !vm._isDestroyed) {
      callHook(vm, 'updated')
    }
  }
}

这个方法就是遍历queue并获取各个watcher的实例，vm_watcher和watcher相同则表示这是个渲染watch，并且如果它已经mounted，而且没有被销毁时就执行updated钩子。 为什么vm._watcher === watcher时，就说明它是渲染watcher？因为在创建watch实例的时候有这样一段逻辑

if (isRenderWatcher) {
   vm._watcher = this
}

  如果是一个渲染watcher的话就使vm._watcher = this，所以这里我们这样去判断，再往下看

if (options) {
  this.deep = !!options.deep // Boolean
  this.user = !!options.user
  this.lazy = !!options.lazy
  this.sync = !!options.sync
  this.before = options.before
}

  我们的before方法也是被保留下来的，所以我们才可以遍历去调用它。下面我们再去看beforeDestroy、destroyed钩子的调用。这两个钩子是在组件销毁过程中去调用的，销毁方法定义同样是在lifecycle.js中的。

Vue.prototype.$destroy = function () {
  const vm: Component = this
  if (vm._isBeingDestroyed) {
    return
  }
  callHook(vm, 'beforeDestroy')
  vm._isBeingDestroyed = true
  // remove self from parent
  const parent = vm.$parent
  if (parent && !parent._isBeingDestroyed && !vm.$options.abstract) {
    remove(parent.$children, vm)
  }
  // teardown watchers
  if (vm._watcher) {
    vm._watcher.teardown()
  }
  let i = vm._watchers.length
  while (i--) {
    vm._watchers[i].teardown()
  }
  // remove reference from data ob
  // frozen object may not have observer.
  if (vm._data.__ob__) {
    vm._data.__ob__.vmCount--
  }
  // call the last hook...
  vm._isDestroyed = true
  // invoke destroy hooks on current rendered tree
  vm.__patch__(vm._vnode, null)
  // fire destroyed hook
  callHook(vm, 'destroyed')
  // turn off all instance listeners.
  vm.$off()
  // remove __vue__ reference
  if (vm.$el) {
    vm.$el.__vue__ = null
  }
  // release circular reference (#6759)
  if (vm.$vnode) {
    vm.$vnode.parent = null
  }
}

  组件销毁大致逻辑就是如果已经在销毁状态则直接返回，否则先调用beforeDestroy，然后将vm._isBeingDestroyed设置为true，然后获取父级vm实例，这个$parent是在initLifecycle函数中被赋值的，然后判断父组件实例还存在并且父组件没有在销毁过程中，而且被销毁组件不是抽象的，然后就调用remove移除父子关系，remove是定义在shared/util.js下的辅助函数，我们可以在shared/util文档中看到它的逻辑，这个方法就是将数组中指定元素给删除，在这里就是将父组件中的$children中删除要销毁的子组件。 然后就是删除观察者watcher，关于这部分我们现在还不了解它，所以这里也不做深入分析，再后面章节我们了解响应式原理后就知道它做了什么事。 在往后就真正执行我们的销毁逻辑了，先将_isDestroyed设置为true，再调用vm.__patch__方法将dom移除，这里移除方法同样是调用了__patch__，只是把第二个参数设置为null，我们来回想一下，patch的前两个参数是什么，第一个是旧的vnode，第二个是新的vnode，新的vnode是true说明其实是移除节点，实际上无论是挂载、更新还是删除都是调用我们的patch方法的。patch有打补丁的意思，可以认为它是dom的更新方法，挂载是从无到有的更新，而删除则是从有到无的更新，在销毁过程也需要做些特殊处理，比如其子组件的处理，同mounted类似，beforeDestroy钩子是先从父组件开始执的，而destroyed钩子则是先从子组件开始，也就是说patch方法销毁组件时会先深度遍历去销毁它的子组件，有关这部分内容我以后分析。patch完成后就执行我们callHook去调用destroyed钩子了，之后把一些变量置为空。
  这两个小节我们简单介绍了常用的八个生命周期钩子，了解到了这些生命周期的执行时机，由于我们只分析了Vue的挂载过程，所以我们只重点分析了mounted钩子的执行，等后续我们分析了组件更新销毁过程我们会更深入的了解更新和销毁钩子。另外，我们还要两个钩子没有提到那就是activated和deactivated钩子，这两个钩子是与keep-alive相关的钩子，之后我们会有专门的章节讲解keep-alive，那时候我们再来分析这两个生命周期钩子。
