# 6.1 events（2）

  上个小节我们分析了v-on的编译过程，但是我们还有一部分分支没有，我们顺便把这部分逻辑给覆盖到。先来写一个demo：

new Vue({
  el: '#app',
  data: {
    c: {
      click() { console.log('click') },
      b() { console.log('b') }
    }
  },
  template: `
    <div v-on="c">a</div>
  `
})

  你会发现这样的写法像极了v-bind指令批量绑定，事实上也确实如此，它们在解析过程也是一样的逻辑，在代码生成过程中的逻辑也是类似的，也就是执行genData时候：

  const dirs = genDirectives(el, state)
  if (dirs) data += dirs + ','

  会有一个v-on指令，然后就去genDirectives中执行state.directives.on方法，on方法定义在compiler/directives/on.js

export default function on (el: ASTElement, dir: ASTDirective) {
  if (process.env.NODE_ENV !== 'production' && dir.modifiers) {
    warn(`v-on without argument does not support modifiers.`)
  }
  el.wrapListeners = (code: string) => `_g(${code},${dir.value})`
}

  同bind方法类似，on方法会给el添加一个wrapListeners方法，这个方法会返回一个_g方法包裹的代码串。
  然后再genData最后执行

  // v-on data wrap
  if (el.wrapListeners) {
    data = el.wrapListeners(data)
  }

  执行el.wrapListeners将data字符串传入。

  target._g = bindObjectListeners

_g方法是bindObjectListeners方法

export function bindObjectListeners (data: any, value: any): VNodeData {
  if (value) {
    if (!isPlainObject(value)) {
      process.env.NODE_ENV !== 'production' && warn(
        'v-on without argument expects an Object value',
        this
      )
    } else {
      const on = data.on = data.on ? extend({}, data.on) : {}
      for (const key in value) {
        const existing = on[key]
        const ours = value[key]
        on[key] = existing ? [].concat(existing, ours) : ours
      }
    }
  }
  return data
}

  首先要判断value是一个纯对象，否则会报一个警告。如果value是一个纯对象就会执行else逻辑，首先如果data.on如果存在那就先copy一份保存到on变量中，不存在则初始为{}。
  然后就是将value中的事件都添加到data.on中，最后返回data。
  接下来我们就来分析Vue是怎样把事件绑定到真实dom的。这要从我们的patch过程说起了，这部分内容可以在我们2.5小节看到。
  在createPatchFunction函数开头有这样一段代码

  const cbs = {}

  const { modules, nodeOps } = backend

  for (i = 0; i < hooks.length; ++i) {
  cbs[hooks[i]] = []
  for (j = 0; j < modules.length; ++j) {
    if (isDef(modules[j][hooks[i]])) {
      cbs[hooks[i]].push(modules[j][hooks[i]])
    }
  }

  这会将我们modules中的一些方法函数注册到cbs中，modules中存放着一些平台化的方法，这些方法包括对events、attrs、domProps、以及style和class的逻辑，这些模块有着对应patch的hook钩子，比如说在create阶段需要处理事件，那么事件modules就会暴露一个create函数，下面我们到web/runtime/modules/events中去找事件的处理模块

export default {
  create: updateDOMListeners,
  update: updateDOMListeners,
  destroy: (vnode: VNodeWithData) => updateDOMListeners(vnode, emptyNode)
}

  可以看到这里暴露了三个hook钩子，create、update、destroy。但是它们实际上都是暴露了同一个方法updateDOMListeners。我们先看看这三个钩子调用的时机，在patch.js中有invokeCreateHooks，它会取出cbs.create数组并遍历调用其函数。而invokeCreateHooks函数的调用有两个地方，一个是在createElm中，另一个是在组件初始化的initComponent中。
  我们接下来看看updateDOMListeners的逻辑

function updateDOMListeners (oldVnode: VNodeWithData, vnode: VNodeWithData) {
  if (isUndef(oldVnode.data.on) && isUndef(vnode.data.on)) {
    return
  }
  const on = vnode.data.on || {}
  const oldOn = oldVnode.data.on || {}
  // vnode is empty when removing all listeners,
  // and use old vnode dom element
  target = vnode.elm || oldVnode.elm
  normalizeEvents(on)
  updateListeners(on, oldOn, add, remove, createOnceHandler, vnode.context)
  target = undefined
}

  可以看到这个方法定义比较简单，但是内部逻辑却是一点都不少，这里先判断新旧虚拟dom是否有on属性，如果都没有的话那么是不需要处理事件的，那么直接return就行了。
  接下来获取新旧vnode的on属性保存在on和oldOn变量中，接下来取vnode.elm也就是真实的节点，这里要注意我们在调用这个方法时已经创建了其真实dom。所以这里是可以取到的。接下来会标准化事件

export const RANGE_TOKEN = '__r'
export const CHECKBOX_RADIO_TOKEN = '__c'
// normalize v-model event tokens that can only be determined at runtime.
// it's important to place the event as the first in the array because
// the whole point is ensuring the v-model callback gets called before
// user-attached handlers.
function normalizeEvents (on) {
  /* istanbul ignore if */
  if (isDef(on[RANGE_TOKEN])) {
    // IE input[type=range] only supports `change` event
    const event = isIE ? 'change' : 'input'
    on[event] = [].concat(on[RANGE_TOKEN], on[event] || [])
    delete on[RANGE_TOKEN]
  }
  // This was originally intended to fix #4521 but no longer necessary
  // after 2.5. Keeping it for backwards compat with generated code from < 2.4
  /* istanbul ignore if */
  if (isDef(on[CHECKBOX_RADIO_TOKEN])) {
    on.change = [].concat(on[CHECKBOX_RADIO_TOKEN], on.change || [])
    delete on[CHECKBOX_RADIO_TOKEN]
  }
}

  主要是处理一些兼容性问题。
  然后接下来是执行关键函数updateListeners，最后将target重置，这里target是全局变量，被多处使用。
updateListeners定义在core/vdom/helpers/update-listeners下

export function updateListeners (
  on: Object,
  oldOn: Object,
  add: Function,
  remove: Function,
  createOnceHandler: Function,
  vm: Component
) {
  let name, def, cur, old, event
  for (name in on) {
    def = cur = on[name]
    old = oldOn[name]
    event = normalizeEvent(name)
    /* istanbul ignore if */
    if (__WEEX__ && isPlainObject(def)) {
      cur = def.handler
      event.params = def.params
    }
    if (isUndef(cur)) {
      process.env.NODE_ENV !== 'production' && warn(
        `Invalid handler for event "${event.name}": got ` + String(cur),
        vm
      )
    } else if (isUndef(old)) {
      if (isUndef(cur.fns)) {
        cur = on[name] = createFnInvoker(cur, vm)
      }
      if (isTrue(event.once)) {
        cur = on[name] = createOnceHandler(event.name, cur, event.capture)
      }
      add(event.name, cur, event.capture, event.passive, event.params)
    } else if (cur !== old) {
      old.fns = cur
      on[name] = old
    }
  }
  for (name in oldOn) {
    if (isUndef(on[name])) {
      event = normalizeEvent(name)
      remove(event.name, oldOn[name], event.capture)
    }
  }
}

  这里接收6个参数，新的on和旧的oldOn，最后一个参数是vm，中间的几个参数都是传入的一些回调函数。
  首先定义name, def, cur, old, event，然后遍历on，遍历的键用name来保存，然后将def和cur都赋值为on[name]，old则是保存旧事件old[name]。
  接下来要用normalizeEvent方法去标准化事件name，这里就是把我们编译时特殊处理的事件名称恢复，并且取到修饰符标识。方法如下：

const normalizeEvent = cached((name: string): {
  name: string,
  once: boolean,
  capture: boolean,
  passive: boolean,
  handler?: Function,
  params?: Array<any>
} => {
  const passive = name.charAt(0) === '&'
  name = passive ? name.slice(1) : name
  const once = name.charAt(0) === '~' // Prefixed last, checked first
  name = once ? name.slice(1) : name
  const capture = name.charAt(0) === '!'
  name = capture ? name.slice(1) : name
  return {
    name,
    once,
    capture,
    passive
  }
})

  这里还做了缓存优化。
  接下来如果cur不存在，会报一个警告说无效的事件xx。再如果old不存在的话说明需要绑定新事件。这里会判断如果cur.fns不存在，那就将cur和on[name]都赋值为createFnInvoker的调用结果

export function createFnInvoker (fns: Function | Array<Function>, vm: ?Component): Function {
  function invoker () {
    const fns = invoker.fns
    if (Array.isArray(fns)) {
      const cloned = fns.slice()
      for (let i = 0; i < cloned.length; i++) {
        invokeWithErrorHandling(cloned[i], null, arguments, vm, `v-on handler`)
      }
    } else {
      // return handler return value for single handlers
      return invokeWithErrorHandling(fns, null, arguments, vm, `v-on handler`)
    }
  }
  invoker.fns = fns
  return invoker
}

  这个方法会返回一个invoker函数，这里会将传入的cur赋值给invoker.fns，也就是当前事件对象，它可能是一个单函数，也可能是函数构成的数组。
  invoker函数逻辑也很简单，如果是单个函数的，就去调用它并会返回它的调用结果。如果是一个数组那就依次去调用它的方法，当然这里是不会执行的。

注意

这里会把我们定义的函数挂载到invoker.fns上，这么做的意义是什么呢？我们先留一个悬念，后面我们就会分析。

  invokeWithErrorHandling之前我们已经见过了，其实就是去调用传入的方法，之前我们已经分析过了。 回到updateListeners方法中我们再去看接下来的逻辑

  if (isTrue(event.once)) {
    cur = on[name] = createOnceHandler(event.name, cur, event.capture)
  }

  这里会判断是否使用once修饰符，并且将cur和on[name]重新赋值，这里调用了一个传入的函数createOnceHandler，这里注意传入的第二个参数cur其实就是我们上面已经处理过的invoker函数了。

function createOnceHandler (event, handler, capture) {
  const _target = target // save current target element in closure
  return function onceHandler () {
    const res = handler.apply(null, arguments)
    if (res !== null) {
      remove(event, onceHandler, capture, _target)
    }
  }
}

  这个方法也是比较简单，就是返回了一个内层函数，这个内层函数会调用handler，handler就是我们的invoker函数，这里判断如果返回值不为null我们就将这个事件给移除，因为once修饰符就是只监听一次事件。只要它成功执行我们就把它移除。
  这里为什么要判断不等于null呢？我们知道invoker函数调用时如果是只有一个事件函数的话会返回其执行结果，而如果是一个函数数组的执行则不返回。那岂不是只有绑定单个事件时才会移除吗？注意我们这里判断的是!== null。也就是说undefined也是认为执行成功的，那这里为什么判断是null就要移除事件呢？
  在之前我们生成代码时候会生成大量if(xx) return null。到这里是不是一切就清楚了，因为返回null是意味着被一些修饰符所拦截，那么函数并没有正常执行，那么就不能去移除。除非你故意要让你的程序返回null。   接下来就是真正给dom添加事件的方法了，也就是add方法的调用

  add(event.name, cur, event.capture, event.passive, event.params)

add方法内容是比较多的，因为篇幅原因我们把它放到下个小节去分析。
