# 6.1 events（3）

  这个小节我们来分析add方法，这个方法本身并不复杂，但是由于会处理很多问题，所以就显得非常复杂了

function add (
  name: string,
  handler: Function,
  capture: boolean,
  passive: boolean
) {
  // 省略部分
  target.addEventListener(
    name,
    handler,
    supportsPassive
      ? { capture, passive }
      : capture
  )
}

  我们先来看add方法的关键性逻辑，其实就是调用el.addEventListener添加一个事件监听，并不是所有浏览器addEventListener都支持第三个options选项，这么定义可以判断是是否支持options选项。 接下来我们看这段省略的代码

  // async edge case #6566: inner click event triggers patch, event handler
  // attached to outer element during patch, and triggered again. This
  // happens because browsers fire microtask ticks between event propagation.
  // the solution is simple: we save the timestamp when a handler is attached,
  // and the handler would only fire if the event passed to it was fired
  // AFTER it was attached.
  // 异步边缘案例6566：内部点击事件触发补丁，事件处理程序在patch期间附加到外部元素，
  // 然后再次触发。发生这种情况是因为浏览器在事件传播之间触发微任务节拍。解决方案很简单：
  // 我们在附加处理程序时保存时间戳，只有当传递给它的事件在附加后被激发时，处理程序才会激发
  if (useMicrotaskFix) {
    const attachedTimestamp = currentFlushTimestamp
    const original = handler
    handler = original._wrapper = function (e) {
      if (
        // no bubbling, should always fire.
        // this is just a safety net in case event.timeStamp is unreliable in
        // certain weird environments...
        // 无冒泡，应该总是触发。这只是一种安全措施，
        // 以防某些奇怪的环境中的事件时间戳不可靠的情况。
        e.target === e.currentTarget ||
        // event is fired after handler attachment
        // 事件触发在附加程序之后
        e.timeStamp >= attachedTimestamp ||
        // bail for environments that have buggy event.timeStamp implementations
        // 为有bug的event.timeStamp实现提供担保
        // #9462 iOS 9 bug: event.timeStamp is 0 after history.pushState
        // iOS中event.timeStamp 在history.pushState后是0
        // #9681 QtWebEngine event.timeStamp is negative value
        // QtWebEngine内核会取到一个负数值
        e.timeStamp <= 0 ||
        // #9448 bail if event is fired in another document in a multi-page
        // electron/nw.js app, since event.timeStamp will be using a different
        // starting reference
        // 如果在多页面Electron/NW.js应用程序中的另一个文档中触发bailifevent，
        // 则event.timeStamp将使用不同的起始参考。
        e.target.ownerDocument !== document
      ) {
        return original.apply(this, arguments)
      }
    }
  }

这里注释了#6566号issue，大致描述是这个样子的

  <div class="header" v-if="expand"> // block 1
    <i @click="expand = false, countA++">Expand is True</i>//element 1
  </div>
  <div class="expand" v-if="!expand" @click="expand = true, countB++"> // block 2
    <i>Expand is False</i> // element 2
  </div>

有这样一个例子，当我点击element1时，expand会变为false，那么就会更新dom去渲染block2，但是事实上有四种情况
• 点击事件监听block 1，block2效果很好
• 点击事件监听element 1，element 2效果很好
• 点击事件监听block 1，element 2改成expand为true可以。但是不能变回去。
• 点击事件监听element 1和block 2，不能改成expand为false。但可以更改expand为true。
然后尤大给与了回复：
• 触发内部点击事件<i>，触发 nextTick 的第一次更新（微任务）
• 微任务在事件冒泡到外部 div 之前被处理。在更新期间，点击侦听器被添加到外部 div。
• 因为DOM结构是一样的，所以外层div和内层元素都被复用了。
• 事件最终到达外部 div，触发第一次更新添加的监听器，进而触发第二次更新。
这在修复中非常棘手，其他利用微任务进行更新排队的库也有这个问题（例如 Preact）。React 似乎没有这个问题，因为他们使用合成事件系统（可能是由于像这样的边缘情况）。
  这已经是v2.4的问题了。接下来我们看这部分代码是怎么处理的，这里首先是判断useMicrotaskFix，其定义如下

const useMicrotaskFix = isUsingMicroTask && !(isFF && Number(isFF[1]) <= 53)

  isUsingMicroTask是否能适用微任务，这个东西我们在nextTick小节我们已经分析过了，如果浏览器支持微任务的话，那么我们Vue的更新都将采用微任务队列的异步来处理，这将会有极大的性能提升。但是使用微任务就会带来上述冒泡问题，所以接下来就要对此做处理。
  接下来定义attachedTimestamp保存currentFlushTimestamp，这个东西我们也见过它，在分析nextTick的调度函数flushSchedulerQueue执行时会首先将其记录一个时间，这个时间记录是比较复杂的

// Async edge case #6566 requires saving the timestamp when event listeners are
// attached. However, calling performance.now() has a perf overhead especially
// if the page has thousands of event listeners. Instead, we take a timestamp
// every time the scheduler flushes and use that for all event listeners
// attached during that flush.
// 异步边缘情况#6566要求在附加事件侦听器时保存时间戳。然而，调用performance.now（）会
// 产生性能开销，特别是当页面有数千个事件侦听器时。相反，我们在调度程序每次刷新时
// 都会获取一个时间戳，并将其用于刷新期间连接的所有事件侦听器。
export let currentFlushTimestamp = 0

// Async edge case fix requires storing an event listener's attach timestamp.
// 异步边缘案例修复需要存储事件侦听器的附加时间戳。
let getNow: () => number = Date.now

// Determine what event timestamp the browser is using. Annoyingly, the
// timestamp can either be hi-res (relative to page load) or low-res
// (relative to UNIX epoch), so in order to compare time we have to use the
// same timestamp type when saving the flush timestamp.
// All IE versions use low-res event timestamps, and have problematic clock
// implementations (#9632)
// 确定浏览器正在使用的事件时间戳。令人讨厌的是，时间戳可以是高分辨率（相对于页面加载）
// 或低分辨率（相对于UNIX历元），因此为了比较时间，我们必须在保存刷新时间戳时使用相同
// 的时间戳类型。所有IE版本都使用低分辨率事件时间戳，并且有问题的时钟实现（#9632）
if (inBrowser && !isIE) {
  const performance = window.performance
  if (
    performance &&
    typeof performance.now === 'function' &&
    getNow() > document.createEvent('Event').timeStamp
  ) {
    // if the event timestamp, although evaluated AFTER the Date.now(), is
    // smaller than it, it means the event is using a hi-res timestamp,
    // and we need to use the hi-res version for event listener timestamps as
    // well.
    // 如果事件时间戳（虽然在Date.now（）之后计算）小于它，则意味着事件使用的是高分辨率时间戳，
    // 我们还需要使用高分辨率版本的事件侦听器时间戳
    getNow = () => performance.now()
  }
}

  由于不同浏览器会使用不同的时间精度，这就使我们getNow定义比较复杂，这里首先要判断是浏览器环境，并且不是ie浏览器，因为对于ie来说performance.now()返回的时间和Date.now()是相同，这倒是省去了我们不少事。
  接下来会获取window.performance，如果存在的话的话继续下一步判断，这时候会调用getNow方法，也就是Date.now生成时间戳，然后用document.createEvent方法创建一个事件并取事件时间戳，虽然事件时间戳是在Date.now后面调用的，但是Date.now任然比后者大，说明是高分辨率时间。那么就将getNow赋值为performance. now()。
  然后接下来就是在组件发生更新时执行flushSchedulerQueue时首先会调用getNow方法并将时间保存在currentFlushTimestamp，然后我们回到我们add中的逻辑

  if (useMicrotaskFix) {
    const attachedTimestamp = currentFlushTimestamp
    const original = handler
    handler = original._wrapper = function (e) {
      if (
        e.target === e.currentTarget ||
        e.timeStamp >= attachedTimestamp ||
        e.timeStamp <= 0 ||
        e.target.ownerDocument !== document
      ) {
        return original.apply(this, arguments)
      }
    }
  }

  我们把注释都去掉会看的更清晰一点，这里会获取当前flush的时间戳，并且保存在attachedTimestamp。然后用original保存原函数handler，然后给handler重新赋值，这里给原函数original添加一个_wrapper函数，这样的好处是最后调用时能绑定到原函数this。但是不是任何情况下都是会调用原函数的，这里做了几个判断：

  1. e.target === e.currentTarget也就是当前点击元素和事件绑定元素一致。不存在冒泡行为，那么就可以直接调用了，我们要解决的就是微任务在事件冒泡前处理的问题，如果没有冒泡的话那么这个问题也不存在。
  2. e.timeStamp >= attachedTimestamp，事件触发在微任务更新之后，但是如果时间精度不够的话可能也会相等。这时也要执行方法。
  3. e.timeStamp <= 0 || e.target.ownerDocument !== document这些是处理一些特殊情况下timeStamp问题。

  满足这几个情况就认为事件应该被调用。
  我们回过头来看updateListeners剩余逻辑

    if (isUndef(cur)) {
    } else if (isUndef(old)) {
    } else if (cur !== old) {
      old.fns = cur
      on[name] = old
    }

  上面我们讲的是old不存在时候逻辑，也就是添加事件时候逻辑，如果old和cur都存在并且cur !== old要执行的逻辑就非常简单了，这是更新时的逻辑，我们要做的事也很简单，直接将old.fns赋值为cur，on[name]赋值为old。这里的old其实就更新前的invoker函数，invoker就像一个容器，它的执行内容都是invoker.fns中的函数，事件改变其实就fns中的内容改变了，所以直接把old.fns替换就行了，至于添加事件监听的过程也不需要重新执行了，因为事件名都是一致的。
  再往下就跳出了for循环，接下来又是一个for循环

  for (name in oldOn) {
    if (isUndef(on[name])) {
      event = normalizeEvent(name)
      remove(event.name, oldOn[name], event.capture)
    }
  }

  因为更新后dom可能没有事件了，那原本绑定的事件就要移除，所以就找oldOn中有的事件而on中没有的事件把它给移除，调用的是remove方法

function remove (
  name: string,
  handler: Function,
  capture: boolean,
  _target?: HTMLElement
) {
  (_target || target).removeEventListener(
    name,
    handler._wrapper || handler,
    capture
  )
}

  就是调用原生removeEventListener删除事件监听。到这里我们普通标签事件就分析完毕了。不知道你有没有发现，我们这里一直都是对data.on处理，而对于组件的事件来说，直接监听事件需要.native修饰符，并且会被放到data.nativeOn中。
  在3.1小节我们分析了组件vnode的创建过程，在createComponent中有这样一段逻辑：

  // extract listeners, since these needs to be treated as
  // child component listeners instead of DOM listeners
  // 提取侦听器，因为这些侦听器需要被视为子组件侦听器，而不是DOM侦听器
  const listeners = data.on
  // replace with listeners with .native modifier
  // so it gets processed during parent component patch.
  // 替换为带有“native”修饰符的侦听器，以便在父组件patch期间对其进行处理
  data.on = data.nativeOn

  也就是把data.on保存到listeners中，data.on重新赋值为data.nativeOn，那之后在patch时的处理data.on其实就是data.nativeOn了，这样就没有任何问题了。而原本的data.on其实应该是监听组件内部emit的自定义事件。
  值得一提的是对于组件data.on后续处理和我们前面流程是没什么不同的，但组件vnode.elm其实就是组件根节点，所以组件的nativeOn事件都会绑定到其根节点dom上。
