# 8.9 router-view组件

  到现在为止我们已经了解了vue-router的初始化过程、路由匹配过程、导航流程、URL切换等内容，但是这些过程都是隐式的，用户感受不到或者感受不明显的过程。所以显然我们还有很重要的一部分逻辑没有去分析，那就是路由视图的问题。通俗的讲就是：组件是怎样渲染的，什么时候渲染的？
  要解决这个问题，我们首先要了解一下vueRouter的内置组件，也就是router-view，这是路由视图的组件。如果我们没有在代码中去写router-view组件，那么我们整个代码都不会生效，可以说vueRouter渲染组件完全依赖于router-view组件。
  下面我们就来看看router-view的定义：

export default {
  name: 'RouterView',
  functional: true,
  props: {
    name: {
      type: String,
      default: 'default'
    }
  },
  render (_, { props, children, parent, data }) {
  }
}

  首先它的代码是比较多的，毕竟是比较复杂的。但我们不要畏惧它，整体来看它其实就是个简单的组件，而且比平常组件还要简单，因为它是一个函数组件。函数组件会在options中配置functional：true来标识。函数组件是一个无状态组件，类似于一个纯函数，也就是说它没有自己的数据data，它的渲染结果只取决于外部传入的内容，如果外部传入相同的数据，那么它的渲染结果也是相同的。可以看到这里只有props和render等选项而没有data之类的选项。因为这个关系在vue2中函数组件性能远大于有状态组件。
  可以看到这里props传入一个name参数，默认为default，这个就对应我们配置路由component时内部自动格式化为components：{ default：component }，接下来重点看render函数

  // used by devtools to display a router-view badge
  // devtools用于显示路由器视图标记 
  data.routerView = true

  // directly use parent context's createElement() function
  // so that components rendered by router-view can resolve named slots
  // 直接使用父上下文的createElement()函数以便router-view渲染的组件可以解析命名的slot
  const h = parent.$createElement
  const name = props.name
  const route = parent.$route
  const cache = parent._routerViewCache || (parent._routerViewCache = {})

  首先是初始化一些变量，把data.routerView置位true。data是从第二个参数中解构出来的。render函数通常是只有一个参数即h，但是在函数组件中是有两个参数的，一些实例化的参数会放到第二个参数中去。
  接下来是获取h，但是这个h不是直接从render函数参数中获取的，而是取自父组件的render函数，这里也注释说明直接用父组件的h函数以便于解析命名插槽。接下来获取name和route，cache则是keepalive相关逻辑，用来做缓存的。

// determine current view depth, also check to see if the tree
// has been toggled inactive but kept-alive.
// 确定当前视图深度，同时检查树是否已切换为非活动状态但仍保持活动状态
let depth = 0
let inactive = false
while (parent && parent._routerRoot !== parent) {
   const vnodeData = parent.$vnode ? parent.$vnode.data : {}
   if (vnodeData.routerView) {
      depth++
   }
   if (vnodeData.keepAlive && parent._directInactive && parent._inactive) {
      inactive = true
   }
   parent = parent.$parent
}
data.routerViewDepth = depth

  接下来是去找路由的嵌套关系，定义深度depth为0，然后inactive不活跃状态标记为false。然后while循环，parent存在并且parent._routerRoot不是parent时执行循环。循环体内先获取vnodeData，如果vnodeData.routerView存在则使depth加1。
  这部分内容比较抽象，我们不妨写个小dome来分析它

  {
    path: '/',
    name: 'home',
    component: Parent,
    children: [{
      path: '/child',
      name: 'child',
      component: Child
    }]
  }

  定义路由如上所示，我们在App.vue中只写一个router-view标签就可以了，然后定义一个父路由组件Parent和子路由组件Child，然后任意写入一些内容，但是父组件中要再写一个router-view，这样就能实现嵌套路由。父组件中还要留下一个跳转到子路由的入口。
  这样当我们第一个router-view标签渲染时vnodeData.routerView显然是不存在的，这时depth为0，然后接下来keepalive这部分逻辑显然也是不会执行的。接下来把parent的父组件实例赋值给parent，在我们的简单例子中，parent初始为App.vue的实例，那么parent.$parent就是根组件了。这个时候再去while判断时会发现parent和_rootRouter是相同的，所以会跳出循环，其实这也符合预期的。因为第一个route-view标签其层级必然是一级，当然这里是用0来标识第一级，也是合乎逻辑的。
  然后接下来把data.routerViewDepth设置为depth，再往下的if判断执行不到

  // render previous view if the tree is inactive and kept-alive
  // 如果树处于非活动状态并保存了活动状态，则渲染上一视图
  if (inactive) {
    const cachedData = cache[name]
    const cachedComponent = cachedData && cachedData.component
    if (cachedComponent) {
      // #2301
      // pass props
      if (cachedData.configProps) {
        fillPropsinData(cachedComponent, data, cachedData.route, cachedData.configProps)
      }
      return h(cachedComponent, data, children)
    } else {
      // render previous empty view
      return h()
    }
  }

  简单来看就是如果是非活跃的并且被keepalive缓存那么就渲染之前的内容。然后就是处理参数之类的。接下来depth就派上用场了

  const matched = route.matched[depth]
  const component = matched && matched.components[name]

  // render empty node if no matched route or no config component
  // 如果没有匹配的路由或配置组件，则呈现空节点
  if (!matched || !component) {
    cache[name] = null
    return h()
  }
  // cache component
  cache[name] = { component: component };

  通过depth来从route.matched中获取record，route.matched数组就是根据路由层级顺序来存放record的，所以通过depth正好可以取到当前层级的record。当然这里record是被命名为matched，表示匹配到了。然后就是获取对应的组件，接下来如果没有匹配成功则呈现空节点，匹配到的话就把它缓存下来。

  // attach instance registration hook
  // this will be called in the instance's injected lifecycle hooks
  data.registerRouteInstance = (vm, val) => {
    // val could be undefined for unregistration
    const current = matched.instances[name]
    if (
      (val && current !== vm) ||
      (!val && current === vm)
    ) {
      matched.instances[name] = val
    }
  }

  接下来定义一个注册实例的方法data.registerRouteInstance，这个方法主要是从matched.instance中拿到对应的name的实例，这里就是default。如果val存在并且current不等于vm就把matched.instances[name]赋值为val，如果val不存在那就注销逻辑了，并且要保证current就是vm时就执行注销。
  这里注释说明这个方法是在被注入的生命周期中调用的，那么在哪里注入生命周期了呢？还记得我们install时候通过Vue.mixin全局混入了beforeCreate钩子，这里面执行了一个registerInstance方法

  Vue.mixin({
    beforeCreate () {
      if (isDef(this.$options.router)) {
        this._routerRoot = this
        this._router = this.$options.router
        this._router.init(this)
        Vue.util.defineReactive(this, '_route', this._router.history.current)
      } else {
        this._routerRoot = (this.$parent && this.$parent._routerRoot) || this
      }
      registerInstance(this, this)
    },
    destroyed () {
      registerInstance(this)
    }
  })

  我们来看看registerInstance方法做什么事

  const registerInstance = (vm, callVal) => {
    let i = vm.$options._parentVnode
    if (isDef(i) && isDef(i = i.data) && isDef(i = i.registerRouteInstance)) {
      i(vm, callVal)
    }
  }

  先获取vm.$options._parentVnode，这个东西我们应该还不陌生，_parentVnode其实就是组件vnode，而vnode其实是组件要渲染的内容。这个之前我们都已经分析过了就不再赘述了。接下来逻辑也很多简单就是找到data.registerRouteInstance并调用它。这里是注册逻辑传入了第二个val参数，我们可以看到在destroy钩子中执行时就没有传入第二个参数，所以是执行注销逻辑。

  // also register instance in prepatch hook
  // in case the same component instance is reused across different routes
  ;(data.hook || (data.hook = {})).prepatch = (_, vnode) => {
    matched.instances[name] = vnode.componentInstance
  }

  然后初始化了prepatch方法，prepatch是预包装的意思，其实是为了组件复用做准备的，可以看到这里就将组件实例添加到instances对象中了，instances我们早就见过了，但是一直不知道它里面什么时候被添加内容。

// register instance in init hook
// in case kept-alive component be actived when routes changed
data.hook.init = (vnode) => {
  if (vnode.data.keepAlive &&
    vnode.componentInstance &&
    vnode.componentInstance !== matched.instances[name]
  ) {
    matched.instances[name] = vnode.componentInstance
  }
// if the route transition has already been confirmed then we weren't // able to call the cbs during confirmation as the component was not 
// registered yet, so we call it here.
  handleRouteEntered(route)
}

  接下来定义init钩子，因为如果组件被keepAlive保存，但是之后又被修改了，那么之前缓存的内容就过期了，所以这种情况下保证init中可以更新instances中内容。下面这个handleRouteEntered方法其实就是执行beforRouterEnter中传给next的回调函数，也就是我们导航流程最后一步，所以就很莫名其妙，它这里解释是说如果导航被确认时在确认过程中不能调用cbs，因为不能获取到vm，所以在这里调用了它。这句话说明没有一点毛病，但是我们之前导航被确认后再经过一个nextTick后就可以获取到组件实例了。而且它也是这么做的，我们已经分析过了

  if (this.router.app) {
    this.router.app.$nextTick(() => {
      handleRouteEntered(route)
    })
  }

  但是好像仔细一想nextTick后调用才是莫名其妙，因为dom更新是在nextTick之前的，dom更新前init肯定会执行，所以显然init会在nextTick之前更新，而在执行过后回调函数就会被删除，所以我想不到为什么会在这里专门定义两次。

const configProps = matched.props && matched.props[name]
// save route and configProps in cache
if (configProps) {
  extend(cache[name], {
    route,
    configProps
  })
  fillPropsinData(component, data, route, configProps)
}
return h(component, data, children)

  接下来就是获取参数并缓存，最后使用h函数生成vnode并返回。这里处理参数用到一个fillPropsinData方法

function fillPropsinData (component, data, route, configProps) {
  // resolve props
  let propsToPass = data.props = resolveProps(route, configProps)
  if (propsToPass) {
    // clone to prevent mutation
    propsToPass = data.props = extend({}, propsToPass)
    // pass non-declared props as attrs
    const attrs = data.attrs = data.attrs || {}
    for (const key in propsToPass) {
      if (!component.props || !(key in component.props)) {
        attrs[key] = propsToPass[key]
        delete propsToPass[key]
      }
    }
  }
}

  就是处理props，简单了解一下就可以了，也没有多大用处，resolveProps方法如下：

function resolveProps (route, config) {
  switch (typeof config) {
    case 'undefined':
      return
    case 'object':
      return config
    case 'function':
      return config(route)
    case 'boolean':
      return config ? route.params : undefined
    default:
      if (process.env.NODE_ENV !== 'production') {
        warn(
          false,
          `props in "${route.path}" is a ${typeof config}, ` +
          `expecting an object, function or boolean.`
        )
      }
  }
}

  其实就对应组件传参的三种模式，我们就不细说了。
  分析完router-view组件后我们再回过头来考虑导航时DOM什么时候更新的。我们先回到router.init时候，在transitionTo方法执行完成后执行了history.listen方法

  history.listen(route => {
    this.apps.forEach(app => {
      app._route = route
    })
  })

然后看listen方法

  listen (cb: Function) {
    this.cb = cb
  }

  listen接收一个函数参数cb，然后将cb函数则是接收route参数，遍历apps并将其每一项的app._route都设置为route。listen方法则是将history.cb设置为cb。那么cb是什么时候调用的呢？
  在history.updateRoute方法中

  updateRoute (route: Route) {
    this.current = route
    this.cb && this.cb(route)
  }

  会将this.current设置为route，并调用cb并将route作为参数传入。   updateRoute方法之前已经说过，它是在confirmTransition方法成功回调中调用的

this.confirmTransition(
  route,
  () => {
    this.updateRoute(route)
  }, err => {}
)

  那app._route设置为route有什么用呢？这里app就是根Vue实例，我们回到路由注册时候，在install时通过Vue.mixin混入了beforeCreate钩子中

beforeCreate () {
  if (isDef(this.$options.router)) {
    this._routerRoot = this
    this._router = this.$options.router
    this._router.init(this)
    Vue.util.defineReactive(this, '_route',this._router.history.current)
  } else {}
}

  执行了Vue.util.defineReactive方法把this._route对象变为响应式的，这样在不论在哪种方式导航时都会执行transitionTo方法，然后执行confirmTransition成功后执行回调，回调函数又会调用updateRoute从而执行app._route = route触发set拦截从而触发更新。
  所以这就是我们导航流程第十一步DOM更新，这样是不是就分析了完了呢？我们看似完整的闭环背后其实有个很大的漏洞，Vue.util.definReactive处理过的真的是响应式的吗？Vue的响应式并不仅仅是通过defineProperty劫持就够了，真正的响应式机制还是依赖于Vue的依赖收集机制，Vue收集依赖是在get中做的。如果我们不去触发一次get拦截，那么即使set执行时也不能触发更新，因为所要更新的组件没有收集到。
  这里VueRouter设计是比较巧妙的，这种巧妙在不经意间融入到router-view的设计中，所以我们很容易就忽略了。router-view中有这样一段代码：

  const route = parent.$route

  任谁也想不到这里隐藏一个get操作，我们都认为它是获取route以方便后续多次使用，我们再来看看$route是什么：

  Object.defineProperty(Vue.prototype, '$router', {
    get () { return this._routerRoot._router }
  })

  Object.defineProperty(Vue.prototype, '$route', {
    get () { return this._routerRoot._route }
  })

  在install时会执行这两个逻辑，访问$route实际上是访问_routerRoot._route，也就是根Vue的_route，这就和我们响应式相呼应了，这时就会触发一次get拦截从而收集到渲染的组件，之后就修改时就可以触发更新了。
  这样我们就把路由渲染和更新的原理分析完毕了，下个小节我们来分析另一个组件router-link。
