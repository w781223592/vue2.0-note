# 8.8 路径切换

  前面我们主要是围绕初始化时transitionTo方法来分析的，主要分析了matcher匹配和导航守卫的内容。但是在分析和demo测试过程中会发现URL在这个过程也会发生改变，接下来我们就来看看URL是怎样发生改变的吧。
  我们接下来要分析URL是怎么被改变的，所以我们需要找到什么时候才会发生路径切换，从我们编程式导航角度来说，我们调用router.push时会发生路由的跳转，同时URL也会发生改变。下面来看router.push

  push (location: RawLocation, onComplete?: Function, onAbort?: Function) {
    // $flow-disable-line
    if (!onComplete && !onAbort && typeof Promise !== 'undefined') {
      return new Promise((resolve, reject) => {
        this.history.push(location, resolve, reject)
      })
    } else {
      this.history.push(location, onComplete, onAbort)
    }
  }

  push方法的定义是很简单的，如果成功和失败回调都没有被定义，并且Promise可以使用的话就返回一个Promise对象，并执行history.push，并将resolve和reject作为push的成功和错误回调传入。如果用户定义了成功和失败函数就执行调用history的push方法。history.push如下：

  push (location: RawLocation, onComplete?: Function, onAbort?: Function) {
    const { current: fromRoute } = this
    this.transitionTo(
      location,
      route => {
        pushHash(route.fullPath)
        handleScroll(this.router, route, fromRoute, false)
        onComplete && onComplete(route)
      },
      onAbort
    )
  }

  这个是hash路由的push方法定义，这里结构获取current对象并重命名为fromRoute，接下来同样是执行了transitionTo方法，但是这里成功回调有所不同，这里执行了pushHash方法，这个方法我们来看看做了什么

function pushHash (path) {
  if (supportsPushState) {
    pushState(getUrl(path))
  } else {
    window.location.hash = path
  }
}

  如果支持原生pushState则执行pushState方法，否则直接修改location.hash为path，getUrl方法如下：

function getUrl (path) {
  const href = window.location.href
  const i = href.indexOf('#')
  const base = i >= 0 ? href.slice(0, i) : href
  return `${base}#${path}`
}

  定义十分简单就是获取location.href完整路径经过简单处理后返回，所以这里url是完整的一个路径，接下来看pushState方法

export function pushState (url?: string, replace?: boolean) {
  saveScrollPosition()
  // try...catch the pushState call to get around Safari
  // DOM Exception 18 where it limits to 100 pushState calls
  const history = window.history
  try {
    if (replace) {
      // preserve existing history state as it could be overriden by the user
      const stateCopy = extend({}, history.state)
      stateCopy.key = getStateKey()
      history.replaceState(stateCopy, '', url)
    } else {
      history.pushState({ key: setStateKey(genStateKey()) }, '', url)
    }
  } catch (e) {
    window.location[replace ? 'replace' : 'assign'](url)
  }
}

  首先执行saveScrollPosition执行方法保存滚动位置，然后获取window.history，如果replace为true，接下来就是copy history.state并保存在stateCopy，然后给stateCopy.key赋值为getStateKey，这个东西其实是个由window.performance.now返回的数字转化的字符串。然后调用history.replaceState()方法。如果replace为false则是调用history.pushState去添加一条历史记录，并修改URL。这里history是原生的history对象而不是我们router中定义的History类实例化对象。
  所以这里主要是针对push方法和replace方法跳转的逻辑，如果执行过程有错误的话则会执行catch方法，这里也说了在Safari浏览器种限制100条pushState记录，所以执行是可能报错的，报错时就执行window.location的replace或assign方法去跳转。
  getStateKey()是用来获取当前时间_key，genStateKey则是重新调用window.performance.now方法重新计算key，setStateKey则是将_key进行修改赋值并返回。这部分逻辑如下：

const Time =
  inBrowser && window.performance && window.performance.now
    ? window.performance
    : Date

export function genStateKey (): string {
  return Time.now().toFixed(3)
}
let _key: string = genStateKey()
export function getStateKey () {
  return _key
}
export function setStateKey (key: string) {
  return (_key = key)
}

  我们再来看看saveScrollPosition方法

function saveScrollPosition () {
  const key = getStateKey()
  if (key) {
    positionStore[key] = {
      x: window.pageXOffset,
      y: window.pageYOffset
    }
  }
}

  根据key值将一个含有x、y的对象保存到positionStore对象中。其实就是去保pageXOffset和pageYOffset。
  执行完pushHash方法后我们的URL就被改变了，然后就是执行handleScroll方法，这个主要是滚动行为的处理，我们把它放在vue-router文档(第四部分)去分析。最后就是执行用户定义的成功回调。
  接下来再分析一下router.replace，显然它和push基本上一致，不同处是在transitionTo回调中执行的是replaceHash方法

function replaceHash (path) {
  if (supportsPushState) {
    replaceState(getUrl(path))
  } else {
    window.location.replace(getUrl(path))
  }
}

  同样的这里支持pushState时改为调用replaceState，不支持时使用location的replace方法跳转。下面看replaceState

export function replaceState (url?: string) {
  pushState(url, true)
}

  这就更简单了，直接是pushState方法第二个参数replace传入true，这就和我们之前分析对应上了。
  下面来看几个其他跳转方法

  go (n: number) {
    this.history.go(n)
  }
  back () {
    this.go(-1)
  }
  forward () {
    this.go(1)
  }

  这个主要是调用history.go方法前进或者后退。history.go方法定义也很简单

  go (n: number) {
    window.history.go(n)
  }

  通过window.history.go方法实现。
  分析完编程式导航后，我们还可能通过router-link方式导航、或者通过直接修改url来导航，还可能通过点击浏览器的前进和回退按钮导航。router-link是内置组件，我们抛开它不谈，那么另外两种方式是怎么实现导航的呢？其实我们也不难猜想，这两种方式是浏览器带给我们的导航方式，也就是并不是我们代码去实现的，想知道什么时候需要导航那么肯定是要进行监听的。那我们就看看怎么监听URL变化吧。
  在router.init初始化过程中执行了transitionTo方法，不顾它执行成功与失败都会执行一个setupListeners回调，从名称就可以知道是启动监听器的意思。

  const setupListeners = routeOrError => {
    history.setupListeners()
    handleInitialScroll(routeOrError)
  }
  history.transitionTo(
    history.getCurrentLocation(),
    setupListeners,
    setupListeners
  )

  再来看history.setupListeners

  setupListeners () {
    if (this.listeners.length > 0) {
      return
    }
    const router = this.router
    const expectScroll = router.options.scrollBehavior
    const supportsScroll = supportsPushState && expectScroll
    if (supportsScroll) {
      this.listeners.push(setupScroll())
    }
    const eventType = supportsPushState ? 'popstate' : 'hashchange'
    window.addEventListener(
      eventType,
      handleRoutingEvent
    )
    this.listeners.push(() => {
      window.removeEventListener(eventType, handleRoutingEvent)
    })
  }

  首先判断history.listeners.length > 0就什么都不做直接返回。然后获取router对象，再之后是设置滚动相关逻辑。接着再定义eventType即事件类型，如果支持pushState的话就使用popstate，否则就等于hashchange。然后就根据事件类型进行监听，监听回调则是handleRoutingEvent方法。最后则是往listeners中push一个移除监听的方法。
  接下来我们看handleRoutingEvent方法

  const handleRoutingEvent = () => {
    const current = this.current
    if (!ensureSlash()) {
      return
    }
    this.transitionTo(getHash(), route => {
      if (supportsScroll) {
        handleScroll(this.router, route, current, true)
      }
      if (!supportsPushState) {
        replaceHash(route.fullPath)
      }
    })
  }

  这里先获取current对象，然后执行ensureSlash并判断返回为false就return。如果为true就继续往下执行transitionTo方法，只是这里的回调方法发生了变化，同样会处理滚动，如果不支持pushState时还要执行replaceHash。
ensureSlash方法如下

function ensureSlash () {
  var path = getHash();
  if (path.charAt(0) === '/') {
    return true
  }
  replaceHash('/' + path);
  return false
}

  确保有斜杠，通过getHash方法获取path，getHash之前已经分析过了就是返回#号后面的内容，这里就是确保#号后面的是/，如果不是的话就拼一个斜杠执行replaceHash。 值得一提的是：在HashHistory初始化时候也执行了这个方法

  constructor (router: Router, base: ?string, fallback: boolean) {
    super(router, base)
    // check history fallback deeplinking
    if (fallback && checkFallback(this.base)) {
      return
    }
    ensureSlash()
  }

  我们打开项目时http://localhost:8080这样的路径然后会自动在后面拼接/#/，就是因为这里执行了ensureSlash，这时候getHash会返回空字符串，所以就会拼接一个/并且执行replaceHash

function replaceHash (path) {
  if (supportsPushState) {
    replaceState(getUrl(path))
  } else {
    window.location.replace(getUrl(path))
  }
}

  而replaceHash执行replaceState时又执行getUrl('/')``，getUrl获取到href为http://localhost:8080/

function getUrl (path) {
  const href = window.location.href
  const i = href.indexOf('#')
  const base = i >= 0 ? href.slice(0, i) : href
  return `${base}#${path}`
}

  最后会将href + # + path返回，所以我们最终访问的是： http://localhost:8080/#/，并且没有留下http://localhost:8080访问的历史，因为我们通过replaceState方法进行了一次跳转并不会留下记录。
  上个小节最后我们留了两个坑，现在我们来解决一下history.ensureURL的问题。

  ensureURL (push?: boolean) {
    const current = this.current.fullPath
    if (getHash() !== current) {
      push ? pushHash(current) : replaceHash(current)
    }
  }

  这个方法通过比较fullPath与getHash，如果不是相同的，那么url应该被更新，正常情况它们应该是相同的。然后根据传入的参数选择用pushHash还是replaceHash方法。
  ensureURL在很多地方都有使用，比如在confirmTransition方法回调中就有涉及，对于我们重定向而言，比如下面例子

{
    path: '/',
    name: 'home',
    redirect: '/home',
    // component: () => import('../views/Home.vue')
}, {
    path: '/home',
    component: () => import('../views/Home.vue')
},

  我们将根路径path：/重定向到/home，它在初始化时执行transitionTo会执行到match去匹配路径，最终匹配到的是path: /home，这样在执行confirmTransition回调时执行到ensureURL方法，这时候getHash为/，但是匹配到fullPath是/home，所以它将会把URL变为/home，并且push参数没有传，所以不会有/的历史栈。
