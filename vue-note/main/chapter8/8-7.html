# 8.5 导航守卫（下）

runQueue方法回调通过next可以等待异步组件执行完成后再执行，我们先看它的代码逻辑

  runQueue(queue, iterator, () => {
    // wait until async components are resolved before
    // extracting in-component enter guards
    const enterGuards = extractEnterGuards(activated)
    const queue = enterGuards.concat(this.router.resolveHooks)
    runQueue(queue, iterator, () => {
      if (this.pending !== route) {
        return abort(createNavigationCancelledError(current, route))
      }
      this.pending = null
      onComplete(route)
      if (this.router.app) {
        this.router.app.$nextTick(() => {
          handleRouteEntered(route)
        })
      }
    })
  })

  可以看到这里是我们导航第七个流程：在被激活的组件里调用beforeRouteEnter守卫。也就是这里执行extractEnterGuards方法，这个方法比其他方法复杂一些

function extractEnterGuards (
  activated: Array<RouteRecord>
): Array<?Function> {
  return extractGuards(
    activated,
    'beforeRouteEnter',
    (guard, _, match, key) => {
      return bindEnterGuard(guard, match, key)
    }
  )
}

这里看起来复杂一些，其实是自定义了bind函数，我们来看bindEnterGuard方法

function bindEnterGuard (
  guard: NavigationGuard,
  match: RouteRecord,
  key: string
): NavigationGuard {
  return function routeEnterGuard (to, from, next) {
    return guard(to, from, cb => {
      if (typeof cb === 'function') {
        if (!match.enteredCbs[key]) {
          match.enteredCbs[key] = []
        }
        match.enteredCbs[key].push(cb)
      }
      next(cb)
    })
  }
}

  这里也是返回一个类似守卫结构的函数routeEnterGuard，然后这个函数就是执行我们守卫钩子并返回。但是这里和平常守卫有些不一样，一般情况我们守卫的三个参数是to、from和next，但是这里执行guard第三个参数传的却是重新定义的一个函数，当然这里不用担心我们调用不到next的而导致程序卡在这里。因为它在最后会调用next(cb)，所以这里要分析的是在调用next之前加了什么逻辑，cb参数又是什么。
  这里是判断cb是不是一个function，如果是的话将它push到match.enteredCbs的对应key属性下。match是一条record，一条record又可能对应几个组件，每个组件又对应一个key。cb其实就是我们定义守卫时调用next传入的参数，我们可以来看官网描述的示例：

beforeRouteEnter (to, from, next) {
  next(vm => {
    // 通过 `vm` 访问组件实例
  })
}

  可以看到，这里的next支持传入一个函数，并且这个函数参数vm其实就组件的实例。这里其实是将组件实例暴露给用户，方便用户使用，因为这个时候还没有初始化组件实例，所以也是获取不到this的。所以这里采用回调函数方式，只要保证组件实例化后在调用回调就可以通过vm取到组件实例以方便用户使用。
  值得一提的是，我们这里能够直接获取组件的守卫得益于之前已经完成异步组件解析，从之前的分析可以看到我们异步组件解析逻辑就是把异步加载的组件赋值回到record的components，最后它其实和同步组件没有区别了，所以接下来才能无视同步和异步去解析它的'beforeRouteEnter'守卫。
  我们继续来看runQueue回调中的内容，执行完extractEnterGuards方法并将结果保存在enterGuards中后，又定义了一个queue数组，这个数组是连接enterGuards和this.router.resolveHooks组成的。
  导航第八个流程：调用全局beforeResolve守卫。也就是这里router.resolveHooks中的内容，和全局前置守卫的过程是差不多的：

  beforeResolve (fn: Function): Function {
    return registerHook(this.resolveHooks, fn)
  }

router.beforeResolve执行会将函数push到router.resolveHooks中，然后在接下来的迭代过程中执行。
  接下来就是再一次调用runQueue，只不过现在queue重定义了，回调函数也被重定义了。queue中的内容已经没有什么需要特别说明的了，接下来就是看新的回调函数做了什么事。
  新的回调也非常简单，判断如果this.pending不等于route，就中止执行。如果相等的话就将this.pending赋值为null。然后执行成功回调onComplete(route)。然后如果this.router.app存在的话就在一个nextTick后执行handleRouteEntered方法。我们先来看onComplete方法，这个方法是confirmTransition接收的第二个参数

  () => {
    this.updateRoute(route)
    onComplete && onComplete(route)
    this.ensureURL()
    this.router.afterHooks.forEach(hook => {
      hook && hook(route, prev)
    })

    // fire ready cbs once
    if (!this.ready) {
      this.ready = true
      this.readyCbs.forEach(cb => {
        cb(route)
      })
    }
  }

  这里的this是history对象，先调用updateRoute方法

  updateRoute (route: Route) {
    this.current = route
    this.cb && this.cb(route)
  }

  updateRoute方法先是将this.current更新为新的route，this.cb存在的话则调用cb()。这其实就是我们执行的第九步：导航被确认。
  然后这里又执行了一个onComplete函数，这个方法是transitionTo接收的成功回调，这是一个非必传选项，所以我们也暂时不用考虑它。
  接下来执行router.afterHooks中的hook函数，也就是我们第十步：调用全局的afterEach 钩子，最后如果this.ready还是false则将其赋值为true并执行readyCbs中的回调。这个是与history.onReady方法相关的，这里我们先不去分析ready这部分逻辑。
  第十一步则是触发DOM更新，到现在我们完全没有发现与DOM更新相关逻辑，所以我们先不去考虑DOM更新的内容。
  我们来看最后一步：调用beforeRouteEnter守卫中传给next的回调函数，创建好的组件实例会作为回调函数的参数传入。这也就是解决之前this获取不到的问题。这个时候既然DOM已经更新了，说明Vue组件都已经实例化完成，所以这个时候是可以获取到组件实例的，那么之前我们beforeRouteEnter守卫中next的回调函数的vm参数就可以得到了。所以我们来看handleRouteEntered：

export function handleRouteEntered (route: Route) {
  for (let i = 0; i < route.matched.length; i++) {
    const record = route.matched[i]
    for (const name in record.instances) {
      const instance = record.instances[name]
      const cbs = record.enteredCbs[name]
      if (!instance || !cbs) continue
      delete record.enteredCbs[name]
      for (let i = 0; i < cbs.length; i++) {
        if (!instance._isBeingDestroyed) cbs[i](instance)
      }
    }
  }
}

  这里遍历route.matched，然后通过for in循环record.instances，也就是去获取组件实例，然后再从record.enteredCbs去找对应name的cbs，这个东西是之前调用beforeRouteEnter时，会将回调函数都push 到enteredCbs[name]中，之前已经分析过了。如果组件实例或者回调函数不存在就跳过这次循环。获取到本次循环的cbs后就把它从enteredCbs中删除。我们获得的cbs是一个数组，下面我们模拟一个enteredCbs的结构：

record：{
  enteredCbs: {
    default: [vm => {...}]
  }
}

default数组中的函数就是我们beforeRouteEnter守卫定义时调用next时传入的函数，然后可以看到它的形参就是vm实例。
  我们回到函数中看最后一段for循环，遍历cbs并且如果组件没有被销毁就执行回调函数并且将instance作为参数传入，所以我们这里vm就是instance。那到这里我们导航守卫逻辑都分析完毕了。但是这里我们还有两个坑没解决，一个比较重要的是DOM更新的问题；另一个则是history.ensureURL函数的问题，这个函数在多个地方都有被调用，但是我们还没有分析它的逻辑。这两个问题我们会在后面小节去解决它。
