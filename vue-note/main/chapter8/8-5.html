# 8.5 导航守卫（上）

  上个小节我们分析了transitionTo方法首先会执行match方法，接下来我们继续往下看transitionTo

  transitionTo (
    location: RawLocation,
    onComplete?: Function,
    onAbort?: Function
  ) {
    let route
    // catch redirect option https://github.com/vuejs/vue-router/issues/3201
    try {
      route = this.router.match(location, this.current)
    } catch (e) {
      this.errorCbs.forEach(cb => {
        cb(e)
      })
      // Exception should still be thrown
      throw e
    }
    const prev = this.current
    this.confirmTransition(
      route,
      () => {
        this.updateRoute(route)
        onComplete && onComplete(route)
        this.ensureURL()
        this.router.afterHooks.forEach(hook => {
          hook && hook(route, prev)
        })

        // fire ready cbs once
        if (!this.ready) {
          this.ready = true
          this.readyCbs.forEach(cb => {
            cb(route)
          })
        }
      },
      err => {
        if (onAbort) {
          onAbort(err)
        }
        if (err && !this.ready) {
          // Initial redirection should not mark the history as ready yet
          // because it's triggered by the redirection instead
          // https://github.com/vuejs/vue-router/issues/3225
          // https://github.com/vuejs/vue-router/issues/3331
          if (!isNavigationFailure(err, NavigationFailureType.redirected) || prev !== START) {
            this.ready = true
            this.readyErrorCbs.forEach(cb => {
              cb(err)
            })
          }
        }
      }
    )
  }

  定义prev=this.current，然后接下来逻辑都是执行confirmTransition方法，看着很长的一段都是confirmTransition方法的参数，第一个参数是route，第二个参数是一个函数，第三个函数是错误回调。
  然后来看confirmTransition

  confirmTransition (route: Route, onComplete: Function, onAbort?: Function) {
    const current = this.current
    this.pending = route
    const abort = err => {...}
    const lastRouteIndex = route.matched.length - 1
    const lastCurrentIndex = current.matched.length - 1
    if (
      isSameRoute(route, current) &&
      // in the case the route map has been dynamically appended to
      lastRouteIndex === lastCurrentIndex &&
      route.matched[lastRouteIndex] === current.matched[lastCurrentIndex]
    ) {
      this.ensureURL()
      if (route.hash) {
        handleScroll(this.router, current, route, false)
      }
      return abort(createNavigationDuplicatedError(current, route))
    }

    const { updated, deactivated, activated } = resolveQueue(
      this.current.matched,
      route.matched
    )
    const queue: Array<?NavigationGuard>
    const iterator = (hook: NavigationGuard, next) => {...}
    runQueue(queue, iterator, () => {...})
  }

  我们先来简单分析一下confirmTransition，首先定义current，给this.pending赋值为route参数，然后定义abort方法。然后定义两个索引位。
  接下来是一个if判断，先通过isSameRoute方法判断是不是相同路由，这里传入了route和current对象，current是一个用来保存当前路由的属性，一开始会设置它为START。route是当前跳转的路径，然后我们来看isSameRoute方法

export function isSameRoute (a: Route, b: ?Route, onlyPath: ?boolean): boolean {
  if (b === START) {
    return a === b
  } else if (!b) {
    return false
  } else if (a.path && b.path) {
    return a.path.replace(trailingSlashRE, '') === b.path.replace(trailingSlashRE, '') && (onlyPath ||
      a.hash === b.hash &&
      isObjectEqual(a.query, b.query))
  } else if (a.name && b.name) {
    return (
      a.name === b.name &&
      (onlyPath || (
        a.hash === b.hash &&
      isObjectEqual(a.query, b.query) &&
      isObjectEqual(a.params, b.params))
      )
    )
  } else {
    return false
  }
}

  如果b === START，START是起始路由，就返回a === b，显然这里a是我们定义的路由，它们是不相等的。再判断如果b不存在就返回false，再如果a.path和b.path都存在就判断两个path去掉尾部/是否是相同的并且要满足onlyPath为true或者两个对象hash值相同且isObjectEqual(a.query, b.query)为true。

function isObjectEqual (a = {}, b = {}): boolean {
  // handle null value #1566
  if (!a || !b) return a === b
  const aKeys = Object.keys(a).sort()
  const bKeys = Object.keys(b).sort()
  if (aKeys.length !== bKeys.length) {
    return false
  }
  return aKeys.every((key, i) => {
    const aVal = a[key]
    const bKey = bKeys[i]
    if (bKey !== key) return false
    const bVal = b[key]
    // query values can be null and undefined
    if (aVal == null || bVal == null) return aVal === bVal
    // check nested equality
    if (typeof aVal === 'object' && typeof bVal === 'object') {
      return isObjectEqual(aVal, bVal)
    }
    return String(aVal) === String(bVal)
  })
}

  isObjectEqual看似比较复杂，其实就是比较两个对象是否相同，这里相同不是指相同引用，而是指是否有相同的键值对。
  所以上面是判断query是否相同。再往下则是判断是否都存在name，然后就和上面过程基本相同，先判断name相同，并且hash、query、params都相同。因为params就是配合name使用的。当然如果传入onlyPath为true就只需要满足那么相同就好了。
  然后回到confirmTransition函数中，这个if判断仅有isSameRoute相同是不够的还要保证route.matched和current.matched数组长度相同并且最后一个元素也相同，在说为什么这样做之前我们先说说matched是什么。其实之前我们在createRoute方法中已经分析过它了。详见vue-router文档。它是一个从根路径到下级路径一直到当前路径一个数组记录，它的每一个元素都是一个record。这里注释说防止动态修改，但是比较遗憾没能找到什么情况会满足isSameRoute但不是同一个路由。
  那这里如果跳转的是同一个路由的话就会报一个错误并中止程序。在这之前还需要执行this.ensureURL()，然后如果route.hash存在还要执行handleScroll方法。最后执行abort并返回。abort这部分非主线逻辑我们放到vue-router文档(第三部分)分析。
接下就要正式进入导航守卫的分析了。

const { updated, deactivated, activated } = resolveQueue(
  this.current.matched,
  route.matched
)

  先执行resolveQueue方法，然后从其执行结果中解构出update、deactivated、activated。这里传入的参数就不再赘述了。然后我们来看resolveQueue方法

function resolveQueue (
  current: Array<RouteRecord>,
  next: Array<RouteRecord>
): {
  updated: Array<RouteRecord>,
  activated: Array<RouteRecord>,
  deactivated: Array<RouteRecord>
} {
  let i
  const max = Math.max(current.length, next.length)
  for (i = 0; i < max; i++) {
    if (current[i] !== next[i]) {
      break
    }
  }
  return {
    updated: next.slice(0, i),
    activated: next.slice(i),
    deactivated: current.slice(i)
  }
}

  可以看到这个方法功能十分简单，先找到current和next数组长度比较长的那个数组长度。然后拿max开始遍历数组，直到第一次找到current和next中不同的元素时跳出循环，这时i保存着不同元素开始的位置。然后根据i去截取数组并返回update、deactivated、activated。updated是相同的record组成的数组，也就是需要更新的内容。activated是新的路径不同部分组成的数组，也就是需要激活的。deactivated是旧的路径不同部分组成的数组，也就是失活的部分。
  再往下看confirmTransition方法，先是定义了queue数组和一个迭代器函数iterator，然后执行runQueue并将queue、iterator分别作为第一、二个参数传入。runQueue函数第三个参数是一个函数，如下所示：

  runQueue(queue, iterator, () => {
    // wait until async components are resolved before
    // extracting in-component enter guards
    const enterGuards = extractEnterGuards(activated)
    const queue = enterGuards.concat(this.router.resolveHooks)
    runQueue(queue, iterator, () => {
      if (this.pending !== route) {
        return abort(createNavigationCancelledError(current, route))
      }
      this.pending = null
      onComplete(route)
      if (this.router.app) {
        this.router.app.$nextTick(() => {
          handleRouteEntered(route)
        })
      }
    })
  })

  下面我们来看runQueue方法，定义于util/async.js中

export function runQueue (queue: Array<?NavigationGuard>, fn: Function, cb: Function) {
  const step = index => {
    if (index >= queue.length) {
      cb()
    } else {
      if (queue[index]) {
        fn(queue[index], () => {
          step(index + 1)
        })
      } else {
        step(index + 1)
      }
    }
  }
  step(0)
}

  定义step函数，step接收一个参数index，最后执行step(0)。step函数执行是一个递归的过程，如果queue[index]存在就调用fn方法并将queue[index]作为参数传入，然后第二个参数是一个递归调用step的函数，但是每次调用要使index + 1。如果queue[index]是一个空值的话就直接执行step(index + 1)。当然如果index超出最大索引就执行cb回调函数，也就是我们上面传入的第三个参数。这里的fn是第二个参数，也就是iterator。下面来看看iterator的定义

  const iterator = (hook: NavigationGuard, next) => {
    if (this.pending !== route) {
      return abort(createNavigationCancelledError(current, route))
    }
    try {
      hook(route, current, (to: any) => {
        if (to === false) {
          // next(false) -> abort navigation, ensure current URL
          this.ensureURL(true)
          abort(createNavigationAbortedError(current, route))
        } else if (isError(to)) {
          this.ensureURL(true)
          abort(to)
        } else if (
          typeof to === 'string' ||
          (typeof to === 'object' &&
            (typeof to.path === 'string' || typeof to.name === 'string'))
        ) {
          // next('/') or next({ path: '/' }) -> redirect
          abort(createNavigationRedirectedError(current, route))
          if (typeof to === 'object' && to.replace) {
            this.replace(to)
          } else {
            this.push(to)
          }
        } else {
          // confirm transition and pass on the value
          next(to)
        }
      })
    } catch (e) {
      abort(e)
    }
  }

  在confirmTransition方法开头this.pending被设置为route，如果这里pending不等于route的话，那就说明有问题了，就中止执行。
  接着尝试执行hook方法，这个hook方法是什么呢？我们先来看下面一个例子：

  router.beforeEach((to, from, next) => {
    // ...
    next()
  })

  这是一个全局路由守卫的示例，这个参数同样是个函数并且有三个函数，是不是和hook函数有点类似呢？这样看貌似也不是很像，但是我们把hook的参数route和current翻译一下呢？就是要跳转的路径和当前路径。是不是就和to、from对应起来了呢？然后就是看next是不是也一样，先看hook的第三个参数定义的方法，该方法接收一个参数to。然后第一段逻辑就是判断to === false的话中止执行，这是不是就和我们的next(false)对应了。那么接下来的逻辑也就简单了。如果to是一个error对象时也要中止。如果to是string或者to是object并且to至少存在path和name中的一个属性，那就要中止并报一个重定向错误，因为这里是通过next指定跳转路径导致不是按原来路径跳转了。
  但是这个行为是被允许的，所以这里并不是被直接中止，还有后续逻辑，比如to是一个对象并且to.place属性为true时会执行this.replace去跳转，否则就通过push方法去跳转。这两个跳转方法我们应该不会陌生。最后else逻辑就是一般情况下我们直接调用next()时候执行了。这时next(to)执行的实际上传入的next方法，也就是执行step(index + 1)，即执行下一步。直到queue队列中的方法都执行完成才会执行runQueue中的cb回调。
  到现在其实我们应该已经猜到queue应该就是各个导航守卫函数队列了，虽然我们没有分析它。这个我们会在下一小节分析。对于我们hook中next方法的设计我们在分析过程中可能比较懵，但是结合官方文档我们很容易就理解，下面是官文中对next的描述：
next: Function: 一定要调用该方法来 resolve 这个钩子。执行效果依赖 next 方法的调用参数。
（1）next(): 进行管道中的下一个钩子。如果全部钩子执行完了，则导航的状态就是 confirmed (确认的)。
（2）next(false): 中断当前的导航。如果浏览器的 URL 改变了 (可能是用户手动或者浏览器后退按钮)，那么 URL 地址会重置到 from 路由对应的地址。
（3）next('/') 或者 next({ path: '/' }): 跳转到一个不同的地址。当前的导航被中断，然后进行一个新的导航。你可以向 next 传递任意位置对象，且允许设置诸如 replace: true、name: 'home' 之类的选项以及任何用在 router-link 的 to prop 或 router.push 中的选项。
（4）next(error): (2.4.0+) 如果传入 next 的参数是一个 Error 实例，则导航会被终止且该错误会被传递给 router.onError() 注册过的回调。
  到这里我们应该就了解为什么使用全局导航守卫最后都要调用next()方法了。那么下一小节我们就基于导航解析流程来具体分析各个导航守卫吧！
