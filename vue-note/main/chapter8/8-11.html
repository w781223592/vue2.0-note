# 8.11 history模式

  之前我们一直分析的都是hash模式，这小节我们就来看看history模式有什么不同。其实我们知道history不同之处就是在url处理上，hash路由会多一个#，这个带来的好处就是hash值改变不会刷新页面，对服务器没有影响。而history模式则不同，改变url路径后会重新请求资源，但是H5提供的pushState()和replaceState()允许直接更改前端路由而不刷新页面。但是用户直接去访问页面的一个路由时会由于匹配不到相应的资源而返回404，所以这需要服务端的正确配置才可以。
  我们单页应用只有一个html文件，比如我们访问http://oursite.com是可以访问到的，但是如果直接修改url访问http://oursite.com/user时因为这个url下没有html资源所以就会返回404了。
  接下来我们就看看history代码实现有什么不同吧。

  constructor (router: Router, base: ?string, fallback: boolean) {
    super(router, base)
    // check history fallback deeplinking
    if (fallback && checkFallback(this.base)) {
      return
    }
    ensureSlash()
  }
  // history
  constructor (router: Router, base: ?string) {
    super(router, base)

    this._startLocation = getLocation(this.base)
  }

  这两个分别是HashHistory和HTML5History的构造器，ensureSlash方法保证了路径有#/，而history模式下则是执行getLocation方法

export function getLocation (base: string): string {
  let path = window.location.pathname
  const pathLowerCase = path.toLowerCase()
  const baseLowerCase = base.toLowerCase()
  // base="/a" shouldn't turn path="/app" into "/a/pp"
  // https://github.com/vuejs/vue-router/issues/3555
  // so we ensure the trailing slash in the base
  if (base && ((pathLowerCase === baseLowerCase) ||
    (pathLowerCase.indexOf(cleanPath(baseLowerCase + '/')) === 0))) {
    path = path.slice(base.length)
  }
  return (path || '/') + window.location.search + window.location.hash
}

  我们以http://localhost:8080/home为例来分析，先获取pathname，也就是/home。然后将它转化为小写，同时把base也转化成小写。然后是解决一个base为/a时会把path为/app转化为/a/pp，为了解决这个问题处理方案是如果base和path完全一样那就只保留一个，如果path包含base，那么必须在base后拼接/时path仍然包含base时才做处理，比如base=/a，path=/a/app则只保留一个/a，最后结果为/a/app。
  最后返回path或者/，后面并加上search和hash。

  const handleRoutingEvent = () => {

  // history
  const handleRoutingEvent = () => {
    const current = this.current

    // Avoiding first `popstate` event dispatched in some browsers but first
    // history route not updated since async guard at the same time.
    // 避免在某些浏览器中调度第一个“popstate”事件，但由于异步保护，第一个历史路由未同时更新。
    const location = getLocation(this.base)
    if (this.current === START && location === this._startLocation) {
      return
    }
    this.transitionTo(location, route => {
      if (supportsScroll) {
        handleScroll(router, route, current, true)
      }
    })
  }
  window.addEventListener('popstate', handleRoutingEvent)
  this.listeners.push(() => {
    window.removeEventListener('popstate', handleRoutingEvent)
  })

  事件监听也略有不同，总的来说差距不大，监听这块只监听了popstate

  push (location: RawLocation, onComplete?: Function, onAbort?: Function) {
    const { current: fromRoute } = this
    this.transitionTo(location, route => {
      pushState(cleanPath(this.base + route.fullPath))
      handleScroll(this.router, route, fromRoute, false)
      onComplete && onComplete(route)
    }, onAbort)
  }

  replace (location: RawLocation, onComplete?: Function, onAbort?: Function) {
    const { current: fromRoute } = this
    this.transitionTo(location, route => {
      replaceState(cleanPath(this.base + route.fullPath))
      handleScroll(this.router, route, fromRoute, false)
      onComplete && onComplete(route)
    }, onAbort)
  }

push方法和replace方法也差距不大，在路径上多拼了this.base，然后这里是直接执行pushState和replaceState，而hash模式则是调用pushHash和replaceHash方法，这两个方法实质上是多调用了getUrl去处理#，然后再调用pushState和replaceState。这里是直接调用pushState和replaceState就不会有#号。 总的来说hash和history没有很大差距，代码差距主要在url处理方面，在实现原理上没有太大差距。
