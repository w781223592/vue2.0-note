# 8.3 动态路由

  动态路由是vue-router 4.x的一个概念，尽管3.x已经有这个API了，但是官网几乎没有给它篇幅去介绍。首先区分一点，动态路由不是动态路由匹配。关于这个我们可以翻看vue-router 4.x文档去帮助理解： 动态路由 (opens new window)
  上小节我们分析到router.matcher对象是由createMatcher方法返回的。接下来我们就分析一下createMatcher方法，在create-matcher.js中我们找到了它的定义

export function createMatcher (
  routes: Array<RouteConfig>,
  router: VueRouter
): Matcher {
  const { pathList, pathMap, nameMap } = createRouteMap(routes)

  function addRoutes (routes) {
    createRouteMap(routes, pathList, pathMap, nameMap)
  }
  function addRoute (parentOrRoute, route) {}
  function getRoutes () {}
  function match (
    raw: RawLocation,
    currentRoute?: Route,
    redirectedFrom?: Location
  ): Route {}
  function redirect (
    record: RouteRecord,
    location: Location
  ): Route {}
  function alias (
    record: RouteRecord,
    location: Location,
    matchAs: string
  ): Route {}
  function _createRoute (
    record: ?RouteRecord,
    location: Location,
    redirectedFrom?: Location
  ): Route {}
  return {
    match,
    addRoute,
    getRoutes,
    addRoutes
  }
}

  可以看到这个方法是非常复杂的，我们还是按照惯例用到哪方法分析哪个方法。简单来看这个方法就是返回了一个带有match、addRoute、getRouters、addRoutes方法的对象。所以router.matcher就是这么一个对象，那么接下来我们就看看它的这几个方法分别是做什么的。
  这里调用createMatcher方法传入了两个参数，一个是options.routes，一个是router对象

this.matcher = createMatcher(options.routes || [], this)

  然后我们来看createMatcher方法逻辑，首先调用createRouteMap(routes)方法，然后解构获取pathList、pathMap、nameMap。然后加下来是addRoutes方法的定义，这个方法是createMatcher最后返回方法之一，所以我们来看看它做了什么事：

  function addRoutes (routes) {
    createRouteMap(routes, pathList, pathMap, nameMap)
  }

  奇怪的是它又一次调用了createRouteMap，并且传入了上次调用它时获取的三个变量作为参数。所以接下来我们需要分析createRouteMap做了什么事。该方法定义于create-route-map.js中：

export function createRouteMap (
  routes: Array<RouteConfig>,
  oldPathList?: Array<string>,
  oldPathMap?: Dictionary<RouteRecord>,
  oldNameMap?: Dictionary<RouteRecord>,
  parentRoute?: RouteRecord
): {
  pathList: Array<string>,
  pathMap: Dictionary<RouteRecord>,
  nameMap: Dictionary<RouteRecord>
} {
  // the path list is used to control path matching priority
  const pathList: Array<string> = oldPathList || []
  // $flow-disable-line
  const pathMap: Dictionary<RouteRecord> = oldPathMap || Object.create(null)
  // $flow-disable-line
  const nameMap: Dictionary<RouteRecord> = oldNameMap || Object.create(null)

  routes.forEach(route => {
    addRouteRecord(pathList, pathMap, nameMap, route, parentRoute)
  })
//省略
  return {
    pathList,
    pathMap,
    nameMap
  }
}

  这个方法最终就是返回pathList, pathMap, nameMap这三个变量。然后函数开头就是定义了这三个变量，如果传入了第2、3、4参数那么就去这些参数值，否则就初始化为空值。
  接下来就是遍历routes数组，然后循环调用addRouteRecord方法，那么接下来就看addRouteRecord方法了

function addRouteRecord (
  pathList: Array<string>,
  pathMap: Dictionary<RouteRecord>,
  nameMap: Dictionary<RouteRecord>,
  route: RouteConfig,
  parent?: RouteRecord,
  matchAs?: string
) {}

  它总共接收6个参数，但是我们这里只传入了五个参数，至于第五个参数这里也是空的，因为我们parentRoute是空的。接着往下看：

const { path, name } = route
  if (process.env.NODE_ENV !== 'production') {
    assert(path != null, `"path" is required in a route configuration.`)
    assert(
      typeof route.component !== 'string',
      `route config "component" for path: ${String(
        path || name
      )} cannot be a ` + `string id. Use an actual component instead.`
    )

    warn(
      // eslint-disable-next-line no-control-regex
      !/[^\u0000-\u007F]+/.test(path),
      `Route with path "${path}" contains unencoded characters, make sure ` +
        `your path is correctly encoded before passing it to the router. Use ` +
        `encodeURI to encode static segments of your path.`
    )
  }

  从route对象中获取我们定义的path和name，然后下一句逻辑是对错误的一个提示。还记得我们assert方法吗？在第一个参数的条件不成立的情况下就会报第二参数所提示的错误。这里path != null不成立也就是path为空时才会报错"path" is required in a route configuration.译为path是route配置所必须的。接下来是判断route的component如果是一个string的话，说明它不是一个真实的组件，这时会报一个错让你使用真实的组件。
  接下来是一个warn，warn定义于assert相同，只不过它是通过console.warn()打印一个警告，而不是去抛出异常。这个正则稍微有些绕，\u0000-\u007F对应128个ASCII码，这个正则是用来检验path中是否含有这128个字符以外的字符，比如中文字符，如果有的话正则检查为true，然后前面取反则是false，但是warn中再次取反判断则为true，所以会触发警告。这个警告大致意思是：路径为xxx的路由包含未编码的字符，请确保编码正确再使用路由。可以通过encodeURI进行编码。

  const pathToRegexpOptions: PathToRegexpOptions =
    route.pathToRegexpOptions || {}
  const normalizedPath = normalizePath(path, parent, pathToRegexpOptions.strict)

  接下来又定义了两个变量，pathToRegexpOptions是v2.6新增编译正则的选项。然后定义了normalizedPath，这个是normalizedPath方法标准化后的path，其定义如下：

function normalizePath (
  path: string,
  parent?: RouteRecord,
  strict?: boolean
): string {
  if (!strict) path = path.replace(/\/$/, '')
  if (path[0] === '/') return path
  if (parent == null) return path
  return cleanPath(`${parent.path}/${path}`)
}

  可以看到这个方法十分简单，如果!strict就把path末尾的/置为空字符串。如果path第一位是/或者parent不存在都直接返回path，不然的话就将parent.path和path通过/连接并经过cleanPath方法处理后返回。
  这部分逻辑主要是为方便我们去定义路由路径。我们书写子路由时就不需要写全部路径，如下官网嵌套路由示例：

const router = new VueRouter({
  routes: [
    {
      path: '/user/:id',
      component: User,
      children: [{
        path: 'profile',
        component: UserProfile
      }]
    }
  ]
})

  定义子路由时不需要写成/user/profile，vue-router内部会自动处理。cleanPath方法定义在util/path.js中

export function cleanPath (path: string): string {
  return path.replace(/\/(?:\s*\/)+/g, '/')
}

  主要作用就是将//变为一个/，避免一些笔误。我们继续往下看addRouteRecord方法

  if (typeof route.caseSensitive === 'boolean') {
    pathToRegexpOptions.sensitive = route.caseSensitive
  }

  const record: RouteRecord = {
    path: normalizedPath,
    regex: compileRouteRegex(normalizedPath, pathToRegexpOptions),
    components: route.components || { default: route.component },
    alias: route.alias
      ? typeof route.alias === 'string'
        ? [route.alias]
        : route.alias
      : [],
    instances: {},
    enteredCbs: {},
    name,
    parent,
    matchAs,
    redirect: route.redirect,
    beforeEnter: route.beforeEnter,
    meta: route.meta || {},
    props:
      route.props == null
        ? {}
        : route.components
          ? route.props
          : { default: route.props }
  }

  如果定义了caseSensitive属性则赋值给pathToRegexpOptions的sensitive属性，接下来是定义了record对象。这个其实就是在我们的配置基础上做了一些变动，并且增加了一些属性。接下来是对route.children的处理

if (route.children) {
  // Warn if route is named, does not redirect and has a default child route.
  // If users navigate to this route by name, the default child will
  // not be rendered (GH Issue #629)
  if (process.env.NODE_ENV !== 'production') {
    if (
      route.name &&
      !route.redirect &&
      route.children.some(child => /^\/?$/.test(child.path))
    ) {
      warn(
        false,
        `Named Route '${route.name}' has a default child route. ` +
          `When navigating to this named route (:to="{name: '${
            route.name
          }'}"), ` +
          `the default child route will not be rendered. Remove the name from ` +
          `this route and use the name of the default child route for named ` +
          `links instead.`
      )
    }
  }
  route.children.forEach(child => {
    const childMatchAs = matchAs
      ? cleanPath(`${matchAs}/${child.path}`)
      : undefined
    addRouteRecord(pathList, pathMap, nameMap, child, record, childMatchAs)
  })
}

  首先如果我们路由已经命名，而且我们没有重定向路径，并且它有一个子路由，那么它会发出警告。接下来遍历route.children数组，如果matchAs存在的话就返回拼接后的路径，否则返回undefined。最后递归调用addRouteRecord方法，这时的调用会传入matchAs参数，但是这个目前这个过程中matchAs都是undefined。

  if (!pathMap[record.path]) {
    pathList.push(record.path)
    pathMap[record.path] = record
  }

  判断pathMap是否没有record.path，如果没有的话，就把对应path添加到pathList数组中，并且pathMap中保存这个映射关系。

  if (route.alias !== undefined) {
    const aliases = Array.isArray(route.alias) ? route.alias : [route.alias]
    for (let i = 0; i < aliases.length; ++i) {
      const alias = aliases[i]
      if (process.env.NODE_ENV !== 'production' && alias === path) {
        warn(
          false,
          `Found an alias with the same value as the path: "${path}". You have to remove that alias. It will be ignored in development.`
        )
        // skip in dev to make it work
        continue
      }

      const aliasRoute = {
        path: alias,
        children: route.children
      }
      addRouteRecord(
        pathList,
        pathMap,
        nameMap,
        aliasRoute,
        parent,
        record.path || '/' // matchAs
      )
    }
  }

  这段代码是路由的别名配置，这个东西用的实在是不多，我们简单看一下就行了，实在没必要去大费周章去分析它。但是可以看一下它最后的也调用了addRouteRecord，这里传入了第六个参数，也就是matchAs，这也解开了我们心中的疑惑，那就是matchAs的作用就是为别名服务的。目前为止我们传入的三个变量已经知道其中两个是干什么用的了，还有最后nameMap没有使用过，接下来就有它发挥的时候了。

  if (name) {
    if (!nameMap[name]) {
      nameMap[name] = record
    } else if (process.env.NODE_ENV !== 'production' && !matchAs) {
      warn(
        false,
        `Duplicate named routes definition: ` +
          `{ name: "${name}", path: "${record.path}" }`
      )
    }
  }

  如果name存在的话并且nameMap中没有对应的name属性，那就把这个路由record按照name与record的映射关系保存到nameMap集合中。如果存在并且!matchAs则说明存在重复的name，那么就报一个警告。那么到这里addRouteRecord方法就算是分析完了。然后我们还是回到createRouteMap方法中继续往下分析

// ensure wildcard routes are always at the end
for (let i = 0, l = pathList.length; i < l; i++) {
  if (pathList[i] === '*') {
    pathList.push(pathList.splice(i, 1)[0])
    l--
    i--
  }
}

  这个逻辑就非常好理解了，因为注释实在写的太清楚了，就是确保通配符路由总是在最后，因为如果通配符在放在前面的话，那么后面路由就都不会匹配到了，就全部匹配到通配符*所对应的路由组件了。

if (process.env.NODE_ENV === 'development') {
  // warn if routes do not include leading slashes
  const found = pathList
  // check for missing leading slash
    .filter(path => path && path.charAt(0) !== '*' && path.charAt(0) !== '/')

  if (found.length > 0) {
    const pathNames = found.map(path => `- ${path}`).join('\n')
    warn(false, `Non-nested routes must include a leading slash character. Fix the following routes: \n${pathNames}`)
  }
}

  这部分逻辑是在开发环境执行的，就是去判断除了通配符path外的其他path是否含有/，如果不含有的话则报警告。因为这时的路由已经是被处理过的了，所有的path都应该是带有/的。最后createRouteMap就是返回我们pathList、pathMap、nameMap。
  现在回过头来思考，createRouteMap方法只是定义了pathList、pathMap、nameMap并通过addRouteRecord方法去修改这三个变量。最后createRouteMap在对它们简单校验后返回。
  然后我们再往回看，我们是在createMatcher方法分析addRoutes方法时用到的createRouteMap方法的。官网上就有router.addRoutes方法。当然这里的addRoutes方法和那并不是一个东西，因为这里我们分析的是matcher.addRoutes。当然这里既然它也叫addRoutes方法，那么它们之间肯定是有联系的。我们不妨先看看router.addRoutes定义：

  addRoutes (routes: Array<RouteConfig>) {
    if (process.env.NODE_ENV !== 'production') {
      warn(false, 'router.addRoutes() is deprecated and has been removed in Vue Router 4. Use router.addRoute() instead.')
    }
    this.matcher.addRoutes(routes)
    if (this.history.current !== START) {
      this.history.transitionTo(this.history.getCurrentLocation())
    }
  }

  比较遗憾的是，这个方法在vue-router 4.x以上版本废弃了，这里我们分析的是3.x版本，但是它也给我们报了一个警告，它告诉我们可以用addRoute方法代替。但是既然还能用我们不妨也先看看它做了什么。可以看到它就是调用了matcher.addRoutes方法。所以我们现在就可以知道，为什么addRoutes又重复调用createRouteMap并且把上一次调用的结果当做参数传入了。
  第一次调用是在createMather开头获取pathList、pathMap、nameMap时，这些东西都是我们初始定义的静态routes数据来生成的。但是如果我们有一个需要，比如在做权限管理时，往往需要根据账号角色来生成不同的路由，比如一个公司的后台系统，运营人员可能只能访问运营相关页面的路由，而不能访问财务相关的页面，但是我们一开始又不知道这个人是什么角色的账号，只有等他登录后才能知道角色信息，这时候就需要登录后根据角色条件去动态添加路由。也就是调用router.addRoutes方法，这样就是二次调用createRouteMap方法了。这时需要把原来静态配置生成的旧路由传进去，这样createRouteMap就复用原来pathList等数据而不是新建一个变量。这样就实现了添加路由工能。router.addRoutes方法最后还有一段逻辑，但我们暂时不在这里分析它。
  既然已经分析了addRoutes方法，我们不妨来看看它的迁移方法addRoute，我们先来看router.addRoute

  addRoute (parentOrRoute: string | RouteConfig, route?: RouteConfig) {
    this.matcher.addRoute(parentOrRoute, route)
    if (this.history.current !== START) {
      this.history.transitionTo(this.history.getCurrentLocation())
    }
  }

我们发现和之前基本上一样的，唯一不同的是传入的参数原来是整个routes配置数组，现在变成parentOrRoute和route。看到这个我们可以猜想它是做了更细致的添加，可以选择添加的某一父级下。这样应该会获得更快的速度。然后来看matcher.addRoute方法

  function addRoute (parentOrRoute, route) {
    const parent = (typeof parentOrRoute !== 'object') ? nameMap[parentOrRoute] : undefined
    // $flow-disable-line
    createRouteMap([route || parentOrRoute], pathList, pathMap, nameMap, parent)

    // add aliases of parent
    if (parent && parent.alias.length) {
      createRouteMap(
        // $flow-disable-line route is defined if parent is
        parent.alias.map(alias => ({ path: alias, children: [route] })),
        pathList,
        pathMap,
        nameMap,
        parent
      )
    }
  }

  首先是定义parent，如果parentOrRoute不是object的话，那就是一个string，那就根据parentRoute作为key去从nameMap中取路由对象，如果是object的话就直接赋值为undefined。然后同样是调用createRouteMap方法，这就基本和addRoutes过程相同了。不同的是第一个参数传入route或者parentOrRoute，并将其用数组包裹，因为我们传入的值必须是个数组类型，如果不传第二个参数，其实就说明第一个参数传入的是要添加的路由，这时第一个参数必须是object类型，最后这里也传入第五个参数parent参数，也就是把这个路由对象放到哪一父级下。如果为undefined则说明要添加到第一级下。最后一个是对别名的判断，就不去分析它了，这个极少用的属性也实在想不出有什么应用场景了。
  我们知道matcher对象是返回四个属性的分别是match、addRoutes、addRoute、getRoutes。至于getRoutes也是后来新增的方法，它实在太过简单，我们就不专门叙述它了。最后就只剩match方法了，那么下个小节我们就来分析match方法。
