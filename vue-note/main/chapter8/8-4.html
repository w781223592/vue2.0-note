# 8.4 match

  在上小节我们分析matcher对象的addRoute方法，这个小节我们来分析match方法。match方法是history.transitionTo方法中会用到的方法，这我们在第2小节已经提到过了，接下来我们就来看看match方法做了什么。
  在路由初始化过程中会执行history.transitionTo方法:

  history.transitionTo(
    history.getCurrentLocation(),
    setupListeners,
    setupListeners
  )

  第一个参数是history.getCurrentLocation()，我们先以hash路由为例，在history/hash.js中来看getCurrentLocation的定义

  getCurrentLocation () {
    return getHash()
  }

  定义很简单就是调用了getHash方法，然后我们去找getHash方法，在当前文件下就看以找到它的定义

export function getHash (): string {
  // We can't use window.location.hash here because it's not
  // consistent across browsers - Firefox will pre-decode it!
  let href = window.location.href
  const index = href.indexOf('#')
  // empty path
  if (index < 0) return ''

  href = href.slice(index + 1)
  return href
}

  首先是一行注释告诉我们不能直接去获取window.location.hash，因为它在不同浏览器之间表现不同，在火狐浏览器种会预解码。这个方法主要是获取当前url#号后面内容并返回。然后接下来transitionTo方法就执行match方法了。

route = this.router.match(location, this.current)

  这里传入了location和this.current，location就是上面getHash的返回值，在new History时，this.current等于一个START常量。START定义于util/route中

export const START = createRoute(null, {
  path: '/'
})

  所以我们再去找createRoute方法的实现

export function createRoute (
  record: ?RouteRecord,
  location: Location,
  redirectedFrom?: ?Location,
  router?: VueRouter
): Route {
  const stringifyQuery = router && router.options.stringifyQuery

  let query: any = location.query || {}
  try {
    query = clone(query)
  } catch (e) {}
  const route: Route = {
    name: location.name || (record && record.name),
    meta: (record && record.meta) || {},
    path: location.path || '/',
    hash: location.hash || '',
    query,
    params: location.params || {},
    fullPath: getFullPath(location, stringifyQuery),
    matched: record ? formatMatch(record) : []
  }
  if (redirectedFrom) {
    route.redirectedFrom = getFullPath(redirectedFrom, stringifyQuery)
  }
  return Object.freeze(route)
}

  首先是从options上获取stringifyQuery，它是对路由参数进行加密的选项，我们这里不用去关注它。然后从location上获取query，这里START只传了path: '/'，所以肯定是没有query的。接下来尝试去克隆query对象，这个我们也不用去管它。
  接下来就是定义route对象，可以看到这个对象其实就是this.$route对象了，我们取query参数，params参数等都需要通过它来获取。除了fullPath和matched方法需要特别说明外其他属性我们就不多赘述了。但是这两个方法可能会占用较多的篇幅，所以我就不在正文中去分析它了。vue-router文档(第一部分)。最后返回了冻结的route对象，也就是一旦定义了就不能去修改它了。 我们回到create-matcher.js文件中去找match方法

  function match (
    raw: RawLocation,
    currentRoute?: Route,
    redirectedFrom?: Location
  ): Route {
    const location = normalizeLocation(raw, currentRoute, false, router)
    const { name } = location

    if (name) {
    } else if (location.path) {
    }
    // no match
    return _createRoute(null, location)
  }

  首先是定义了location，它是由normalizeLocation方法返回的。然后获取location.name保存在name中。然后接下来是对name和location.path的判断，这部分判断逻辑比较多，所以为了阅读先不把它放这里，之后分析到时再贴出这部分代码。
  然后我们来看一下normalizeLocation方法是怎么实现的，在util/location.js可以找到它的定义，这个方法也是比较复杂的，如果我们在这里分析它的话，那我们文章对match方法的分析就不够条理了。所以还是把它放到vue-router文档(第二部分)中去分析。我们这里分析一下它传入了几个参数。这里传入四个参数，raw是match接收的第一个参数，差不多就相当于window.location.hash，这里我们先当它是一个/；第二个参数currentRoute也就是前面说的this.current对象；第三个参数是false；第四个参数是router。
  接下来简单分析了if(name)逻辑，如果location.name存在的话会执行下面逻辑：

  const record = nameMap[name]
  if (process.env.NODE_ENV !== 'production') {
    warn(record, `Route with name '${name}' does not exist`)
  }
  if (!record) return _createRoute(null, location)
  const paramNames = record.regex.keys
    .filter(key => !key.optional)
    .map(key => key.name)

  if (typeof location.params !== 'object') {
    location.params = {}
  }

  首先获取record并且判断它是否存在，如果不存在就报一个warn，并且返回一个_createRoute方法的执行结果。
  再往下定义paramNames，params主要用作动态路由匹配，这里获取record.regex中的keys，在分析addRouteRecord时我们没有分析regex，现在我们来看看regex是个什么，在create-route-map.js中有compileRouteRegex的定义，regex对象就是由其生成的：

function compileRouteRegex (
  path: string,
  pathToRegexpOptions: PathToRegexpOptions
): RouteRegExp {
  const regex = Regexp(path, [], pathToRegexpOptions)
  if (process.env.NODE_ENV !== 'production') {
    const keys: any = Object.create(null)
    regex.keys.forEach(key => {
      warn(
        !keys[key.name],
        `Duplicate param keys in route with path: "${path}"`
      )
      keys[key.name] = true
    })
  }
  return regex
}

  主要逻辑就是通过Regexp方法传入path和[]来生成的。这个方法其实是引入了一个js库path-to-regexp (opens new window)

const keys = [];
const regexp = pathToRegexp("/foo/:bar", keys);
// regexp = /^\/foo(?:\/([^\/#\?]+?))[\/#\?]?$/i
// keys = [{ name: 'bar', prefix: '/', suffix: '', pattern: '[^\\/#\\?]+?', modifier: '' }]

  使用示例如上所示，这个库会根据path去生成一个路由，传入第二参数还会生成一个keys数组。match方法就是对这个keys先进行过滤，然后再通过map得到一个由其name属性构成的数组。然后如果location.params不存在时初始化其值。

  if (currentRoute && typeof currentRoute.params === 'object') {
    for (const key in currentRoute.params) {
      if (!(key in location.params) && paramNames.indexOf(key) > -1) {
        location.params[key] = currentRoute.params[key]
      }
    }
  }

  获取currentRoute.params并判断如果是object的话，遍历params并且如果key在paramNames数组中的话，就把它加到location.params中。

  location.path = fillParams(record.path, location.params, `named route "${name}"`)
  return _createRoute(record, location, redirectedFrom)

  把处理过的record.path赋值给location.path，通过fillParams对path的params进行处理，下面来看一下fillParams方法

const regexpCompileCache: {
  [key: string]: Function
} = Object.create(null)

export function fillParams (
  path: string,
  params: ?Object,
  routeMsg: string
): string {
  params = params || {}
  try {
    const filler =
      regexpCompileCache[path] ||
      (regexpCompileCache[path] = Regexp.compile(path))
    // Fix #2505 resolving asterisk routes { name: 'not-found', params: { pathMatch: '/not-found' }}
    // and fix #3106 so that you can work with location descriptor object having params.pathMatch equal to empty string
    if (typeof params.pathMatch === 'string') params[0] = params.pathMatch
    return filler(params, { pretty: true })
  } catch (e) {
    if (process.env.NODE_ENV !== 'production') {
      // Fix #3072 no warn if `pathMatch` is string
      warn(typeof params.pathMatch === 'string', `missing param for ${routeMsg}: ${e.message}`)
    }
    return ''
  } finally {
    // delete the 0 if it was added
    delete params[0]
  }
}

  这个方法看起来比较复杂，我们简单分析一下就好了，执行Regexp.compile方法，这个方法会根据传入的path返回一个新的函数，这个新函数会可以将params参数和之前的path处理成一个动态路由匹配形式的path，第二个配置pretty会对动态匹配的内容解码，下面是几个示例

  var toPath = pathToRegexp.compile('/user/:id')

  toPath({ id: 123 }) //=> "/user/123"
  toPath({ id: 'café' }) //=> "/user/caf%C3%A9"
  toPath({ id: '/' }) //=> "/user/%2F"
  
  toPath({ id: ':' }) //=> "/user/%3A"
  toPath({ id: ':' }, { pretty: true }) //=> "/user/:"

  再回到match方法，最后返回_createRoute，这是我们第二次见这个方法了，我们把它放到后面分析。然后我们来分析name不存在时候逻辑，当name不存在时，我们还要判断path是否存在，如果path存在，也就是走else if(location.path)的逻辑。值得庆幸的是这个逻辑要少的多

  location.params = {}
  for (let i = 0; i < pathList.length; i++) {
    const path = pathList[i]
    const record = pathMap[path]
    if (matchRoute(record.regex, location.path, location.params)) {
      return _createRoute(record, location, redirectedFrom)
    }
  }

  这里先初始化location.params，然后遍历pathList，然后根据遍历出的path去从pathMap去找record。然后通过matchRoute去匹配path，如果匹配成功则调用_createRoute方法。matchRoute如下

function matchRoute (
  regex: RouteRegExp,
  path: string,
  params: Object
): boolean {
  const m = path.match(regex)

  if (!m) {
    return false
  } else if (!params) {
    return true
  }
  for (let i = 1, len = m.length; i < len; ++i) {
    const key = regex.keys[i - 1]
    if (key) {
      // Fix #1994: using * with props: true generates a param named 0
      params[key.name || 'pathMatch'] = typeof m[i] === 'string' ? decode(m[i]) : m[i]
    }
  }
  return true
}

  就是简单的用record.regex正则去匹配location.path，匹配结果保存在m中，如果m为undefined说明匹配失败返回false，如果匹配成功但是没有params就直接返回true，不需要执行额外功能。这里params传入了一个{}，判断是true，所以会执行下面逻辑，也就是给params添加对应的匹配属性与匹配值，最后再返回true。
  在match方法最后是没有匹配到路由的情况

  // no match
  return _createRoute(null, location)

  仍然是调用_createRoute方法，所以这里我们就来分析_createRoute方法。

  function _createRoute (
    record: ?RouteRecord,
    location: Location,
    redirectedFrom?: Location
  ): Route {
    if (record && record.redirect) {
      return redirect(record, redirectedFrom || location)
    }
    if (record && record.matchAs) {
      return alias(record, location, record.matchAs)
    }
    return createRoute(record, location, redirectedFrom, router)
  }

  这个方法看起来比较简单，实则不然，可以看到这里分三种情况，每一种情况都调用一个方法，我们先看createRoute方法，这个方法在本小节开头已经分析过了，最后就是得到route对象。关于别名的逻辑我们就不去分析，然后我们来看重定向逻辑，很容易就可以看到第一个if逻辑就是有关重定向的。如果我们定义路由时配置了redirect，那么就会执行到redirect方法：

  function redirect (
    record: RouteRecord,
    location: Location
  ): Route {
    const originalRedirect = record.redirect
    let redirect = typeof originalRedirect === 'function'
      ? originalRedirect(createRoute(record, location, null, router))
      : originalRedirect

    if (typeof redirect === 'string') {
      redirect = { path: redirect }
    }

    if (!redirect || typeof redirect !== 'object') {
      if (process.env.NODE_ENV !== 'production') {
        warn(
          false, `invalid redirect option: ${JSON.stringify(redirect)}`
        )
      }
      return _createRoute(null, location)
    }

    const re: Object = redirect
    const { name, path } = re
    let { query, hash, params } = location
    query = re.hasOwnProperty('query') ? re.query : query
    hash = re.hasOwnProperty('hash') ? re.hash : hash
    params = re.hasOwnProperty('params') ? re.params : params

    if (name) {
      // resolved named direct
      const targetRecord = nameMap[name]
      if (process.env.NODE_ENV !== 'production') {
        assert(targetRecord, `redirect failed: named route "${name}" not found.`)
      }
      return match({
        _normalized: true,
        name,
        query,
        hash,
        params
      }, undefined, location)
    } else if (path) {
      // 1. resolve relative redirect
      const rawPath = resolveRecordPath(path, record)
      // 2. resolve params
      const resolvedPath = fillParams(rawPath, params, `redirect route with path "${rawPath}"`)
      // 3. rematch with existing query and hash
      return match({
        _normalized: true,
        path: resolvedPath,
        query,
        hash
      }, undefined, location)
    } else {
      if (process.env.NODE_ENV !== 'production') {
        warn(false, `invalid redirect option: ${JSON.stringify(redirect)}`)
      }
      return _createRoute(null, location)
    }
  }

  这个方法看起来有点长，但是它的逻辑一点都不复杂。首先我们要知道redirect有三种定义方式，它可以是一个string，也可以是个对象或者函数，这个在官方文档很容易就可以找到，知道这个我们接下来分析就简单的多。函数首先是去判断redirect是不是一个函数，如果是函数的话就执行originalRedirect(createRoute(record, location, null, router))并保存其返回结果到redirect；如果不是一个函数就直接用redirect。这个并不难理解

  const router = new VueRouter({
    routes: [
      { path: '/a', redirect: to => {
        // 方法接收 目标路由 作为参数
        // return 重定向的 字符串路径/路径对象
      }}
    ]
  })

  定义为一个函数会接收一个to参数，这个to也就是要跳转的路由，所以这里传入的参数也必须是一个Route对象，但是这个对象还没有生成，所以要调用createRoute去生成这个对象再传入。然后这个函数要求用户返回一个object或者string。所以最后结果都是一个string或object。
  既然有两种结果，那么接下来肯定要针对两种情况去讨论了。所以接下来肯定是if判断，首先redirect如果是字符串的话，就要把它包装成一个对象{path：redirect}。所以object才是redirect的标准格式。但是用户是不可靠的，vueRouter不能保证用户一定能按规则来写代码，如果用户定义函数没有按照规则返回内容，那显然程序就会出问题，所以这里需要判断redirect是不是一个object，如果不是的话就给用户报一个警告并能保证程序能够继续往下进行。
  再往下就是获取name，如果我们定义redirect为一个对象，并且给它配置了name参数的话就能够得到name值，然后再获取query、hash、params等参数，如果重定向时重新配置了参数就取新的参数，否则就从location中取原本的参数。
  接下来就和match逻辑类似了，对name和path分别讨论，如果name存在并且能匹配到对应的record，那就再次调用match并返回，但是这里传入了一个对象，这个对象结构和normalizeLocation返回的对象是一致的，由于_normalized为true，所以在match时候就不用再次normalizeLocation。因为match中对于name存在时有对于参数方面的处理，这里又可能涉及query、params的改变，所以只需要重新执行match就可以了。
  然后如果name不存在，只存在path的话就稍微复杂一些，这里需要再次处理query和params。因为在match中没有对path处理参数，而是在normalizeLocation时候处理的。因为path的匹配比name匹配要麻烦的多。这里path分了三步走，先处理相对重定向路径。如果我们重定向../则要往前跳一级。然后是要处理新的params参数。至于query就不需要管了，直接作为参数传给match方法就行了，但是redirect的path定义不能带有query参数，如果你有一个路由path为/home，而重定向/home?id=1，这将会匹配不到路由，就是因为这里没有处理query。解决这个方法很简单，不要写这么奇怪的代码或者重新让它标准化：

  return match({
    _normalized: true,
    path: resolvedPath,
    query,
    hash
  }, undefined, location)

  这里是path逻辑第三步，和name逻辑一样都是调用match，因为match中第一句逻辑就是调用normalizeLocation方法，这里面就有处理path的逻辑了，但是这里把_normalized标记为true，它就不会重新初始化了，最简单的改法就是把这里的true改为false，就可以避免这个问题了。或者就是去里面找到相关的处理query的逻辑，比如parsePath等方法然后在这里调用也可以解决问题。
  这个小节我们主要分析了match方法，总的来说match就是根据传入的位置和路径对象计算出新的位置并去匹配对应路由，然后根据新的位置和record创建新的路径对象Route。match方法是路由比较关键的一个东西，它就负责把我们传入的跳转path或者name等属性与我们配置的路由进行匹配，如果能找到相应的路由配置，我们就能获取到我们想要渲染的组件，这就为我们渲染对应的组件打下基础了。
