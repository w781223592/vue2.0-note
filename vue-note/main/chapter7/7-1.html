  我们知道Vue中有很多好用的内置组件可供我们使用比如component、transition、transition-group、keep-alive还有slot，没错我们slot也是内置组件。但是我们放在上一章节去分析，因为它实际上是配合指令来使用的，并不是属于常规上的组件定义，它的实现都是依赖编译环节的处理。那我们这个章节就会来分析其他组件，这些组件才像是我们传统意义上的组件。

# 7.1 keep-alive（上）

  这小节我们来分析.组件的实现，在core/components/keep-alive.js下，可以看到keep-alive组件和我们正常组件差不多

  name: 'keep-alive',
  abstract: true,

  props: {
    include: patternTypes,
    exclude: patternTypes,
    max: [String, Number]
  }

  这里会定义一个abstract属性，标识它是抽象的。在initLifecycle中我们会为组件建立父子关系，但是如果发现父组件是抽象的那就会找父组件的父组件，也就是说抽象组件是不做真正的渲染的。
  这里有三个props
  include - 字符串或正则表达式。只有名称匹配的组件会被缓存。
  exclude - 字符串或正则表达式。任何名称匹配的组件都不会被缓存。
  max - 数字。最多可以缓存多少组件实例。
  我们再来看它的render函数

  render () {
    const slot = this.$slots.default
    const vnode: VNode = getFirstComponentChild(slot)
    const componentOptions: ?VNodeComponentOptions = vnode && vnode.componentOptions
    if (componentOptions) {
      // check pattern
      const name: ?string = getComponentName(componentOptions)
      const { include, exclude } = this
      if (
        // not included
        (include && (!name || !matches(include, name))) ||
        // excluded
        (exclude && name && matches(exclude, name))
      ) {
        return vnode
      }

      const { cache, keys } = this
      const key: ?string = vnode.key == null
        // same constructor may get registered as different local components
        // so cid alone is not enough (#3269)
        ? componentOptions.Ctor.cid + (componentOptions.tag ? `::${componentOptions.tag}` : '')
        : vnode.key
      if (cache[key]) {
        vnode.componentInstance = cache[key].componentInstance
        // make current key freshest
        remove(keys, key)
        keys.push(key)
      } else {
        // delay setting the cache until update
        this.vnodeToCache = vnode
        this.keyToCache = key
      }
      vnode.data.keepAlive = true
    }
    return vnode || (slot && slot[0])
  }

  首先会获取它的默认插槽slot。然后调用getFirstComponentChild去获取第一个子组件

export function getFirstComponentChild (children: ?Array<VNode>): ?VNode {
  if (Array.isArray(children)) {
    for (let i = 0; i < children.length; i++) {
      const c = children[i]
      if (isDef(c) && (isDef(c.componentOptions) || isAsyncPlaceholder(c))) {
        return c
      }
    }
  }
}

  这个方法定义也比较简单，就是遍历children然后找到第一个是组件的vnode或者是异步占位vnode返回。
  然后会获取vnode.componentOptions，然后会通过getComponentName方法去获取组件名

function getComponentName (opts: ?VNodeComponentOptions): ?string {
  return opts && (opts.Ctor.options.name || opts.tag)
}

  然后取到props中的include和exclude配置。然后就是判断如果不在include中或者是exclude排除在外的vnode会直接返回，而不去缓存。

function matches (pattern: string | RegExp | Array<string>, name: string): boolean {
  if (Array.isArray(pattern)) {
    return pattern.indexOf(name) > -1
  } else if (typeof pattern === 'string') {
    return pattern.split(',').indexOf(name) > -1
  } else if (isRegExp(pattern)) {
    return pattern.test(name)
  }
  /* istanbul ignore next */
  return false
}

  matches方法就是分别去匹配，如果是一个存放name的数组那就按数组去匹配，如果是按都会分割的字符串就将其处理数组去匹配。如果是正则直接用正则去匹配
  然后从this中解构出cache和keys，这两个属性是created钩子中添加的

  created () {
    this.cache = Object.create(null)
    this.keys = []
  },

  然后去定义唯一标识key，如果vnode.key本身就存在那么就用本身的key，否则就是使用组件构造器cid + '::' + 组件tag的形式。因为同一个构造器可能被注册为不同的本地组件，所以不能只使用cid。
  如果cache中能取到这个vnode那就把vnode.componentInstance设置为缓存vnode中的componentInstance，然后把key从对应keys中删除，然后再次添加。
  如果不存在缓存则将this.vnodeToCache赋值为vnode，this.keyToCache赋值为key。
  接下来将vnode.data.keepAlive置为true，最后返回vnode或slot[0]。这段逻辑是什么意思呢？我们来看mounted方法

  mounted () {
    this.cacheVNode()
    this.$watch('include', val => {
      pruneCache(this, name => matches(val, name))
    })
    this.$watch('exclude', val => {
      pruneCache(this, name => !matches(val, name))
    })
  },

  mounted时会调用cacheVNode方法，然后会监听include和exclude的变化。我们先来看cacheVNode

  methods: {
    cacheVNode() {
      const { cache, keys, vnodeToCache, keyToCache } = this
      if (vnodeToCache) {
        const { tag, componentInstance, componentOptions } = vnodeToCache
        cache[keyToCache] = {
          name: getComponentName(componentOptions),
          tag,
          componentInstance,
        }
        keys.push(keyToCache)
        // prune oldest entry
        if (this.max && keys.length > parseInt(this.max)) {
          pruneCacheEntry(cache, keys[0], keys, this._vnode)
        }
        this.vnodeToCache = null
      }
    }
  },

  首先获取cache, keys, vnodeToCache, keyToCache，如果vnodeToCache存在再进步执行逻辑然后就是将需要缓存的组件vnode内容缓存到cache中，并在keys中添加keyToCache。最后判断是否定义最大缓存数量，如果定义的话就会判断缓存数是否等于最大缓存。
  如果相等的话会调用pruneCacheEntry方法，这里第二个参数会传入keys[0]，也就是第一个被添加进去的缓存。

function pruneCacheEntry (
  cache: CacheEntryMap,
  key: string,
  keys: Array<string>,
  current?: VNode
) {
  const entry: ?CacheEntry = cache[key]
  if (entry && (!current || entry.tag !== current.tag)) {
    entry.componentInstance.$destroy()
  }
  cache[key] = null
  remove(keys, key)
}

  这里的cache[key]如果存在的话，并且当前渲染的vnode不存在或者当前渲染的tag不等于缓存的entry的tag。那么就将这个缓存的组件给销毁，并且将它缓存的内容清空，并且移除keys。
  那么这是什么意思呢？其实我们keys数组的作用就是按活跃顺序去保存我们组件key。这个过程是这样的：首先我们缓存了一个A组件，然后将A组件key添加到keys，然后在cache中缓存A组件，然后我们切换到B组件，把B组件同样缓存，但是如果我们再次切换到A时，那么发现A已经缓存了，那么直接从缓存中A组件实例给新的vnode，而不是重新去创建组件实例。这个时候还要做一个操作，那就是key从keys中删除，之后重新push一次，这样有什么不同呢？其实就是把它移到keys的栈顶，这样的话离栈顶近的其实就是最近活跃的，keys中第一个key可以认为是最不活跃的或者是长期不活跃的，那么如果超出缓存长度那就要先删除不活跃的组件。这是一种LRU缓存机制。
  然后我们来看监听器逻辑，也就是pruneCache函数

function pruneCache (keepAliveInstance: any, filter: Function) {
  const { cache, keys, _vnode } = keepAliveInstance
  for (const key in cache) {
    const entry: ?CacheEntry = cache[key]
    if (entry) {
      const name: ?string = entry.name
      if (name && !filter(name)) {
        pruneCacheEntry(cache, key, keys, _vnode)
      }
    }
  }
}

  这个逻辑也很简单，如果我们修改了include或者exclude，那么要重新定义缓存内容，遍历cache，如果cache[key]存在则说明没有被清除，然后进行下一步，判断如果filter返回false那么就要调用pruneCacheEntry删除这个缓存。
  接下来我们看渲染过程，在首次渲染时其实正常渲染逻辑没什么区别，但是如果是更新时候那么prepatch hook函数，从而会执行updateChildComponent，其实是和我们之前插槽更新是一样的，因为本质上keep-alive组件就是在处理插槽的内容。

  const newScopedSlots = parentVnode.data.scopedSlots
  const oldScopedSlots = vm.$scopedSlots
  const hasDynamicScopedSlot = !!(
    (newScopedSlots && !newScopedSlots.$stable) ||
    (oldScopedSlots !== emptyObject && !oldScopedSlots.$stable) ||
    (newScopedSlots && vm.$scopedSlots.$key !== newScopedSlots.$key) ||
    (!newScopedSlots && vm.$scopedSlots.$key)
  )

  这部分oldScopedSlots 不等于emptyObject并且其$table不为true，然后

  const needsForceUpdate = !!(
    renderChildren ||               // has new static slots
    vm.$options._renderChildren ||  // has old static slots
    hasDynamicScopedSlot
  )

  实际上这几个都是true，所以needsForceUpdate 也为true，那么就需要强制更新了。然后会强制更新

  // resolve slots + force update if has children
  if (needsForceUpdate) {
    vm.$slots = resolveSlots(renderChildren, parentVnode.context)
    vm.$forceUpdate()
  }

  注意这里的vm是keep-alive标签，所以我们这里会强制更新keep-alive组件，就是会重新执行其render函数

  vnode.componentInstance = cache[key].componentInstance
  // make current key freshest
  remove(keys, key)
  keys.push(key)

  然后执行这段逻辑，直接将旧的组件实例取出来赋值给它。然后会进入patch阶段，在patch阶段执行createComponent方法

function createComponent(vnode, insertedVnodeQueue, parentElm, refElm) {
  let i = vnode.data
  if (isDef(i)) {
    const isReactivated = isDef(vnode.componentInstance) && i.keepAlive // keepAlive相关逻辑
    if (isDef(i = i.hook) && isDef(i = i.init)) {
      i(vnode, false /* hydrating */) // 调用init方法
    }
    if (isDef(vnode.componentInstance)) {
      initComponent(vnode, insertedVnodeQueue)
      insert(parentElm, vnode.elm, refElm)
      if (isTrue(isReactivated)) {
        reactivateComponent(vnode, insertedVnodeQueue, parentElm, refElm)
      }
      return true
    }
  }
}

  这个时候isReactivated 其实就为true了，然后执行init方法

  if (
    vnode.componentInstance &&
    !vnode.componentInstance._isDestroyed &&
    vnode.data.keepAlive
  ) {
    // kept-alive components, treat as a patch
    const mountedNode: any = vnode // work around flow
    componentVNodeHooks.prepatch(mountedNode, mountedNode)
  } else {
    const child = vnode.componentInstance = createComponentInstanceForVnode(
      vnode,
      activeInstance
    )
    child.$mount(hydrating ? vnode.elm : undefined, hydrating)}
  }

  这时候if也是满足的，所以会执行if中逻辑，然后再次调用prepatch，实际上就是跳过了组件实例化的过程而直接去调用updateChildComponent去更新组件。注意这里updateChildComponent是去处理keep-alive子组件的更新，而上面调用是更新keep-alive组件的，这里更新的对象是不同的。
  更新完成后回到createComponent方法中，判断如果vnode上存在组件实例的话执行initComponent方法，这个方法会处理我们的insertedVnodeQueue，这个我们在生命周期小节已经分析过，这里就不在赘述了，因为它实在解释起来太复杂了。
  我们来看isReactivated为true时的这段代码会执行reactivateComponent。

  function reactivateComponent(vnode, insertedVnodeQueue, parentElm, refElm) {
    let i
    // hack for #4339: a reactivated component with inner transition
    // does not trigger because the inner node's created hooks are not called
    // again. It's not ideal to involve module-specific logic in here but there doesn't seem to be a better way to do it.
   //由于内部节点的createdhooks不会再次被调用，因此激活的组件内的transition不会被触发。在这里涉及模块特定逻辑不是最理想的，但似乎没有更好的方法来实现它。
    let innerNode = vnode
    while (innerNode.componentInstance) {
      innerNode = innerNode.componentInstance._vnode
      if (isDef(i = innerNode.data) && isDef(i = i.transition)) {
        for (i = 0; i < cbs.activate.length; ++i) {
          cbs.activate[i](emptyNode, innerNode)
        }
        insertedVnodeQueue.push(innerNode)
        break
      }
    }
    // unlike a newly created component,
    // a reactivated keep-alive component doesn't insert itself
    insert(parentElm, vnode.elm, refElm)
  }

  这里是为解决组件缓存后触发不了transition动画的问题，这里我们就不详细分析了，我们后面就会分析transition的实现了，这里它也说了这块实现并不是很理想，但没有更好的实现方式了。
