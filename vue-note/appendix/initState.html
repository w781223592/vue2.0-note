# 初始化state

这里我们先简单分析一下initState方法，先来看看下面函数：

export function initState (vm: Component) {
  vm._watchers = []
  const opts = vm.$options
  if (opts.props) initProps(vm, opts.props)
  if (opts.methods) initMethods(vm, opts.methods)
  if (opts.data) {
    initData(vm)
  } else {
    observe(vm._data = {}, true /* asRootData */)
  }
  if (opts.computed) initComputed(vm, opts.computed)
  if (opts.watch && opts.watch !== nativeWatch) {
    initWatch(vm, opts.watch)
  }
}

initState函数主要功能就是将props、methods、computed、watch中的数据初始化，并通过observe函数为data添加响应式，这里我们可以先关注一下初始化的顺序，它是先初始化props，然后初始化methdos、data、computed，watch，这个其实是比较重要的，也就是说props的优先级最高，data、computed、watch都可以访问到props中的数据，这个顺序决定着它们在初始化过程中能否访问到其他选项中的数据，当然methods是一个特殊选项，因为它不是在初始化中调用的，所以它能访问data中的内容。我们接下来可能不会在按顺序来分析它们。 我们现在只分析一下initData方法，其他内容在以后的章节中分析。

function initData (vm: Component) {
  let data = vm.$options.data
  data = vm._data = typeof data === 'function'
    ? getData(data, vm)
    : data || {}
  if (!isPlainObject(data)) {
    data = {}
    process.env.NODE_ENV !== 'production' && warn(
      'data functions should return an object:\n' +
      'https://vuejs.org/v2/guide/components.html#data-Must-Be-a-Function',
      vm
    )
  }
  // proxy data on instance
  const keys = Object.keys(data)
  const props = vm.$options.props
  const methods = vm.$options.methods
  let i = keys.length
  while (i--) {
    const key = keys[i]
    if (process.env.NODE_ENV !== 'production') {
      if (methods && hasOwn(methods, key)) {
        warn(
          `Method "${key}" has already been defined as a data property.`,
          vm
        )
      }
    }
    if (props && hasOwn(props, key)) {
      process.env.NODE_ENV !== 'production' && warn(
        `The data property "${key}" is already declared as a prop. ` +
        `Use prop default value instead.`,
        vm
      )
    } else if (!isReserved(key)) {
      proxy(vm, `_data`, key)
    }
  }
  // observe data
  observe(data, true /* asRootData */)
}

initData函数首先声明data为vm.options.data，再判断了data是否是function类型，若是则执行getData方法并传入data函数和vm两个参数，其定义如下：

export function getData (data: Function, vm: Component): any {
  // #7573 disable dep collection when invoking data getters
  pushTarget()
  try {
    return data.call(vm, vm)
  } catch (e) {
    handleError(e, vm, `data()`)
    return {}
  } finally {
    popTarget()
  }
}

先简单理解为data函数的调用返回给vm._data，之后响应式原理章节再仔细分析，之后判断data如果不是一个对象就在开发环境报一个警告。

在学习4.1小节时我们补充新的内容，在合并配置小节我们分析过合并时会有data的合并，所以这里是合并过的data函数，但是不管如何data函数执行都会返回一个对象，之后会保存在vm实例上，这里可以看到它其实是保存在vm._data上的。但是我们从来没有这样去访问data中的数据，我们总是这样写this.a，而不是this._data.a。这个问题往下看我们就会了解。 再往下获取data的keys数组，遍历keys数组，判断key有没有在methods或者props中定义过，如果定义过则在开发环境报警告，在!isReserved(key)下执行proxy(vm, '_data', key)方法，最后调用observe方法，observe方法我们会在4.1小节详细介绍。

我们来看proxy方法的定义

const sharedPropertyDefinition = {
  enumerable: true,
  configurable: true,
  get: noop,
  set: noop
}

export function proxy (target: Object, sourceKey: string, key: string) {
  sharedPropertyDefinition.get = function proxyGetter () {
    return this[sourceKey][key]
  }
  sharedPropertyDefinition.set = function proxySetter (val) {
    this[sourceKey][key] = val
  }
  Object.defineProperty(target, key, sharedPropertyDefinition)
}

proxy方法通过Object.defineProperty做数据劫持，sharedPropertyDefinition方法其实就是定义了一个公共的访问器属性，这里传入的是vm对象，所以就是对vm对象的各个对应data中的key做了劫持，这里我们访问this.a其实访问的是this._data.a，这就很好的解释了我们上面的问题。但是这也可以看到我们没有预先定义在data中定义的属性是没有响应式的，在做数据劫持方面vue2和vue3的proxy相去甚远。

在组件化3.1小节我们在Vue.extend中执行到了defineComputed方法，所以我们在这里分析一下defineComputed方法

export function defineComputed (
  target: any,
  key: string,
  userDef: Object | Function
) {
  const shouldCache = !isServerRendering()
  if (typeof userDef === 'function') {
    sharedPropertyDefinition.get = shouldCache
      ? createComputedGetter(key)
      : createGetterInvoker(userDef)
    sharedPropertyDefinition.set = noop
  } else {
    sharedPropertyDefinition.get = userDef.get
      ? shouldCache && userDef.cache !== false
        ? createComputedGetter(key)
        : createGetterInvoker(userDef.get)
      : noop
    sharedPropertyDefinition.set = userDef.set || noop
  }
  if (process.env.NODE_ENV !== 'production' &&
      sharedPropertyDefinition.set === noop) {
    sharedPropertyDefinition.set = function () {
      warn(
        `Computed property "${key}" was assigned to but it has no setter.`,
        this
      )
    }
  }
  Object.defineProperty(target, key, sharedPropertyDefinition)
}

它接收了三个参数，一个是目标对象target；第二是key，也就是计算属性名；第三个是userDef，也就是我们定义的计算属性方法。 首先这里定义shouldCache在非服务端渲染时为true，否则为false，然后判断我们写的计算属性是一个函数的话将sharedPropertyDefinition.get设为shouldCache，如果shouldCache是true执行createComputedGetter方法，否则执行createGetterInvoker方法，这时候是没有set的，所以将公共访问器set设置为空函数。 如果我们计算属性写的是一个对象形式则执行else中逻辑，这里是把公共访问器的get设置为计算属性的get，并判断这个get存在的话再判断shouldCache为true并且userDef.cache非false执行createComputedGetter，否则执行createGetterInvoker，最后如果get不存在的话就把sharedPropertyDefinition.get设置为空函数noop。然后还要判断set存在，则公共访问器的set属性设置为set，否则为空函数。 再往下则是对开发环境做一个校验，就是当我们没有设置set函数时却对计算属性进行了赋值，那么会报一个警告说：计算属性xx被赋值但是没有setter。最后就是对计算属性做劫持了，即通过Object.defineProperty方法将计算属性变为响应式，访问器属性设置为我们的sharedPropertyDefinition公共访问器属性，这里我们把它成为公共访问器属性或许并不贴切，变量名本意是‘公共属性定义’，但是它定义的内容是关于访问器的，所以我把它称为‘公共访问器属性’，实际上它在每次使用前会重新定义get和set方法，叫它‘公共访问器模板’会更加贴切。 下面是createComputedGetter方法的定义：

function createComputedGetter (key) {
  return function computedGetter () {
    const watcher = this._computedWatchers && this._computedWatchers[key]
    if (watcher) {
      if (watcher.dirty) {
        watcher.evaluate()
      }
      if (Dep.target) {
        watcher.depend()
      }
      return watcher.value
    }
  }
}

这部分内容我们会在第4.7小节分析计算属性时候做分析。 然后看defineComputed方法最后也是通过Object.defineProperty方法劫持了数据。 下面再介绍几个前面使用到的辅助函数 isPlainObject 判断是否是原始对象类型as shared/util.js


const _toString = Object.prototype.toString
export function isPlainObject (obj: any): boolean {
  return _toString.call(obj) === '[object Object]'
}

hasOwn判断是否是自有属性as shared/util.js

const hasOwnProperty = Object.prototype.hasOwnProperty
export function hasOwn (obj: Object | Array<*>, key: string): boolean {
  return hasOwnProperty.call(obj, key)
}

isReserved检查一个字符串是否以$或者_开头

export function isReserved (str: string): boolean {
  const c = (str + '').charCodeAt(0)
  return c === 0x24 || c === 0x5F
}

charCodeAt方法返回字符串指定位置字符的 Unicode 编码: 0x24为十六进制数，0o24（八进制），0b11（二进制）

我们把defineComputed方法放在这里分析是因为它定义在state.js中，它不仅是在Vue.extend中会用到，在initState方法中执行的initComputed方法也用到了这个方法。我们现在就来分析initComputed方法

function initComputed (vm: Component, computed: Object) {
  // $flow-disable-line
  const watchers = vm._computedWatchers = Object.create(null)
  // computed properties are just getters during SSR
  const isSSR = isServerRendering()

  for (const key in computed) {
    const userDef = computed[key]
    const getter = typeof userDef === 'function' ? userDef : userDef.get
    if (process.env.NODE_ENV !== 'production' && getter == null) {
      warn(
        `Getter is missing for computed property "${key}".`,
        vm
      )
    }

这里传入两个参数，实例vm和computed，首先将vm._computedWatchers定义为空对象，并用watchers保存它的引用，之后定义isSSR，这是判断是否是服务端渲染的标识，关于服务端渲染我们可能不会花时间去分析它。 最后一块逻辑是个大的for in循环，遍历computed对象并取到对应key的值保存在userDef中，然后就是取计算属性的getter函数，我们在写计算属性时一般会写成一个函数形式，其实就是计算属性的get函数，其实计算属性也是有set函数的，如果我们想去写一个set函数就需要将计算属性写成一个对象形式。所以这里就是判断我们配置的计算属性是一个函数还是一个对象，如果它是一个函数就直接拿它作为getter，否则就取它的get属性作为getter。 如果在开发环境中获取到的getter为null，就报一个警告说xx计算属性没有Getter

    if (!isSSR) {
      // create internal watcher for the computed property.
      watchers[key] = new Watcher(
        vm,
        getter || noop,
        noop,
        computedWatcherOptions
      )
    }

this.dirty = this.lazy // for lazy watchers

这里把this.dirty也置为了true，然后将this.getter赋值为expOrFn，也就是我们这里传入的第二个参数，对于lazy属性在以前的版本其实是叫computed，但是再最新版本中被修改为lazy，这样语义化更明确了，lazy是懒惰的，就是说计算属性不是立即执行的，而是懒执行的，而我们的渲染watcher就是立即执行的，它们体现在下面这段逻辑

      ? undefined
      : this.get()

如果this.lazy为true直接使this.value是undefined，而不是去执行get()，所以计算属性是懒执行的，所以这里用lazy做标识。

在这个时候就调用了defineComputed方法劫持了计算属性，需要特别注意的是这里是将计算属性挂到Sub.prototype上，这样做的好处是复用组件时不用重复去拦截计算属性，所以这里在state.js中的initComputed方法最后defineComputed是不会被执行的，只有在实例化时定义的计算属性才会执行到，目前能想到的场景只有根组件定义时。

接下来我们分析初始化props

function initProps (vm: Component, propsOptions: Object) {
  const propsData = vm.$options.propsData || {}
  const props = vm._props = {}
  // cache prop keys so that future props updates can iterate using Array
  // instead of dynamic object key enumeration.
// 缓存prop键，以便将来的prop更新可以使用数组而不是动态对象键枚举进行迭代
  const keys = vm.$options._propKeys = []
  const isRoot = !vm.$parent
  // root instance props should be converted
  if (!isRoot) {
    toggleObserving(false)
  }
  for (const key in propsOptions) {
    keys.push(key)
    const value = validateProp(key, propsOptions, propsData, vm)
    /* istanbul ignore else */
    if (process.env.NODE_ENV !== 'production') {
      const hyphenatedKey = hyphenate(key)
      if (isReservedAttribute(hyphenatedKey) ||
          config.isReservedAttr(hyphenatedKey)) {
        warn(
          `"${hyphenatedKey}" is a reserved attribute and cannot be used as component prop.`,
          vm
        )
      }
      defineReactive(props, key, value, () => {
        if (!isRoot && !isUpdatingChildComponent) {
          warn(
            `Avoid mutating a prop directly since the value will be ` +
            `overwritten whenever the parent component re-renders. ` +
            `Instead, use a data or computed property based on the prop's ` +
            `value. Prop being mutated: "${key}"`,
            vm
          )
        }
      })
    } else {
      defineReactive(props, key, value)
    }
    // static props are already proxied on the component's prototype
    // during Vue.extend(). We only need to proxy props defined at
    // instantiation here.
    if (!(key in vm)) {
      proxy(vm, `_props`, key)
    }
  }
  toggleObserving(true)
}

initProps方法接收了两个参数，一个vm实例，一个props对象。首先从$options中取propsData，如果为undefined就将它赋值为空对象并用propsData保存它的引用。然后定义keys保存vm.$options._proKeys，如果不存在则赋值为空数组。这里注释说明keys是用来缓存prop键的，这是为了以后更新prop时可以使用数组而不是动态对象键枚举进行迭代。然后又定义isRoot为!vm.$parent的结果，也就是判断是否是根实例。如果不是根实例则调用toggleObserving方法关闭响应式数据定义。toggleObserving方法我们在4.1小节已经讲解过了，它用来控制是否能够定义响应式数据。 接下来for...in...遍历propsOptions对象，并将key值push到keys数组中，接下来定义value等于validateProp的执行结果 我们来分析validateProp方法，该方法定义与util/props.js中

export function validateProp (
  key: string,
  propOptions: Object,
  propsData: Object,
  vm?: Component
): any {
  const prop = propOptions[key]
  const absent = !hasOwn(propsData, key)
  let value = propsData[key]
  // boolean casting
  const booleanIndex = getTypeIndex(Boolean, prop.type)
  if (booleanIndex > -1) {
    if (absent && !hasOwn(prop, 'default')) {
      value = false
    } else if (value === '' || value === hyphenate(key)) {
      // only cast empty string / same name to boolean if
      // boolean has higher priority
      const stringIndex = getTypeIndex(String, prop.type)
      if (stringIndex < 0 || booleanIndex < stringIndex) {
        value = true
      }
    }
  }
  // check default value
  if (value === undefined) {
    value = getPropDefaultValue(vm, prop, key)
    // since the default value is a fresh copy,
    // make sure to observe it.
    const prevShouldObserve = shouldObserve
    toggleObserving(true)
    observe(value)
    toggleObserving(prevShouldObserve)
  }
  if (
    process.env.NODE_ENV !== 'production' &&
    // skip validation for weex recycle-list child component props
    !(__WEEX__ && isObject(value) && ('@binding' in value))
  ) {
    assertProp(prop, key, value, vm, absent)
  }
  return value
}

接下来我们分析methods的初始化

function initMethods (vm: Component, methods: Object) {
  const props = vm.$options.props
  for (const key in methods) {
    if (process.env.NODE_ENV !== 'production') {
      if (typeof methods[key] !== 'function') {
        warn(
          `Method "${key}" has type "${typeof methods[key]}" in the component definition. ` +
          `Did you reference the function correctly?`,
          vm
        )
      }
      if (props && hasOwn(props, key)) {
        warn(
          `Method "${key}" has already been defined as a prop.`,
          vm
        )
      }
      if ((key in vm) && isReserved(key)) {
        warn(
          `Method "${key}" conflicts with an existing Vue instance method. ` +
          `Avoid defining component methods that start with _ or $.`
        )
      }
    }
    vm[key] = typeof methods[key] !== 'function' ? noop : bind(methods[key], vm)
  }
}

medthods的初始化相较于其他选项是比较简单的，这里获取到props，因为props是最先初始化的，所以这里只需要判断key有没有与props冲突就好了。这里先要判断它是一个function，否则的话就报一个警告。如果vm实例上有这个key了并且key是以_和$开头，说明我们定义的方法与Vue的原始方法重叠了，这时就报一个警告。 最后给将函数挂载到vm实例上，如果methods[key]不是一个方法的话就挂载一个空函数，否则就通过bind方法构建一个新的函数并绑定this为vm。bind方法其实就是原生bind方法的实现。
