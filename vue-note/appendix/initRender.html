# 初始化render

initRender函数定义于instance/render.js

export function initRender (vm: Component) {
  vm._vnode = null // the root of the child tree
  vm._staticTrees = null // v-once cached trees
  const options = vm.$options
  const parentVnode = vm.$vnode = options._parentVnode // the placeholder node in parent tree
  const renderContext = parentVnode && parentVnode.context
  vm.$slots = resolveSlots(options._renderChildren, renderContext)
  vm.$scopedSlots = emptyObject
  // bind the createElement fn to this instance
  // so that we get proper render context inside it.
  // args order: tag, data, children, normalizationType, alwaysNormalize
  // internal version is used by render functions compiled from templates
  vm._c = (a, b, c, d) => createElement(vm, a, b, c, d, false) // 模板方法
  // normalization is always applied for the public version, used in
  // user-written render functions.
  vm.$createElement = (a, b, c, d) => createElement(vm, a, b, c, d, true) // 手写render

  // $attrs & $listeners are exposed for easier HOC creation.
  // they need to be reactive so that HOCs using them are always updated
  const parentData = parentVnode && parentVnode.data
  ...省略一些开发时候的警告
}

initRender方法初始化了很多属性，如vm._vnode, vm._staticTrees等还有一些插槽相关的初始化，这部分我们现在还没用到暂时不做分析，这儿比较重要的是vm._c和vm.$createElement方法，这两个方法都是返回createElement方法，不同的是第六个参数一个为true一个为false，为true的时候可以用作手写render创建vnode，为false的时候用来将template编译为render函数转化为vnode

export function createElement (
  context: Component,
  tag: any,
  data: any,
  children: any,
  normalizationType: any,
  alwaysNormalize: boolean
): VNode | Array<VNode> {
  if (Array.isArray(data) || isPrimitive(data)) { // 函数重载如果没有data，则参数前移
    normalizationType = children
    children = data
    data = undefined
  }
  if (isTrue(alwaysNormalize)) {
    normalizationType = ALWAYS_NORMALIZE
  }
  return _createElement(context, tag, data, children, normalizationType)
}

这儿createElement()方法的定义也是十分简单，它接收了6个参数，第一个参数context是当前组件的实例，这个应该是在初始化的时候通过闭包技巧巧妙的绑定的，因为vm.$createElement方法执行的时候return了createElement方法的调用结果，需要注意的是这儿与闭包函数有所不同，它是直接调用了createElement方法，再看它的调用方式：

vnode = render.call(vm._renderProxy, vm.$createElement)

在mountComponent中它是这样使用的，作为一个参数传入的，那么它又是在什么时候调用的呢？既然它是render函数的参数那我们就去看看render函数怎么定义的。我们找不到render函数的定义，但是能找到render的来源：

const { render, _parentVnode } = vm.$options

render是从vm.$options中解构出来的，$options我们都清楚它是Vue选项，这个千呼万唤始出来的render竟然是我们自己写的，那接下来看看我们是怎么去写一个render函数呢？先看下面这个例子

render(createElement) {
    return createElement('div', {
      attrs: {
        id: 'app'
      }
    }, this.message)
}

这是我们手动去写的一个render函数，它的参数就是我们createElement，虽然我们习惯上简写为h，但是官方示例它确确实实是createElement，可能这儿叫createElement也是不太准确的，应该说它是$createElement。所以vm.$createElement方法就是我们在写render函数时候调用的。

我们继续分析createElement()方法的其他参数，第二参数tag，对应我们上面写的render函数就可以找到tag其实就表示div标签；data是一个与模板中 attribute 对应的数据对象，可选参数；children子级虚拟节点 (VNodes)，由 createElement() 构建而成，也可以使用字符串来生成“文本虚拟节点”，也是可选参数，另外根据官方解释它可以直接传文本虚拟节点，即如同我们写的示例一样，也可以传一个由createElement构建的虚拟节点，当有多个节点就可以传一个数组；第五个是normalizationType，这个我没有用过，先不做分析了，据我估计也没什么用，Vue自己用的时候会自己赋值，最后一个参数是alwaysNornalize，这个参数不是我们在写render函数时传入的，是定义的时候已经“写死”的值。

接下来我们看createElement方法的执行逻辑，首先判断data是不是数组或者原始类型属性，因为data是可选项，它本身应该是一个object，如果是数组或者原始类型的的话说明它其实是children，isPrimitive是对非undefined和null的原始类型的判断。这个if逻辑其实是实现一种函数的重载，当没有传data时，data位置的参数就是children，所以这时候要把后面参数整体前移。再往下判断了alwaysNormalize是否为true，如果为true则设置normalizationType等于ALWAYS_NORMALIZE（即常量2），最后return _createElement()，那么接下来就该分析_createElement()了，该方法就定义于createElement方法下边。

这个if...else if判断两种情况，一种是简单标准化，即normalizationType等于1时候，一种是完全标准化（normalizationType等于2），这两个normalize方法定于vdom/helpers/normalize-children.js下 先来看简单标准化代码：

// The template compiler attempts to minimize the need for normalization by
// statically analyzing the template at compile time.
// 模板编译器试图通过在编译时静态分析模板来最小化规范化的需要

// For plain HTML markup, normalization can be completely skipped because the
// generated render function is guaranteed to return Array<VNode>. There are
// two cases where extra normalization is needed:

//对于纯HTML标记，可以完全跳过规范化，因为生成的渲染函数保证返回数组<VNode>。有需要额外规范化的两种情况

// 1. When the children contains components - because a functional component
// may return an Array instead of a single root. In this case, just a simple
// normalization is needed - if any child is an Array, we flatten the whole
// thing with Array.prototype.concat. It is guaranteed to be only 1-level deep
// because functional components already normalize their own children.

// 1.当子级包含组件时-因为功能函数可能返回数组而不是单个根。在这种情况下，

// 只需要一个简单的规范化——如果任何子元素是数组，我们就用数组将整个元素展平

// Array.prototype.concat。它保证只有1级深度，因为功能组件已经规范化了它们自己的子级

export function simpleNormalizeChildren (children: any) {
  for (let i = 0; i < children.length; i++) {
    if (Array.isArray(children[i])) {
      return Array.prototype.concat.apply([], children)
    }
  }
  return children
}

我们可以看到这个代码上面有很长的几段注释，大致意思就是纯的HTML标签可以完全跳过规范化，但是有两种情况需要额外的处理，第一种就是我们上面的simpleNormalizeChildren方法，其调用场景是render函数是编译生成的，理论上children已经是VNode类型的，但在包含子组件时，函数式组件可以是一个数组而不是单个根节点，所以在这种情况我们将children数组给扁平化，即将一个二维数组简化为一维数组。说实话分析到现在我才开始看这几行简简单单代码，之前已经看过很多源码分析的资料，我已经知道这个函数干了什么，但是我去看这个函数逻辑时候，首先是懵逼的，感觉跟自己想的相去甚远，这个函数好像不是用来数组降维的。前面的循环所有人都能想到，但是数组的concat方法不直接用反而要通过apply方法强行调用就有点莫名其妙了，一般情况下我们都是在非数组类型下去调用数组本身的方法才要通过call或者apply去调用，一开始我有想过children不是个数组，也许是个类数组，但是apply第二个参数要求就是数组，传了children也恰恰说明它是个数组，再有就是我们要实现数组降维应该是数组每一项都要去判断和处理，为什么这儿却是遇到第一个数组类型的children[i]就return了呢？

抛开尤大写的这个不谈，我们要实现一个二维数组的降维会怎么做呢？先定义一个空数组来保存处理结果，再遍历原数组，然后判断子项是不是数组，如果不是就push到定义好的空数组中，如果不是就通过concat连接数组，这样我们不管怎么样都要完整遍历一次原数组。再看上面这段代码，它在有一个子项是数组的情况下就会退出循环，首先在遍历上可能会快上不少（关键看数组子项在什么位置出现），再之后只能通过concat方法和apply方法入手了，通过apply调用concat方法第一个参数我们知道是绑定this，这里传了一个[]，那么相当于[].concat，再看第二个参数就是children本身，正好apply第二个参数就是所有参数组成的数组，这就相当于[].concat([], children[0], children[1], ..., children[children.length - 1])，通过apply调用其实是利用apply特性巧妙的把children做了展开接着又利用concat特性一次性就处理好了数组，如果用我们的方法去做的话，要多遍历多少次，又要多执行多少次concat方法呢？其实这还没有完，我一直不知道数组concat可以连接非数组元素，为此我专门写了个示例

  let arr1 = []
  let arr2 = 2
  arr1 = arr1.concat(arr1, arr2)
  console.log(arr1) // [2]

这也就很好的解释了为什么能合并非数组子项和数组子项，因为concat方法就是能对数组和非数组处理，而且字符串本其实也有concat方法，甚至使用apply方法去调用数组concat时第一个参数可以数字或者字符串，只不过这时会给数字和字符串做包装类处理。

所以这就是我们为什么需要学习源码而不是会用就行的又一个重要原因了，学习源码可以让我们了解Vue的核心机制，使我们在开发时候能写出更高效合理的代码，另一方面我们在学习源码过程中可以学到很多知识，这是前辈的经验智慧的结晶，尤大在Vue设计中不放过任何一个可以优化的细节，这值得我们思考与学习。

好了言归正传，如果children本身是一个一维数组的话那就执行到了最后一步逻辑， 直接return children，关于这部分我们之后补充分析，因为我们示例是手写的render函数，这时候执行的是完全标准化方法。 那么接下来再看第二种情况，其调用场景有两种：一种是使用<template>，<slot>，v-for的时候会产生嵌套数组的情况，另一种是用户手写render函数时。

// 2. When the children contains constructs that always generated nested Arrays,
// e.g. <template>, <slot>, v-for, or when the children is provided by user
// with hand-written render functions / JSX. In such cases a full normalization
// is needed to cater to all possible types of children values.
// 2. 当子项包含始终生成嵌套数组的构造时，
//例如，<template>，<slot>，v-for，或当子项由用户提供时
//使用手工编写的渲染函数/JSX。在这种情况下，需要完全规范化
//需要满足所有可能类型的children值
export function normalizeChildren (children: any): ?Array<VNode> {
  return isPrimitive(children)
    ? [createTextVNode(children)]
    : Array.isArray(children)
      ? normalizeArrayChildren(children)
      : undefined
}

normalizeChildren方法定义是十分简单的，通过两个三目运算返回结果，先判断children是否是基础类型，如果是的话说明它是一个文本节点了，那就调用createTextVNode方法创建一个文本节点，并将它放到一个数组里以保持结构一致性，如果不是基础类型那就再判断是不是数组类型，如果是数组的话就执行normalizeArrayChildren方法，如果连数组都不是就返回undefined。

接下来我们就看看normalizeChildren方法的定义，normalizeChildren定义还是比较复杂的

function normalizeArrayChildren (children: any, nestedIndex?: string): Array<VNode> {
  const res = []
  let i, c, lastIndex, last
  for (i = 0; i < children.length; i++) {
    c = children[i]
    if (isUndef(c) || typeof c === 'boolean') continue
    lastIndex = res.length - 1
    last = res[lastIndex]
    //  nested
    if (Array.isArray(c)) {
      if (c.length > 0) {
        c = normalizeArrayChildren(c, `${nestedIndex || ''}_${i}`)
        // merge adjacent text nodes
        if (isTextNode(c[0]) && isTextNode(last)) {
          res[lastIndex] = createTextVNode(last.text + (c[0]: any).text)
          c.shift()
        }
        res.push.apply(res, c)
      }
    } else if (isPrimitive(c)) {
      if (isTextNode(last)) {
        // merge adjacent text nodes
        // this is necessary for SSR hydration because text nodes are
        // essentially merged when rendered to HTML strings
        res[lastIndex] = createTextVNode(last.text + c)
      } else if (c !== '') {
        // convert primitive to vnode
        res.push(createTextVNode(c))
      }
    } else {
      if (isTextNode(c) && isTextNode(last)) {
        // merge adjacent text nodes
        res[lastIndex] = createTextVNode(last.text + c.text)
      } else {
        // default key for nested array children (likely generated by v-for)
        if (isTrue(children._isVList) &&
          isDef(c.tag) &&
          isUndef(c.key) &&
          isDef(nestedIndex)) {
          c.key = `__vlist${nestedIndex}_${i}__`
        }
        res.push(c)
      }
    }
  }
  return res
}

normalizeChildren方法接收两个参数，一个是需要规范化的children，另一个是嵌套索引nestedIndex，第一次调用时候只传入第一个参数，它首先遍历children，如果c是undefined、null或者布尔值，跳过本次循环，如果c是数组并且不是一个空数组，就递归调用normalizeChildren，这时候就会传入第二参数nestedIndex，这个参数标识着每一层的索引，格式_1_0_1，最后返回的结果又用c保存，接下来就是对c中第一个数据和res最后一个数据判断是否都是文本节点，判断方法isTextNode定义如下：

function isTextNode (node): boolean {
  return isDef(node) && isDef(node.text) && isFalse(node.isComment)
}

大致就是判断传入不为空，且text属性不为空，isComment是false，也就是不是注释节点，那它就只能是文本节点。

然后判断c不是个数组，而是个基础类型，说明这是个文本节点，然后又是做了文本节点的合并，注释说明这在服务端渲染中是重要的，当然如果res最后一个不是文本节点就创建一个文本节点并push到res中。

最后c既不是数组又不是原始类型数据，如果还是满足合并条件就继续合并文本节点（可能就是一个普通的VNode），如果不满足合并条件就处理一些逻辑，注释说是这里可能是v-for生成的list，具体的之后了解了v-for再说吧，最后就是return我们的vnode数组res了。 我们接着看_createElement方法接下来的逻辑

先定义两个变量vnode ns，当tag是一个字符串的类型时，定义Ctor变量，给ns赋值context.$vnode.ns || config.getTagNamespace(tag)，config配置表是在core/config.js，里面都是一些空的配置项，因为很多配置项都是与平台相关的，其配置在其他文件下，在platforms/web/runtime/index下配置许多config选项，我们先看看接下来用到的两个配置：

Vue.config.isReservedTag = isReservedTag
Vue.config.getTagNamespace = getTagNamespace

这两个方法真正定义是在platforms/web/util/element.js下，先看getTagNamespace方法

export function getTagNamespace (tag: string): ?string {
  if (isSVG(tag)) {
    return 'svg'
  }
  // basic support for MathML
  // note it doesn't support other MathML elements being component roots
  if (tag === 'math') {
    return 'math'
  }
}

接收tag参数然后判断tag是SVG标签就返回svg，如果tag === 'math'就返回math，isSVG方法也非常简单

isSVG其实就是makeMap方法返回的判断方法，之前在initProxy方法中已经分析过makeMap方法了，这儿就不在详细分析了。

context.$vnode.ns这个暂时还没看到在哪赋值的，getTagNamespace返回值是svg或math没跑了，再看另一个方法isReservedTag

export const isReservedTag = (tag: string): ?boolean => {
  return isHTMLTag(tag) || isSVG(tag)
}

返回是一个Boolean值，isHTMLTag方法同样是由makeMap方法返回的检查函数，是对html保留标签的检查。

对tag做isResercedTag检查通过之后，在非生产环境下判断data.nativeOn并且tag !== 'component'时报一个警告，就是告诉我们在v-on绑定事件时在非组件标签上使用了native修饰符，这是不对滴。

接下来通过new VNode创建一个新的VNode，参数config.parsePlatformTagName其实就是tag，这个方法是创建一个平台相关标签，在web平台这个方法没有去做任何处理，接下来else if和else内容就是组件相关了，最后最外层的else判断tag不是string类型的情况，这部分也是组件相关逻辑，这时vnode等于createComponent()方法的调用返回，这部分逻辑我们在第三章组件化时候做。

最后我们看看_createElement方法的最后一段逻辑：

  if (Array.isArray(vnode)) {
    return vnode
  } else if (isDef(vnode)) {

总体逻辑就是vnode是数组就直接返回vnode，如果不是数组而是一个非空类型数据就对ns和data做判断并执行applyNS方法和registerDeepBindings方法，最后同样返回vnode，如果是个空值则创建一个空的vnode并返回。

下面来看看这两个方法做了什么事

// 添加命名空间
function applyNS (vnode, ns, force) {
  vnode.ns = ns
  if (vnode.tag === 'foreignObject') {
    // use default namespace inside foreignObject
    ns = undefined
    force = true
  }
  if (isDef(vnode.children)) {
    for (let i = 0, l = vnode.children.length; i < l; i++) {
      const child = vnode.children[i]
      if (isDef(child.tag) && (
        isUndef(child.ns) || (isTrue(force) && child.tag !== 'svg'))) {
        applyNS(child, ns, force)
      }
    }
  }
}
